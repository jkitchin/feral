Sparse matrix ordering is the process of finding a permutation of rows and
columns that reduces fill-in during factorization. Fill-in refers to zero
entries in the original matrix that become nonzero in the factors. Without a
good ordering, a sparse matrix can produce dense factors, destroying both
storage and computational efficiency. Ordering is the first step of the
[[file:../concepts/multifrontal-method.org][multifrontal method]] and directly
determines the shape of the elimination tree and the sizes of frontal matrices
citep:feral2026.

** Taxonomy of Ordering Algorithms

Fill-reducing orderings fall into two broad categories:

*** Local Heuristics (Minimum Degree Family)

Greedy algorithms that make locally optimal pivot selections at each step.
At each elimination step, select the vertex with the smallest degree
(number of neighbors) in the current elimination graph.

- *Exact Minimum Degree (MMD)*: Maintains exact degrees. O(n * nnz)
  worst case.
- *Approximate Minimum Degree (AMD)*: Uses degree upper bounds via the
  quotient graph model. Amortized O(nnz). See
  [[file:./approximate-minimum-degree.org][AMD concept article]].
- *Strengths*: Fast (especially for small/medium problems), low overhead,
  simple interface
- *Weaknesses*: Myopic — cannot exploit global structure, produces
  suboptimal fill on irregular and 3D problems

*** Global Partitioning (Nested Dissection Family)

Recursive algorithms that exploit graph structure by finding vertex
separators. The separator vertices are eliminated last, confining fill to
the two disconnected subgraphs.

- *METIS*: Multilevel graph partitioning with FM refinement
  citep:karypis1998fast. See
  [[file:./metis-graph-partitioning.org][METIS concept article]].
- *KaHIP*: Multilevel graph partitioning with max-flow-based refinement
  citep:sanders2011engineering. See
  [[file:./kahip-graph-partitioning.org][KaHIP concept article]].
- *SCOTCH*: Composable strategy-string-based partitioning
  citep:pellegrini1996scotch. See
  [[file:./scotch-graph-ordering.org][SCOTCH concept article]].
- *Strengths*: Provably optimal fill on regular meshes, much better fill
  on 3D and irregular problems, produces balanced elimination trees
  favorable for parallelism
- *Weaknesses*: Higher constant factors, more complex implementation,
  less effective on small problems

For a detailed comparison, see
[[file:./ordering-algorithm-comparison.org][Ordering Algorithm Comparison]].

** FERAL's Ordering Pipeline

All ordering algorithms produce the same output: a permutation vector
=perm: Vec<usize>= mapping new indices to old. This feeds into a common
downstream pipeline:

1. *Ordering*: AMD, METIS, or other → =perm[n]=
2. *Permute pattern*: Apply perm to get P*A*P^T sparsity pattern
3. *Elimination tree*: Construct from permuted pattern
   (George & Liu 1981, O(nnz * alpha(n)))
4. *Postorder*: Reorder elimination tree so children are contiguous
   (enables LIFO assembly and cache locality)
5. *Compose permutation*: =final_perm[k] = amd_perm[postorder[k]]=
6. *Column counts*: Liu's algorithm on the postordered tree
7. *Supernode detection*: Merge columns with identical structure
8. *Memory plan*: Preallocate factor storage

This separation of concerns means any new ordering algorithm only needs to
produce a permutation vector — the rest of the pipeline is shared.

** Elimination Tree

Given the permuted matrix P*A*P^T, the elimination tree encodes the dependency
structure of the factorization. Each column j has a parent p(j) defined as the
row index of the first subdiagonal nonzero in column j of the factor L. The tree
has the property that column j depends only on its descendants in the tree, so
any postorder traversal of the tree produces a valid elimination order.

The elimination tree is constructed from the permuted sparsity pattern using a
well-known O(nnz) algorithm (George and Liu, 1981) citep:george1981computer.
FERAL constructs it as a separate step after ordering.

** Postordering

The elimination tree is postordered so that children of each node are numbered
consecutively. This postordering is critical for two reasons. First, it enables
the LIFO property of the
[[file:../methods/two-arena-memory-model.org][ContribStack]]: in postorder
traversal, a node's contribution block is always the most recently pushed item
when the parent needs it. Second, it improves cache locality because related
frontal matrices are processed in sequence.

** Column Counts and Fill Estimation

After the elimination tree is postordered, column counts (the number of nonzeros
in each column of L) are computed using Liu's algorithm, as implemented in
CHOLMOD's =cholmod_rowcolcounts.c=. These counts are exact for the Cholesky case
and serve as lower bounds for the indefinite case (where delayed pivoting can
increase fill). The column counts feed directly into the =MemoryPlan= that
preallocates storage for the
[[file:../methods/two-arena-memory-model.org][two-arena memory model]].

** Supernode Detection

Supernodes are sets of consecutive columns in the postordered elimination tree
that have identical sparsity structure (same nonzero pattern below the diagonal
block). Amalgamating supernodes into single nodes of the assembly tree converts
many small operations into fewer large dense operations, enabling BLAS-3
performance. FERAL supports two policies: =SupernodePolicy::None= for small
problems (n < ~200) and =SupernodePolicy::Aggressive= for larger problems.

** Ordering Quality and Benchmarking

Key metrics for evaluating ordering quality:

1. *Fill ratio*: nnz(L) / nnz(A) — lower is better. A ratio above 1.5
   warrants investigation.
2. *Operation count*: Total flops for factorization — determined by frontal
   matrix sizes, which are set by the ordering.
3. *Ordering time fraction*: Time spent ordering / total solve time. For
   small matrices ordering can dominate; for large matrices factorization
   dominates.
4. *Elimination tree shape*: Balanced trees (from nested dissection) expose
   more parallelism than tall/skinny trees (from minimum degree).

FERAL's Tier 3 benchmark matrices include cases where AMD gives poor fill
compared to METIS, specifically to test ordering sensitivity. Matrix sources
for evaluation: SuiteSparse Matrix Collection (symmetric indefinite subset),
CUTEst KKT corpus, and constructed stress tests (arrow, bordered diagonal,
random irregular).
