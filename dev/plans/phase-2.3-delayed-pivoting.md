# Implementation Plan: Phase 2.3 — Delayed Pivoting

**Date:** 2026-04-13
**Research note:** dev/research/phase-2.3-delayed-pivoting.md
**Spec sections:** FERAL-PROJECT-SPEC.md §6 (multifrontal),
§7 (numerical pivoting), §8 (BK growth bound).
**Prior phases:** Phase 2.2.2 (column-relative threshold),
Phase 2.2.3 (supernode adjacency fix), Phase 2.2.3 follow-up
(InfNorm scaling default, threshold default removed).

---

## Goal

Make `factor_frontal` able to leave `nelim < ncol` columns
un-eliminated when the column-relative threshold test fails,
and make `factorize_multifrontal` carry those un-eliminated
columns up to the parent supernode where they re-enter BK pivot
search. Match the SSIDS `ldlt_tpp` model: break on first failure
within a supernode, let the parent absorb the tail.

Restore `pivot_threshold = 0.01` as the default — this is the
SSIDS / MUMPS default and the precondition for delays to fire
correctly. The Phase 2.2.3 follow-up removed the default
because the threshold without delays is a net harm; with
delays, the threshold is required for correctness and bounds
the L growth factor by `1/u = 100`.

## Implementation Steps

### Step 1 — Write failing unit tests for the kernel (TDD red phase)

**File:** `tests/delayed_pivoting.rs` (new).
**Deliverable:** ~5 unit tests that fail today and define what
"correct" looks like after the implementation lands.

Tests:
1. `factor_frontal_breaks_on_first_unstable_pivot_when_may_delay`
   — construct a 4×4 frontal where column 0 has a tiny diagonal
   (`d = 1e-12`) and a large off-diagonal (`= 1.0`). With
   `pivot_threshold = 0.01` and `may_delay = true`, expect
   `factor_frontal` to return `nelim = 0`, leaving the trailing
   block intact in `contrib`.
2. `factor_frontal_root_falls_through_to_force_accept` — same
   frontal with `may_delay = false`, expect `nelim = 4` and the
   unstable pivot counted as a zero (existing `ForceAccept`
   behavior).
3. `factor_frontal_partial_elim_returns_correct_l_block` —
   construct a frontal where 2 of 4 columns succeed and 2 fail.
   Verify `nelim = 2`, `l` has 2 columns, `d_diag` has 2
   entries, and `contrib` is `(nrow - 2) × (nrow - 2)` with the
   top-left 2×2 being the un-eliminated columns and the
   remaining trailing rows.
4. `factorize_multifrontal_delays_propagate_to_parent` —
   construct a small symbolic structure where a leaf supernode
   has an unstable pivot at `factor_threshold = 0.01`. Verify
   the parent supernode sees the delay in its `n_delayed_in`,
   that the parent's row_indices include the delayed column,
   and that the parent's `factor_frontal` is called with the
   expanded `ncol = own_ncol + n_delayed_in`.
5. `factorize_multifrontal_delayed_pivot_succeeds_at_parent` —
   end-to-end: a 6×6 KKT where the natural pivot at the leaf
   supernode would fail BK at the leaf but succeeds at the
   parent (because the parent has additional rows whose entries
   make the BK tests pass). Verify the final inertia matches a
   dense LDLT reference.

**Acceptance:** all 5 tests fail with clear error messages
that describe what's missing.

### Step 2 — Add the delayed-pivot data model (no behavior change)

**File:** `src/dense/factor.rs`.
**Deliverable:** `FrontalFactors` gains `nelim: usize` set
equal to `ncol` for now (no behavior change). All existing
tests still pass. `factor_frontal` does not yet use it.

This is a purely additive change to the data structure. The
solve path continues to use `ncol` as before.

### Step 3 — Implement `may_delay` in `factor_frontal`

**File:** `src/dense/factor.rs`.
**Deliverable:** `factor_frontal` takes a new `may_delay: bool`
parameter. When `may_delay = true` and a pivot fails the
threshold test (1×1 column-relative or 2×2 Duff-Reid), the
kernel `break`s out of the `while k < ncol` loop and returns
`FrontalFactors { nelim: k, .. }` with the trailing rows/columns
intact. The `contrib` field is computed as the sub-block
`a[(nelim..nrow), (nelim..nrow)]`, which now contains both the
un-eliminated columns (top-left `(ncol - nelim)²`) and the
non-fully-summed rows (bottom-right `(nrow - ncol)²`).

`try_reject_1x1_frontal` and the 2×2 path both gain the
`may_delay` branch. When `may_delay = false`, behavior is
unchanged from today.

**Caveat:** the existing `try_reject_*` functions zero the L
column and the diagonal of a rejected pivot. The new
`Delayed` path must NOT do that — the column data is needed
intact at the parent. Refactor so the zeroing only happens on
the `ForceAccept` path.

**Acceptance:** Steps 1.1, 1.2, 1.3 unit tests pass. Existing
unit tests in `dense::factor::tests` pass (they all run with
`may_delay = false` so behavior is preserved).

### Step 4 — Wire `may_delay` through `factorize_multifrontal`

**File:** `src/numeric/factorize.rs`.
**Deliverable:** the postorder loop computes `is_root` for each
supernode (a supernode is the root iff it has no parent in the
etree, i.e., no other supernode lists it as a child). It calls
`factor_frontal(.., may_delay = !is_root, ..)`. When the kernel
returns with `nelim < ncol`, the un-eliminated columns are
preserved in the contribution block as the top-left section.

`NodeFactors` gains `nelim: usize` and `n_delayed_in: usize`
fields. The contribution block `ContribBlock` gains
`n_delayed: usize` (count of how many of its row_indices are
delayed pivots vs trailing rows).

**Caveat:** detecting "is this the root supernode?" requires
knowing the parent set. Today's symbolic structure has each
supernode list its `children`; the root is the one that no
other supernode lists as a child. Compute this once at the
start of `factorize_multifrontal`.

**Acceptance:** Step 1.4 unit test passes. Existing tests pass
(no matrix in the existing suite triggers the delay path
unless `pivot_threshold > 0`, which is still off by default).

### Step 5 — Parent assembly: incorporate delayed columns

**File:** `src/numeric/factorize.rs`.
**Deliverable:** `build_row_indices` learns to query each
child's `ContribBlock::n_delayed` and includes the first
`n_delayed` row indices of each child's contrib as
**fully-summed** columns of the parent (not as trailing
rows). The expanded fully-summed column count becomes
`ncol + sum(child.n_delayed)`.

`extend_add` is split into two phases:
- For each child, scatter the top-left `n_delayed × n_delayed`
  block of contrib into the parent's expanded fully-summed
  region at columns `[snode.ncol + delay_offset, ...)`.
- Scatter the cross-block (delayed columns × trailing rows)
  into the parent's mixed region.
- Scatter the bottom-right `(contrib_dim - n_delayed)²` block
  into the parent's trailing region (as before).

`factor_frontal` is then called with `ncol = snode.ncol + sum_delays`.

**Caveat:** the parent's CSC scan for original A entries still
uses `row_indices[..snode.ncol]` (the parent's own native
columns), not the expanded set. Delayed columns from children
already have their A contributions baked into the values being
copied via extend_add. Don't double-count.

**Acceptance:** Step 1.5 unit test passes. The
`tests/parity.rs` suite still passes (matrices that don't need
delays are unaffected because `pivot_threshold = 0.0` and the
delay path is unreachable).

### Step 6 — Solve-side verification

**File:** `src/numeric/solve.rs`.
**Deliverable:** the per-supernode gather/scatter uses `nelim`
in place of `ncol` for the D-block solve loop. The L-solve
loop already uses the L block dimensions which will be
`nelim × nrow` after Step 3. The backward sub uses `nelim` for
its iteration upper bound.

**Caveat:** the `row_indices[nelim..]` for a node that had
delays now contains BOTH delayed columns (which got eliminated
at an ancestor) and trailing rows (which never get eliminated
at this node). The gather pattern `w[i] = y[row_indices[perm[i]]]`
already handles this uniformly because trail rows are read into
`w` and the L21 update flows naturally.

**Acceptance:** all parity tests pass after Step 7 below; no
solve-specific unit tests added (the integration tests in
`tests/delayed_pivoting.rs` cover this).

## Risks and Mitigations

**Risk 1: The contrib block's dual semantics confuse the
extend_add code.** Mitigation: split `extend_add` into
`extend_add_delayed_columns` and `extend_add_trailing_rows`,
each operating on a distinct slice of the child's contrib
block. Fail loudly with a debug assertion if a delayed-column
index lands in the trailing region or vice versa.

**Risk 2: The `nelim < ncol` branch in `factor_frontal` leaves
some local state in an inconsistent place** (e.g., D entries
allocated for `ncol` but only `nelim` populated). Mitigation:
allocate all output buffers based on `nelim` after the loop
breaks, not based on `ncol`. The truncation happens inside
`factor_frontal` before returning.

**Risk 3: Detecting "is this the root supernode?" is wrong on
disconnected matrices.** Mitigation: the etree may have
multiple roots (one per connected component). Treat every
node with no parent as a root. Test on a 2×2 block-diagonal
matrix.

**Risk 4: The parity panel re-curates and the headline number
appears unchanged or smaller.** This already happened in the
Phase 2.2.3 follow-up (the panel reselected matrices when the
default changed). Mitigation: report the parity_config_sweep
delta on the same panel, not just the regenerated panel count.
The session checkpoint should cite both.

**Risk 5: Delayed pivots blow up `nrow` and `ncol` of the
root supernode quadratically on pathological matrices.**
Mitigation: this is a SSIDS / MUMPS shared limitation; no
mitigation in Phase 2.3. If a parity matrix triggers this,
log it and revisit in a future phase.

**Risk 6: ACOPP30 still fails after delayed pivoting.** The
dense-vs-sparse triage in commit `610b213` showed even the
dense path fails on ACOPP30. Mitigation: don't gate Phase 2.3
on closing ACOPP30. Document in the validation report.

## Open questions for the implementer

1. Should `may_delay` be a parameter on `factor_frontal`, or
   should it live on `BunchKaufmanParams`? Parameter is cleaner
   (per-call control); on `params` is easier to thread through.
   **Recommendation: parameter**, because the dense path always
   wants `may_delay = false` and shouldn't have to construct
   different `BunchKaufmanParams`.

2. Should the `nelim` output replace `ncol` in `FrontalFactors`,
   or live alongside? **Recommendation: live alongside**, with
   `ncol` keeping its meaning of "attempted column count" and
   `nelim` being the actual eliminated count. Solve loops use
   `nelim`; the contrib-dim arithmetic uses both.

3. Should the `n_delayed_in` field be on `NodeFactors` or on
   `ContribBlock`? **Recommendation: ContribBlock** because
   that's the inter-supernode communication boundary. `NodeFactors`
   doesn't need it after factorization; only the parent reads
   it during assembly.

4. How should the validation report compare against the
   pre-Phase-2.3 baseline when the panel re-curates? Run
   `parity_config_sweep` BEFORE regenerating the panel, with
   four configurations: pre-Phase-2.3 default, post-Phase-2.3
   default with `may_delay = false`, post-Phase-2.3 default
   with `may_delay = true`, post-Phase-2.3 with full delays
   AND threshold = 0.01. The 4-way comparison shows where each
   feature contributes. **Recommendation: yes, do the 4-way
   comparison.**

## Out of scope for Phase 2.3

- **Reusing factor_frontal in the dense path with delays.**
  The dense path stays at `may_delay = false`.
- **MUMPS-style in-block opportunistic search.** First
  implementation matches SSIDS (break on first failure).
- **Adaptive threshold tuning.** Use SSIDS default `u = 0.01`.
- **GPU offload.** Not relevant.
- **STATICMODE-style early static perturbation.** Use the
  existing `ZeroPivotAction::ForceAccept` only at the root.
- **Matrices that even the dense path fails on (ACOPP30).**
  Tracked but not required to close.

## Follow-up phases enabled by Phase 2.3

- **Phase 2.4: SSIDS-style sperm renumbering.** Coarser
  supernodes give BK more pivot visibility. May help the
  matrices delayed pivoting can't reach.
- **Phase 2.5: in-block opportunistic search (MUMPS-style).**
  Reduce the per-tree delay count by trying more pivots inside
  each supernode before delaying.
- **Phase 2.6: adaptive `u`.** Tune the threshold per matrix
  based on observed pivot quality.
