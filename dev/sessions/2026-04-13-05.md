### Dense factor.rs root-cause tracing

Added `FERAL_DENSE_TRACE=1` env-gated prints to `src/dense/factor.rs`
and ran against ACOPP30_0026. Smoking gun at k=58: the 2×2 block

```
[[ 0       , -4.16e-15 ],
 [ -4.16e-15, -6.08e-9 ]]
```

has `|det| = 1.73e-29`, which passes `count_2x2_inertia`'s `eps²` floor
(4.93e-32) by 350×. At `u = params.pivot_threshold = 0.0` (the dense
default), the Duff-Reid growth bound `(|a22|*rmax + |amax|*tmax)*u ≤ |det|`
collapses to `0 ≤ |det|` and is trivially satisfied. L21 = A21·inv(D)
then scales by ≈10²⁹ and destroys the trailing submatrix. Residual 2.7e-2.

Verified via `grep` that `factor()` is only used by `numeric/factorize.rs`
in a test; the production sparse path uses `factor_frontal()`, so fixing
`factor()` would not regress sparse.

### Dense factor.rs fix attempt (reverted)

Applied two fixes:

- **(a) Duff-Reid u backstop** in the 2×2 growth check: replace
  `let u = params.pivot_threshold;` with
  `let u = params.pivot_threshold.max(f64::EPSILON.sqrt());`
  so the growth bound is not trivially satisfied at u=0.
- **(b) Reducible-column floor** at the top of the BK loop: extend
  `if gamma0 == 0.0` to `if gamma0 ≤ sqrt(eps)`, and inside, force-zero
  the diagonal if `|a[k,k]| ≤ sqrt(eps)` (with a 1×1 inertia count
  otherwise).

Triage on ACOPP30_{0026,0018,0000} produced residuals 1e-13..1e-14
(down from 2.8e-2), with inertia matching the sparse path (71, 137, 1).
Test suite: 80 + all parity/integration tests passed clean. Build and
clippy clean.

Full bench was catastrophic:

| metric               | baseline      | after fix      | delta    |
|----------------------|---------------|----------------|----------|
| dense inertia match  | 99.0%         | 94.5%          | −4.5 pp  |
| dense residual pass  | 99.8%         | 96.7%          | −3.1 pp  |
| dense worst residual | 2.80e-2       | 1.85e0         | worse    |
| dense failures       | 1838          | 8836           | +6998    |
| worst matrix         | ACOPP30_0026  | MISTAKE_0101   | new      |

Root cause of the regression: the sqrt(eps) absolute floor in fix (b)
assumes an equilibrated matrix (||A||∞ ~ 1). The bench corpus is not
equilibrated. For matrices where ||A||∞ is much larger than 1, legitimate
columns with off-diagonal couplings ~1e-10..1e-8 get flagged as
"reducible" and their diagonals force-zeroed. MISTAKE_0101 output shows
expected `(9, 13, 0)` → actual `(8, 13, 1)` — a positive pivot became a
zero pivot.

Reverted fix (b) first (keeping only fix (a)). Triage showed even worse
residuals than baseline: 2.6e2, 1.8e2, 1.1e5 on the three ACOPP30 cases.
Why: when fix (a) rejects the 2×2 at k=58, the fallback is
`do_1x1_pivot(&mut a, n, k, gamma0, ...)` with the *same* `a[k,k] = 0`
diagonal — dividing by ~0 produces gigantic L entries. Fix (a) cannot
stand alone.

Reverted fix (a) too via `git checkout src/dense/factor.rs`. Re-ran the
triage — back to baseline 2.80e-2. Re-ran the full bench — back to
baseline 99.0% / 99.8% / 2.80e-2 ACOPP30_0026. Tree state matches HEAD
(555b579) exactly.

Recorded the attempt in `dev/tried-and-rejected.md` with full evidence,

## Abandoned Approaches

- **Dense ACOPP30 fix via sqrt(eps) reducible-column floor + Duff-Reid
  u backstop.** Traced, implemented, and rejected in this session.
  Full writeup in `dev/tried-and-rejected.md` (2026-04-13 entry).
  Two key lessons:
  1. Absolute numeric floors (e.g. `sqrt(eps)`) only make sense on an
     equilibrated matrix. For a corpus spanning many orders of
     magnitude, any absolute cutoff will hit legitimate pivots in some
     matrix. A scaled floor (running max diagonal, matrix inf-norm) is
     the correct form.
  2. The dense `factor()` path has a latent bug in its 2×2 rejection
     fallback: it assumes `a[k,k]` is a usable 1×1 pivot, which breaks
     when `a[k,k] = 0`. Any future delayed-pivoting attempt in the
     dense path will need to either swap column k with a downstream
     column that has a usable diagonal, or treat the column as
     rank-deficient and skip to k+1.

2. **Choose a direction for the dense ACOPP30 fix.** Two options, both
   viable, with different tradeoffs:

   - **(A) Delayed pivoting in dense.** When the 2×2 growth bound fails
     at `{k, k+1}`, don't force-accept the unstable 2×2. Instead, search
     the remaining trailing submatrix `[k..n)` for a column `j` whose
     diagonal has `|a[j,j]| > some_floor` and whose couplings to column
     k are small — swap `{k, k+1}` with `{j, ...}` and retry. This is
     what SSIDS does via "break out of the loop and leave columns for
     the parent front"; for a dense single-front there's no parent, so
     the equivalent is within-matrix delayed pivoting. More invasive
     but fundamentally sound.

   - **(B) Matrix-scaled reducible floor.** Instead of `sqrt(eps)`, use
     `floor = max(sqrt(eps), sqrt(eps) * max_diag_magnitude)` where
     `max_diag_magnitude` is the running max of `|a[j,j]|` seen so far
     in the factorization. This gives a relative floor and should not
     trip on matrices where ||A||∞ >> 1. Cheaper to implement but more
     heuristic.

   My recommendation is (A) because it matches what the sparse path
   already does and avoids another round of "this absolute threshold
   doesn't work on matrix family X." But (B) might be a quick win that
   moves the dense path from 99.0% to 99.4-99.6% at low risk.

3. **Separately: fix the dense 2×2 rejection fallback bug.** Even
   without the ACOPP30 fix, `do_1x1_pivot(&mut a, n, k, gamma0, ...)`

   bug; it just hasn't been exposed because the u=0 dense default

   zero-diagonal case gracefully — either route to a zero-pivot count
