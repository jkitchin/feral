    independently now that the public signature is frozen.

## Abandoned Approaches

None this session. Considered and deferred:

- **Full peek-ahead panel in this session.** Sketched the replay
  strategy on paper: for trailing column `c`, apply pending pivot
  updates `p=0..c-1` in ascending `p` via the same `axpy_minus`
  kernel as scalar. This is bit-exact with scalar because the
  per-element accumulation order matches (scalar: pivot-outer,
  column-inner; replay: column-outer, pivot-inner; both traverse
  `(i,j)` with updates applied in ascending pivot index). But the
  2×2 pivot / rejection / `may_delay` interaction surface is wide
  enough that a same-session implementation risks landing subtle
  bit-parity bugs. Scoped to Step 4b in `dev/decisions.md`.

## Next Session Should

1. **Step 4b — implement the real peek-ahead panel.** Body lives in
   a new internal `lblt_panel_frontal` + `apply_blocked_schur` pair.
   The outer loop in `factor_frontal_blocked` becomes:

   ```
   while k < ncol {
       let remaining = ncol - k;
       if remaining <= params.block_size || scalar_forced {
           scalar_pivot_step(...)?;
           k += 1 or 2;
       } else {
           let (n_elim, status) = lblt_panel_frontal(...);
           apply_blocked_schur(...);
           match status {
               Full => {}
               BoundaryTriggered => scalar_forced_next_iter = true,
               Delayed => break,
           }
           k += n_elim;
       }
   }
   ```

   The panel handles only 1×1 pivots cleanly; 2×2 candidates trigger
   a boundary (plan §Risks #1 option b). Rejection terminates the
   panel early, caller does one scalar step, then re-enters. Must
   use `schur_kernel::axpy_minus_unroll4_nofma` for bit-parity.
