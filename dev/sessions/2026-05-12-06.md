# Session 2026-05-12-06

## Goal

Review issue #9 (Phase 2.4.3 32×32 in-register SIMD kernel with bit-exact rounding) and decide whether to implement it. Implementation on a feature branch if the work is justified.

## Accomplished

- **Phase 2.4.3 Step 1 (block_ldlt32 scaffold).** New module `src/dense/block_ldlt32.rs` with `BLOCK_SIZE: usize = 32`, `factor_block32` stub delegating to `factor_frontal`, and a bit-parity test harness (`assert_factors_bit_equal`). 3 tests pass trivially under the stub; they become load-bearing when the body diverges from scalar.
- **Phase 2.4.3 Step 2a (block32 trailing-update primitives).** `update_1x1_block32` and `update_2x2_block32` with scalar bodies that byte-match `do_1x1_update` / `do_2x2_update` at n=32. 7 new bit-parity tests at p=0, 5, 10, 28, 30 + zero-pivot + singular-2×2. `do_1x1_update` and `do_2x2_update` promoted from `fn` to `pub(crate) fn` so the test module can call them as the scalar oracle. No behavioral change.
- **Mid-session pivot to BLAS-3.** Per profile re-read (`dev/research/feral-kernel-profile-chainwoo.md`) the 2026-04-27 dominant cost on CHAINWOO_0000 was the 1984×32 root's *trailing-update* (62% of factor time), not the 32×32 in-block factor (≤8%). Wrote `dev/research/blas3-trailing-update.md` and `dev/plans/phase-2.4.3-blas3-trailing-update.md` re-scoping issue #9 toward the BLAS-3 trailing-update kernel.
- **BLAS-3 quad kernel.** `schur_panel_minus_nofma_strided_quad` in `src/dense/schur_kernel.rs` (~340 LoC). One pulp dispatch packs 4 destination columns per src-vector load. Unroll=2 SIMD body chosen specifically to keep 8 accumulators + 2 src + 1 alpha splat within AVX2's 16-ymm register budget. 2 bit-parity tests (176-config sweep + zero-alpha independence) — both pass.
- **Quad wiring.** `apply_blocked_schur_panel` walks trailing columns in groups of 4, falling through to dual for a 2-or-3-column remainder and single for the 0-or-1-column remainder. Three nested `split_at_mut` calls carve four disjoint dst slices.
- **Post-landing measurement showed no corpus win.** Dense p90 unchanged (small-frontal 1.32 → 1.33; medium 1.70 → 1.70). Re-profile via `cargo run --bin diag_supernode_cost --release` showed the new dominant cost is fixed per-supernode overhead: ns/sup 600–1900 vs ns/nnz 30–165 across the long-tail corpus. The ACOPR30_0067 nemin sweep confirmed it (nemin=1 → 242 µs at 493 supernodes, nemin=32 → 152 µs at 158 supernodes; time falls as fronts grow while ns/sup climbs — classic fixed-overhead signature).
- **Decision: park the BLAS-3 work as infrastructure.** `dev/decisions.md` 2026-05-12 (c) records the context, the obsolete profile, and the rationale for retention. The quad kernel is correct (176-config bit-parity + 19 byte-identical `blocked_ldlt` integration tests), in production for every front with ≥4 trailing columns, zero runtime cost on small-front workloads, and re-engages automatically when fronts grow tall-skinny again.
- **Issue #13 opened** ("Per-supernode fixed-overhead reduction"): workspace pooling, `SymmetricMatrix::validate()` bypass, `extend_add` direct slice writes. Acceptance criteria specified.
- **Issue #9 converted to blocked-on-#13** with explicit re-open criterion (ns/nnz dominating ns/sup on a relevant cluster after #13 lands).
