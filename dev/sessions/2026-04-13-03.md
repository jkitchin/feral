### Code changes

- `src/numeric/factorize.rs`:
  - Added `is_root: Vec<bool>` computed once at the top of
    `factorize_multifrontal` by walking
    `symbolic.supernodes[*].children`. Handles disconnected matrices
    (multi-root forests) uniformly.
  - Renamed the loop-local `ncol` binding to `own_ncol = snode.ncol()`
    to distinguish the supernode's native column count from the
    expanded fully-summed count after absorbing delayed children.
  - Computed `n_delayed_in` and `expanded_ncol = own_ncol + n_delayed_in`
    from the children's live `ContribBlock`s before the row-indices
    build.
  - Step 1 assembly still iterates `row_indices[..own_ncol]` — only
    the supernode's native columns are scanned from the permuted CSC.
    Delayed columns' A values are already baked into each child's
    contrib at the child's own Step 1, so scanning them here would
    double-count. Noted in a block comment.
  - `factor_frontal` is called with `expanded_ncol` and
    `may_delay = !is_root[snode_idx]`.
  - Contrib store uses the perm-unpermuted formula
    `contrib_rows[cj] = row_indices[ff.perm[nelim + cj]]` for
    `cj in 0..cdim`. Kernel swaps only touch `[0..expanded_ncol)`,
    so `ff.perm[i] = i` for `i >= expanded_ncol`, and the formula
    reduces to the old straight-slice form on the trailing portion
    of the contrib block.
  - Dropped the `debug_assert(n_delayed == 0)` trip-wire. Its
    successor is `debug_assert!(actual_nrow >= expanded_ncol)`
    which guards the row-indices length.
  - `NodeFactors.n_delayed_in` is now populated from the computed
    `n_delayed_in`. `NodeFactors.ncol` is the expanded attempted
    column count (matching `ff.ncol`).
  - Removed `#[allow(dead_code)]` from `ContribBlock.n_delayed`
    (now consumed by `build_row_indices` and the contrib store).
- `src/numeric/factorize.rs::build_row_indices`: rewritten to
  produce the three-region layout
  `[own_cols | delayed_cols | trailing_sorted]`. Trailing rows are
  deduplicated against the fully-summed set via an
  `is_fully_summed` boolean vector indexed by matrix column, so a
  delayed column that also appears as a pattern row of a parent
  column (via the full symmetric pattern) is not counted twice.
  When every child has `n_delayed == 0` the new code is pointwise
  equivalent to the old implementation.
- `src/numeric/solve.rs`: switched the forward-sub, D-solve, and
  backward-sub loops from `ff.ncol` to `ff.nelim`. The `L` factor
  is sized `(nrow × nelim)` and `d_diag` / `d_subdiag` are sized
  `nelim`, so iterating to `ncol` on a node that delayed columns
  would walk past the end of those arrays. The `ncol == 0` early
  skips became `nelim == 0` so a node that delayed everything
  exits cleanly.

### Integration tests (`tests/delayed_pivoting.rs`)

Added two tests (plan Steps 1.4 and 1.5):

- `factorize_multifrontal_delays_propagate_to_parent` — 4×4 arrow
  matrix `A = [[1e-3,0,0,1],[0,10,0,1],[0,0,10,1],[1,1,1,2]]` with
  `SupernodeParams { nemin: 1, scaling_strategy: Identity }` and
  `pivot_threshold = 0.01`. Column 0's column-relative test
  `|1e-3| vs 0.01*1.0` fails at its leaf supernode, and the kernel
  delays it. The test asserts:
  - at least 2 supernodes (not pinned exactly because
    fundamental-supernode detection can merge adjacent columns),
  - exactly one supernode with `n_delayed_in > 0` (the absorbing
    parent),
  - exactly one supernode with `frontal_factors.n_delayed > 0`
    (the delaying leaf),
  - `parent.ncol >= 1 + parent.n_delayed_in` (the parent's attempted
    fully-summed count includes the delayed column),
  - `parent.nrow >= parent.ncol`.
- `factorize_multifrontal_delayed_pivot_succeeds_at_parent` — same
  matrix, end-to-end: factor with dense LDLᵀ (which runs
  `may_delay = false` and falls through to `ForceAccept` on the
  tiny pivot) and with the sparse multifrontal path, and assert
  `sparse_inertia == dense_inertia`. Also asserts `delays_in_tree > 0`
  as a guard that the test is exercising the delay path rather than
  silently passing on the non-delay code path.

Sparse inertia match against MUMPS: **149820 → 152987 (+2.1 pts,
+3167 matches).** Sparse residual pass: **152453 → 154113 (+1.1 pts,
+1660 passes).** Worst residual dropped from **2.31e11 on SWOPF_0827
to 1.19e0 on HAIFAM_0080** — a 10^11× improvement in the tail.

The SWOPF_0827 catastrophe at 2.31e11 was caused by the 2×2
Duff-Reid det-floor rejection at a non-root supernode falling back
to a single 1×1 ForceAccept, which lost the structural coupling of
the two nearly-singular columns. With `may_delay = true` the kernel
now delays the whole 2×2 block to the parent, where the additional
rows from child contributions make the block pivot cleanly. The
same mechanism is what closed SWOPF, HATFLDBNE, HATFLDF, and the
rest of the family sitting at O(10^9)–O(10^11) residuals.

The dense numbers (`Inertia match 99.0%`, `Residual pass 99.7%`,
`Worst 3.99e-2 on ACOPP30`) are **unchanged**. Dense always runs
`may_delay = false`, so none of this commit's changes affect its
path. The invariant holds.

**Note on the headline number being an honest improvement without
Step 7.** This commit does not touch `pivot_threshold` — it is still
0.0 for sparse callers. The 2×2 det-floor rejection does not depend
on the column-relative threshold, so Step 5 alone is enough to close
SWOPF-family matrices where the failing pivot is a 2×2 growth bound
or det-floor. Step 7 (restore `pivot_threshold = 0.01`) will unlock
the additional column-relative path and is expected to add further
improvements on HYDCAR20, METHANL8, HATFLDG, and others.

## Decisions Made

None that require appending to `dev/decisions.md`. One implementation
choice worth flagging in the journal:

- **Contrib row_indices formula chosen as
  `row_indices[ff.perm[nelim + cj]]` for all `cj in 0..cdim`**,
  uniformly across delayed and trailing positions, rather than
  special-casing the two halves. Justification: BK pivot search
  only swaps within `[0..expanded_ncol)`, so `ff.perm[i] = i`
  identically on the trailing half, and the uniform formula is
  strictly correct. Special-casing would be faster but not by much
  (branch predictor catches it) and the unified form is easier to
  audit for off-by-ones.
