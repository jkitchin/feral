## Decisions Made

- **Blocked path is an opt-in API, not a default flip.**
  `factor_frontal_blocked` is a separate public function rather than
  a `BkConfig::use_blocked` flag or an env-var dispatch inside
  `factor_frontal`. Rationale: the oracle strategy (scalar is the
  ground truth; blocked must match byte-for-byte) is clearest when
  both paths are callable from the same test site. Once Step 4
  flips default routing, `factor_single_front` and the multifrontal
  driver pick the blocked variant based on
  `remaining > block_size`, but the scalar entry point stays
  public for parity tests and for rejection-heavy regressions.

- **Parity oracle is `f64::to_bits`, not `approx_eq`.** The scalar
  kernel produces a specific IEEE 754 rounding trajectory that the
  blocked kernel must reproduce (already an implicit requirement
  from the Phase 2.4.2 FMA-unroll4 reversion on 2026-04-14 — 1-ULP
  drift caused 4 inertia regressions). Enforcing bit-parity makes
  any drift a compile-time-visible test failure rather than a
  weeks-later inertia regression.
