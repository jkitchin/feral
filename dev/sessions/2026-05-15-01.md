## Goal

### 3. Direct inertia comparison vs MA57 — **overturns the issue's hypothesis**

Dumped 4 KKT matrices from a `cb=default` run via
`POUNCE_DUMP_KKT` (iter004, 010, 043, 046). MUMPS failed all 4 with
INFOG(1) = -9 (memory allocation); MA57 succeeded on all 4 →
used MA57 as reference oracle.

feral inertia under `cb=default` matches MA57 **exactly** on all 4:

| matrix   | feral (cb=default)  | MA57                |
|----------|---------------------|---------------------|
| iter004  | (14398, 9602, 0)    | (14398, 9602, 0) ✓  |
| iter010  | (14399, 9601, 0)    | (14399, 9601, 0) ✓  |
| iter043  | (14398, 9602, 0)    | (14398, 9602, 0) ✓  |
| iter046  | (14383, 9617, 0)    | (14383, 9617, 0) ✓  |

So **cascade-break is not producing wrong inertia.** The
"WrongInertia" status pounce reports is the IPM's *expected*
neg-eval count (9601) disagreeing with the matrix's actual
inertia. That's an IPM trajectory bug, not a feral counting bug.

### 4. Located the real mechanism: ~5-OOM solve-accuracy regression

On iter004 (identical input matrix in `cb=off` and `cb=default`):
both configs report Success and the same inertia `(14398, 9602, 0)`.
But the solution vectors differ:

```
||sol_default - sol_off||_inf  = 2.3e-4
||sol_off||_inf                = 16
relative diff                  = 1.4e-5
```

That is ~5 orders of magnitude solve-accuracy loss for
`cb=default` vs `cb=off` on an identical matrix with identical
reported inertia. Plausible mechanism: cascade-break perturbs the
L factor (not just D), so the solve sees a slightly different
system than the one that generated the inertia count.

### 5. Identified why F2.3 iterative refinement isn't catching this

`pounce-feral/src/lib.rs:107` calls `Solver::solve`, **not**
`Solver::solve_refined`. The `RefinementDiagnostics` infra shipped
in commit `c6eee1f` last session is therefore not absorbing this
1e-5 solve error. Wiring `solve_refined` into pounce-feral is the
cheapest candidate fix.

## Decisions Made

- **Issue #17 is reclassified.** It is not an inertia-correctness
  bug. It is a solve-accuracy regression: cascade-break's
  perturbation degrades the L factor enough that downstream IPM
  trajectories diverge on near-singular KKT systems. The fix path
  is "wire `solve_refined` into pounce-feral" first, with
  L-factor perturbation forensics as the deeper investigation.
  Recorded in `dev/decisions.md`.

## Abandoned Approaches

- **Default `cascade_break_ratio = None` in feral as a fix for
  #17.** Considered but rejected: this reverts the cascade-arm
  gate that #15 shipped, and the cascade-break heuristic is
  helping on the matrices where it was calibrated. The right
  cut is to fix the solve-accuracy regression downstream
  (refinement) rather than defang the trigger upstream. Recorded
  in `dev/tried-and-rejected.md`.
