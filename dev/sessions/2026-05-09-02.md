# Session 2026-05-09-02

## Goal

Triage and fix GitHub issue #3: `OrderingMethod::ScotchND` silently
falls back to AMD on KKT-shaped matrices while `resolved_method`
still reports `ScotchND`. Also addresses the issue's secondary
observation (`OrderingMethod::Auto` resolves to `Amd` on n=74892
instead of the expected `MetisND` from `pick_default_method`'s
`n>=5000 && avg_deg<6` rule).

## Accomplished

### SCOTCH driver-level reproducer (commit c7f7364)

`crates/feral-scotch/tests/issue_3_kkt_repro.rs` builds a
PoissonControl K=20 KKT pattern (n=1200) and runs `scotch_order_full`
against `feral_amd::amd_order` directly. Asserts:

- `ScotchStats.n_separator_vertices == 0` — bisection produced no
  separator on this shape.
- `ScotchStats.n_amd_leaf_calls >= 1`.
- `scotch_perm == amd_perm` (bit-identical permutations).

This confirms the bug at the *driver* level: SCOTCH's multilevel
pipeline runs (8 coarsen levels, 416 FM passes) but the recursion
falls into `amd_leaf` for the entire graph because every bisection
attempt returns one empty side. Driven by the degenerate-bisection
recovery at `crates/feral-scotch/src/node_nd.rs:147-150`.

`feral-amd` and `feral-ordering-core` added to feral-scotch's
`[dev-dependencies]` so the test compiles.

### Fix #1 — honest `resolved_method` (commit 0b2099a)

In `src/symbolic/mod.rs::run_external_ordering`, the `ScotchND`
branch now calls `feral_scotch::scotch_order_full` (instead of the
perm-only `scotch_order`) and inspects `ScotchStats.n_separator_vertices`.
When zero — meaning SCOTCH did no real ND work and the returned
permutation is bit-identical to AMD's — `resolved_method` is
re-stamped from `ScotchND` to `Amd` so callers branching on the
field get the truth. The fallback itself is preserved.

New regression test
`issue_3_scotchnd_on_kkt_resolves_to_amd_when_bisection_degenerates`
in `src/symbolic/mod.rs::tests` builds the K=20 KKT, pins
`preprocess=OrderingPreprocess::None` to bypass `LdltCompress`,
requests `ScotchND`, asserts `resolved_method == Amd`. Fails on
parent commit, passes here.

### Fix #2 — `Auto` consistent with the no-arg default (commit 5e1d814)

Two coupled changes in `src/symbolic/mod.rs`:

1. `symbolic_factorize_with_method` now resolves
   `OrderingMethod::Auto` against the *original* matrix's full
   pattern, before any preprocessor (`LdltCompress`) reshapes the
   graph. The concrete method threads through the dispatch.
   `run_external_ordering` drops its internal `choose_adaptive`
   call and gains a `debug_assert_ne!(method, Auto)`.

2. `choose_adaptive`'s residual branch delegates to
   `pick_default_method` instead of returning `Amd` outright.
   Full-symmetric `nnz` is converted back to a stored-equivalent
   via `(full + n) / 2` (exact when the diagonal is stored once
   per row, which `symmetric_pattern` produces). The two existing
   shape-bakeoff branches (`n>100_000 → ScotchND`, `n<10_000 →
   KahipND`) keep priority. `Auto` is now a strict superset of
   `pick_default_method`.

New regression test `issue_3_auto_on_kkt_routes_via_pick_default_method`
builds K=58 (n=10092) PoissonControl KKT, requests `Auto`, asserts
`resolved_method == MetisND` and that it matches what
`symbolic_factorize` picks. Fails on parent commit, passes here.

One existing test updated: `choose_adaptive_rules`'s "everything
else → AMD" assertion (n=50_000, full avg_deg=20) now expects
`MetisND` because the delegation reaches `pick_default_method`'s
`n>10_000 → MetisND` residual rule. Comment in the test records
the arithmetic.

## Decisions Made

Logged to `dev/decisions.md` under the dated header
`2026-05-09 — resolved_method is what ran, not what was asked`.
Three tied points:

1. `resolved_method` is a contract field that must equal the
   concrete algorithm that produced `perm`.
2. `OrderingMethod::Auto` is resolved against the original matrix,
   once, before preprocessing.
3. `choose_adaptive`'s residual delegates to `pick_default_method`.

## Abandoned Approaches

None. Considered and not pursued (out of scope for this fix):

- Making SCOTCH actually work on bordered-KKT shapes by adjusting
  the `max_imbalance` cap or seeding the initial bisection
  differently. Would be a research note plus driver-level changes;
  not necessary because `pick_default_method` already routes these
  matrices to MetisND, and after this session `Auto` does too.
- Hard-erroring on degenerate bisection (instead of silently
  re-stamping `resolved_method`). Would surface unrecoverable
  failures with no caller workaround. Visibility was the actual
  reported bug.
