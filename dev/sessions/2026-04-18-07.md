# Session 2026-04-18-07

## Goal
Sparse-pipeline performance work after the ordering chapter closed.
Two threads:
1. Diagnose and fix the refinement-loop overhead identified by
   `profile_sparse` (11.5x bare-solve cost on small KKT matrices).
2. Diagnose and fix the CRESC132 factor outlier (the worst sparse
   factor ratio in the bench, 521x MUMPS).

## Accomplished

### Refinement: 2-strike plateau exit (commit 1418100)
- New per-stage profiler `src/bin/profile_sparse.rs` (symbolic,
  factor, bare solve, refined solve) with SSIDS/MUMPS sidecar
  comparison. Surfaced refinement as the dominant solve-time
  hotspot.
- Re-exported `solve_sparse` from `src/lib.rs` so the profiler
  can call the bare solve.
- `src/numeric/solve.rs`: `solve_sparse_refined` now exits after
  2 consecutive non-improving steps (`max_stagnant_steps = 2`)
  instead of running 10 iterations to an unreachable epsilon\*sqrt(n)
  relative target.
- Bug history: while tuning the plateau exit, the order-of-operations
  on the `improved` check (`best_r_norm = r_new_norm; let improved
  = r_new_norm < best_r_norm;`) silently capped iterations to 1
  regardless of cap. Five tuning variants gave identical results
  for this reason. Fixed by computing `improved` before updating
  `best_r_norm`.
- 1-strike vs 2-strike empirically chosen: 1-strike killed the
  bouncing-into-basin behavior on borderline matrices like BATCH;
  2-strike preserves it.

### CRESC132 fix: bordered-KKT routing (commit 824d3e6)
- Throwaway `cresc_diag` binary diagnosed CRESC132_0000 with AMD:
  4846 cumulative delays, root frontal 5084x4854 (96% of n=5314),
  factor time 5.7 s. With METIS: 1333 delays, root 2661x2200,
  factor time 480 ms (11x). VESUVIO showed AMD already optimal --
  this is genuinely a bordered-KKT pathology in CRESC.
- Symbolic max\_nrow does NOT predict actual max\_nrow (both AMD
  and METIS report symbolic max\_nrow=8 on CRESC132); delays are
  entirely a numeric artifact. So a "posteriori on symbolic"
  check has no signal -- the original plan for option 2 had to
  be downgraded to a static rule.
- Zero-diagonal hypothesis was wrong: CRESC132 has only 2 zero
  diagonals (the constraint block has been regularized in the
  CUTEst dump).
- `src/symbolic/mod.rs`: new `pick_default_method(n, stored_nnz)`
  helper. Rule:
    n >= 5000 && stored_nnz/n < 6  ->  MetisND
    everything else                 ->  Amd
- `symbolic_factorize` (the default entry point) now applies the
  rule. Callers wanting literal AMD must call
  `symbolic_factorize_with_method(.., OrderingMethod::Amd)`
  explicitly.
- `src/bin/bench.rs`: `FERAL_ORDERING` dispatch is now
  `Option<OrderingMethod>`; unset routes through the heuristic,
  explicit values bypass it. The previous "amd unset-default" was
  bypassing the new heuristic, which is why the first bench run
  with the heuristic in place reported unchanged times.
- New unit test `pick_default_method_rules`.

## Decisions Made

- **`symbolic_factorize` is no longer pure AMD.** The default entry
  point applies a narrow heuristic. The literal-AMD escape hatch is
  `symbolic_factorize_with_method(.., OrderingMethod::Amd)`. This
  is a public-API behavior change; documented in CHANGELOG and
  function docstring. Caught by `pick_default_method_rules` test.
- **Refinement default is best-iterate + 2-strike plateau exit, max
  10 iterations.** Replaces the previous "run to relative target or
  10 steps." A 1-strike rule was tried and rejected because it
  killed bouncing-into-basin behavior on borderline KKT matrices.
- **Posteriori dispatcher on symbolic-only output is not viable.**
  `cresc_diag` proved that symbolic max\_nrow doesn't predict actual
  max\_nrow. A real posteriori check would have to do partial
  numeric factorization.

## Abandoned Approaches

- **1-strike refinement plateau exit.** Lost 131 marginal matrices
  vs baseline (vs 86 for 2-strike) and pushed worst residual from
  2.69e-4 to 3.12e-4. 2-strike preserves baseline worst residual
  exactly while still cutting solve time 27%.
- **Symbolic-max-nrow as ordering-quality signal.** Hypothesized
  that high `sym_max_nrow / n` would flag bordered-KKT-bad
  orderings; verified that symbolic doesn't see delays so the
  signal is uniformly small (~8) regardless of how bad the
  ordering will turn out numerically.
- **Zero-diagonal fraction as bordered-KKT signal.** Hypothesized
  CRESC132 had many literal zero diagonals (the (2,2) block of a
  KKT). Census showed only 2; the dump is already regularized.
