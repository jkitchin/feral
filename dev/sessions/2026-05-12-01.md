# Session 2026-05-12-01

## Goal

Investigate why the rayon-parallel multifrontal driver — now wired
into `Solver::factor` by default (issue #7, prior session) — does
not produce a strong speedup on large matrices, and land any
bounded fix uncovered by the investigation.

## Accomplished

Two perf fixes plus one permanent diagnostic, all bit-exact-safe.

### Fix 1: skip upper-triangle memset on pooled frontal buffer (Pattern 0)

`factor_one_supernode` re-uses `FactorWorkspace::frontal_values` as
the backing store for a new `SymmetricMatrix(nrow × nrow)` on every
supernode. The previous code did `frontal_buf.resize(nrow*nrow, 0.0)`,
which on macOS triggers a full `memset` of the buffer (the underlying
mmap pages are already dirty from prior supernodes). Only the lower
triangle is touched by the dense kernels, so the upper-triangle zero
is dead work.

Added `SymmetricMatrix::from_pooled_buf(n, buf)` in
`src/dense/matrix.rs` which grows the buffer if needed but zeros
only the lower triangle (`n(n+1)/2` cells). Drop-in replacement at
the two sites in `factor_one_supernode` and at the one diagnostic
site in `src/bin/diag_leaf_profile.rs`. Bit-exact: the upper
triangle is never read by any consumer.

### Fix 2: per-worker `local_contribs` pool in the parallel driver

`run_parallel_task` was allocating
`Vec<Option<ContribBlock>>` of length `n_snodes` on every spawned
task — initialized to all-`None` — to stage children's
contribution blocks into a per-task indexed lookup. For cont-201
that's 11 121 tasks × 11 121-element vec × ~72 B = ~9 GB of
allocator churn over a single factor, all hitting the system
allocator concurrently from 8 rayon workers.

Replaced with a pooled `local_contribs` field on `FactorWorkspace`,
pre-sized once per worker in the parallel driver. Slots are
guaranteed `None` between tasks (each task drains its children's
slots into the pool, then takes its own slot out into the shared
store), so no clearing is needed on return. Uses
`std::mem::take` to split-borrow the pooled vec from
`&mut FactorWorkspace` without `unsafe`. Bit-exact: same data, same
order, just heap-allocated once per worker instead of once per task.

### Diagnostic: weighted etree critical-path analyzer

Extended `solver_parallel_profile_breakdown` to compute the true
parallel ceiling = `total_work / longest_weighted_etree_path` for
each test matrix, by walking the symbolic supernodes in postorder
and propagating `earliest_finish[i] = max(earliest_finish[c]) + work_us[i]`
over children `c`. Replaces the misleading naive Amdahl bound
(`total_work / max_single_supernode_work`).

### Critical-path analysis (true parallel ceiling)

| matrix    | naive Amdahl ceiling | **true ceiling** | measured sp@8 (post-fix) |
|-----------|---------------------:|-----------------:|-------------------------:|
| bcsstk38  | 37.84×               | 2.33×            | 1.29×                    |
| bratu3d   | 50.67×               | 1.69×            | 1.56×  (92% of ceiling)  |
| c-big     | 403×                 | **1.04×**        | 1.04×  (at ceiling)      |
| cont-201  | 27.25×               | 4.83×            | 1.44×  (30% of ceiling)  |

**c-big is fundamentally serial**: critical path = 258.6 s of the
268.5 s total work, etree depth 1520. No node-level scheduling fix
can lift this. Within-node parallelism (panel-parallel BK, threaded
GEMM) is the only path to more speedup there — a separate
engineering project from issue #7.

## Decisions Made

Two new entries in `dev/decisions.md` (this date):

1. `SymmetricMatrix::from_pooled_buf` — skip upper-triangle zero on
   pool reuse. Bit-exact safe by inspection of the dense LDLᵀ +
   Schur kernels: both touch only the lower triangle.
2. Pool `local_contribs` per worker in `FactorWorkspace`. Removes
   per-task n_snodes-sized alloc. Bit-exact (same data, same order).

## Abandoned Approaches

- **Pattern 2 (full packed-storage rewrite of the Schur kernel).**
  Surfaced as an option for c-big specifically (largest fronts
  10 000 × 18, so cutting half the memset bandwidth could matter).
  Critical-path analysis (true ceiling 1.04×) proved no node-level
  fix can lift c-big, so the cost/benefit collapsed. Not pursued.
  Pattern 0 captured the easy half of the win uniformly across
  matrices. Logged in `dev/tried-and-rejected.md` (TODO if
  re-attacked).

## Next Session Should

Priority order:

1. **cont-201 still has 30%-of-ceiling parallel headroom** (1.44×
   measured vs 4.83× theoretical at T=8). The shared
   `contrib_blocks` mutex is acquired twice per task (drain + write).
   At 11 121 tasks with avg 10 µs work, mutex traffic could be
   non-trivial. Two investigations worth tracing before any fix:
   - Replace `Mutex<Vec<Option<ContribBlock>>>` with a per-slot
     atomic-pointer or `boxcar`-style append-only vector to remove
     the global lock entirely.
   - Profile rayon task-spawn cost at 11 121 tasks; a chunked
     spawning (group small leaves) may reduce overhead.
2. **c-big is closed for now** at the node-parallel level. If
   anyone wants more speedup there, the only path is within-node
   parallelism (panel-parallel BK or threaded BLAS-3 Schur).
   That's a separate plan and is out of scope until issue #7
   itself is fully retired.
3. **pulp SIMD rewiring (Phase 2.4.3)** is still blocking the
   issue #7 SIMD sub-item. `mul_add_f64s` → `mul_f64s + sub_f64s`
   inside the Schur kernel, plus bit-parity unit tests against the
   scalar path. Independent of this session's wins.
