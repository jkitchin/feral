## Accomplished

### W-2 2×2 inline (no-swap fast path) — landed

**File-level changes (`src/dense/factor.rs`, +345/-86 lines):**

- `lblt_panel_frontal` now accepts `ncol` and `&mut [f64] subdiag`
  parameters and tracks `r` (argmax row) during the gamma0 scan.
- New no-swap 2×2 acceptance branch (~120 LoC) replacing the
  unconditional bail at `akk < alpha_bk * gamma0`. Mirrors
  `scalar_pivot_step:1937-2058` predicates: gates on
  `r == col + 1 && col + 1 < ncol && c + 1 < cap`, then evaluates the
  swap-1×1 alternative, LAPACK-extension 1×1 alternative,
  Duff-Reid growth bound, and SSIDS scale-invariant det floor. On
  accept: writes `d_panel[c]=d11`, `d_panel[c+1]=d22`,
  `subdiag[k+c]=d21`, scales the L block via the 2×2 inverse
  (matches `do_2x2_update:2295-2300`), advances `c` by 2.
- New `peek_ahead_replay(target_col)` primitive extracted from
  `peek_ahead_column`. Walks committed pivots `q=0..n_committed` in
  q-ascending order honoring `subdiag[k+q] != 0` as a 2×2 marker.
  For pairs uses `axpy2_minus_unroll4_nofma` (bit-exact with
  `do_2x2_update`); for singletons uses
  `axpy_minus_unroll4_nofma` (bit-exact with `do_1x1_update`).
- `apply_blocked_schur` now takes `&[f64] subdiag` and gates the
  rank-bs fast path on `!has_2x2 && !any_zero_d && n_elim >= 2`.
  Fallback walks pivot pair-or-singleton (rank-2 axpy2 for pairs,
  rank-1 axpy for singletons).
- New `PanelStatus::ScalarFallbackPeekedNext` variant. When the
  inline 2×2 path bails AFTER `peek_ahead_replay(col+1)` (because of
  swap-1×1, LAPACK-extension, growth, or det-floor failure), col+1
  has been pre-updated. Caller switches `j_start` from
  `k+n_elim+1` to `k+n_elim+2` so `apply_blocked_schur` does not
  double-update col+1.

**Bit-exactness contract preserved:** the deferred path produces the
exact same f64 bit pattern as scalar `factor_frontal`. Rank-bs fast
path stays gated off when any 2×2 is in the panel stream because the
SIMD body's `acc -= a_q * s_q` per-q sequential accumulation is
bit-exact with sequential rank-1 axpys but NOT with axpy2's fused
`(a_0*s_0 + a_1*s_1)` add-then-sub ordering. Lifting this gate is
Phase B-2.

### The bug that almost shipped

After the initial implementation, all 16 `tests/blocked_ldlt.rs`
tests passed byte-identical, but `parity_swopf_0000` regressed to
residual `1.59e-1` (vs target `2.56e-15`) and `parity_himmelbj_0032`
flipped one inertia slot (`(44,13,0)` vs MUMPS `(43,14,0)`).

Root cause: `peek_ahead_replay(col+1)` was called BEFORE the
swap-1×1 / LAPACK-extension / growth / det-floor bail-out tests so
the tests would see post-update `arr`, `gamma_r`, `d22`. When ANY of
those tests failed and the path bailed to `ScalarFallback`, col+1
had been pre-updated. The downstream `apply_blocked_schur` used
`j_start = k+n_elim+1`, which then RE-APPLIED pivots `0..n_elim-1`
to col+1 — a double rank-1 update.

Fix: introduced `PanelStatus::ScalarFallbackPeekedNext` to signal
"col+1 was pre-updated". Caller switches `j_start` to `k+n_elim+2`.
The initial unit fixtures all happened to ACCEPT the inline 2×2 (no
bail-out), missing this code path. SWOPF and HIMMELBJ exercise it
because their KKT structure produces 2×2 candidates that fail one of
the secondary bail-out tests.

## Decisions Made

- W-2 2×2 inline accepts only the no-swap case. Symmetric-swap 2×2
  and rook-rescue 2×2 stay on the scalar path. (Matches plan §2.)
- New diagnostic flag `DISABLE_PANEL_INLINE_2X2` mirrors the
  existing `FORCE_SCALAR_FRONTAL` flag pattern. Both are
  `pub static AtomicBool` and default to `false` so production
  dispatch is unchanged. Intended use: parity-regression triage.
- `PanelStatus::ScalarFallbackPeekedNext` is a public-within-module
  variant added specifically to track the two-column peek-ahead
  state. Future Phase B work that pre-updates additional columns
  (e.g., for an inline rook-rescue path) will need additional
  variants or a count.
