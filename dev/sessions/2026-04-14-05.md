### 2.1 — Corpus expansion and measurement infrastructure

- **2.1.1** — Lifted the Phase 1-era `if mtx.n > 500 { continue; }`
  guard from the bench harness. The sparse pipeline had never been
  exercised on large matrices in Phase 1; lifting the filter in
  commit `c01235f` immediately exposed that sparse residuals were
  10⁴ to 10¹⁴ on n > 500 while canonical MUMPS / SPRAL produced
  machine-precision residuals. Root cause was missing MC64 scaling
  (fixed in 2.2.1).
- **2.1.2** — Sanity-checked a single large problem (BratuProblem
  n=1000) to confirm the correctness surface before building the
  large-corpus harness.
- **2.1.7** + **2.1.8** — Wrote the perf harness (per-matrix timing,
  oracle sidecar integration) and the Phase 2 baseline report
  (`dev/sessions/phase-2-baseline.md`). Every later Phase 2 landing
  is measured against this baseline.
- **2.1.3 / 2.1.5 / 2.1.6** — `collect_kkt_large`, grid benchmark,
  gas/water suites: *deferred*. The 154,588-matrix CUTEst KKT corpus
  already exceeded the exit-criterion sample size; adding more
  matrices would have diluted signal rather than strengthened it.

### 2.2 — Deferred correctness fixes

- **2.2.1** — Global MC64 matching-based scaling. Wrote the research
  note, implementation plan, and the scaling module
  (`src/scaling/mc64.rs`) with the Hungarian kernel. Integrated into
  `symbolic_factorize`, assembly, and solve. The ACOPP30 residual
  gap measured at Phase 1b exit (3.15e-2 vs MUMPS's 5e-14) closed
  on matrices where the Hungarian matching was full. Sessions
  `2026-04-12-02` through `2026-04-13-01`.
- **2.2.2** — Scaling-aware pivot rejection (column-relative 1×1
  threshold + Duff-Reid 2×2 growth bound). Replaces the fragile
  absolute-tolerance pivot test with a column-norm-relative one
  that survives the global rescale MC64 applies. Sessions
  `2026-04-13-01` through `2026-04-13-02`.
- **2.2.3** — Supernode amalgamation adjacency check. Phase 2.2.2
  validation flagged a plateau of 88 sparse-only failures; root
  cause was non-adjacent child supernodes being amalgamated
  together, producing frontal matrices that did not match the
  symbolic index space. Fixed by requiring direct parent-child
  adjacency in amalgamation. Session `2026-04-13-01`.
- **2.2.4** — Consensus re-run and delta: *partial*. The parity test
  suite (31 matrices gated on MUMPS inertia + residual) replaced
  the full consensus re-run for the Phase 2 validation loop.

### 2.3 — Pivoting improvements

- **2.3.2** — Delayed pivoting (SSIDS-style): the major Phase 2.3
  landing. Nine sub-steps across sessions `2026-04-13-02` through
  `2026-04-13-05`:
  1. Kernel plumbing: accept `may_delay` on the factorize call
  2. `n_delayed` out-parameter on the dense factor result
  3. Contribution-block layout: delayed cols in the top-left
  4. `may_delay` wired through `factorize_multifrontal`
  5. Parent-side delay assembly: re-entry of delayed cols at parent
  6. Solve-side `nelim` fix: solve walks the eliminated columns only
  7. `pivot_threshold = 0.01` restored as the sparse-caller default
     after the MC64 rescale justified the SSIDS/MUMPS threshold
  8. Sign-preservation at root-supernode fallback: `ForceAccept` at
     roots preserves the sign of tiny pivots instead of zeroing them
  9. Refinement-termination fix: `max_steps` 3→10 and an early exit
     when `||r||` stops decreasing
  Validation report: `dev/research/phase-2.3-validation.md`.
- **2.3.1** — Threshold partial pivoting (TPP): *subsumed*. TPP and
  delayed pivoting share the same threshold test; TPP without
  delays is just `may_delay = false` at the call site, which is
  the dense path.
- **2.3.3** — A posteriori pivoting (APP): *not implemented*.
  Delayed pivoting subsumes the pivot-correctness target; APP is a
  performance optimization whose value depends on how often the
  optimistic path succeeds. Deferred to Phase 3 if the dense kernel
  performance needs it.
- **2.3.4** — Test against baseline: validated through the parity
  panel + Phase 2.8.1 exit bench.

### 2.4 — Dense kernel performance

- **2.4.1a** — Contribution-block deferral: *rejected null result*.
  The hypothesis was that deferring contribution-block
  materialization until assembly time would drop allocator
  pressure. Implementation landed; benchmark showed no measurable
  improvement on the target partition and a 0.4% regression on the
  dense tail. Reverted in commit `6ca8832`. Documented in
  `dev/tried-and-rejected.md`.
- **2.4.2** — SIMD micro-kernel for Schur complement update. Added
  `pulp 0.22.2` as the SIMD backbone (decision recorded in
  `dev/decisions.md`), built `src/dense/schur_kernel.rs` with
  scalar kernels + test harness, then wired pulp-dispatched
  `axpy_minus_unroll4` and `axpy2_minus_unroll4` into
  `do_1x1_update` and `do_2x2_update`. Sessions
  `2026-04-14-01` through `2026-04-14-02`.
- **2.4.3** — Non-FMA bit-exact kernel. Initial 2.4.2 SIMD kernel
  used `mul_add_f64s` (single-instruction FMA). This produced
  4 sparse inertia mismatches from 1-ULP pivot-classification
  flips at the `zero_tol` boundary — mathematically-valid but
  behaviorally-different from the scalar reference. Rewrote the
  kernels to use separate `mul_f64s` + `sub_f64s` for bit-exact
  matching with the scalar path. Verified by `assert_eq!` unit
  tests across a length sweep up to 1024. Zero sparse inertia
  regressions; dense factor p90 2.27→1.86 vs MUMPS (−18.1%);
  sparse factor p90 3.18→2.82 (−11.3%). Session `2026-04-14-02`.

### 2.5 — Sparse pipeline performance

Phase 2.5 started with a priority inversion: the plan's 2.5.1
(Liu row-subtree column counts) turned out to be 2.6% of the
small-frontal budget, while `amd_order` was 39.8% with a fat tail
reaching ~9.3 ms on n=234 (DISCS family). The plan was reordered
in `dev/decisions.md` ("Phase 2.5 priority reordered") and the
new top-priority task became Phase 2.5.1′.

- **2.5.1′** — AMD + symbolic + numeric fixes. Session 04
  (`2026-04-14-04`). Diagnosis via an instrumented triage binary
  (`examples/triage_discs_amd.rs`) showed that `adj[a].contains(&b)`
  inside the fill loop was the sole bottleneck: on DISCS_0012
  (n=234, n_neigh≈62 per pivot) the code paid 778,064 `contains`
  calls for zero fill inserts, because the reachable set was
  already a clique. Six surgical fixes:
  1. Mark-array fill loop in `src/ordering/amd.rs` (O(deg³) →
     O(deg²) per elimination step)
  2. Dense-clique early exit when `neighbors.len() == n_remaining`
  3. Counting-sort `permute_pattern` (~7× faster on DMN15103
     fully-symmetric n=99)
  4. Removed a dead `for child_s in 0..n_snodes` loop in
     `src/symbolic/supernode.rs` (empty body, O(n²) per matrix)
  5. Etree renumbering from postorder in `src/symbolic/mod.rs`:
     `etree(P·A·Pᵀ)` is the post-renumbering of `etree(A)` when
     `P` is a postorder of `etree(A)`, so the second
     `from_pattern` call was redundant (O(n) renumber)
  6. Removed a dead `let _ = build_csc_transpose(&permuted);`
     call in `src/numeric/factorize.rs` (computed O(nnz) transpose
     then immediately discarded)
  Sparse small-frontal factor/MUMPS p90: 2.81 → 1.99 (target
  ≤ 2.0). Commit `9265f1f`.
- **2.5.1** — Liu row-subtree column counts: *demoted*. At 2.6% of
  the budget with no fat tail, implementing the faster algorithm
  would not have moved the exit-criterion partition. Left for
  Phase 3+ if it becomes budget-relevant.
- **2.5.2** — Parallelism on assembly tree: *not implemented*.
  The small-frontal partition is dominated by per-matrix overhead,
  not per-matrix compute, so per-matrix parallelism has negative
  return. Large-frontal parallelism is a Phase 3+ target.
- **2.5.3** — Better memory allocation: *not implemented*. Current
  `FactorBump` + `ContribPool` LIFO stack is sufficient for the
  exit criterion. Buddy allocator for parallel assembly is a
  Phase 3+ target.
- **2.5.4** — Fill prediction: *not implemented*. Current column
  counts already give a factor-1.2 slack estimate that is never
  the bottleneck.

## Decisions Made

Phase 2's decisions are all recorded in `dev/decisions.md`. The
Phase-2-level ones worth surfacing here:

- **Phase 2 scope correction** (2026-04-12): Phase 2 is not a
  Phase 3 POUNCE integration; it is corpus expansion, measurement
  infrastructure, deferred correctness, and performance on the
  existing (non-IPOPT-loop) bench. POUNCE integration stays in
  Phase 3.
- **pulp 0.22.2 as SIMD backbone** (2026-04-14): accepted as the
  SIMD portability layer after reviewing portable_simd stability,
  std::simd, and hand-rolled intrinsics. pulp's
  `Arch::default().dispatch()` pattern matches faer's approach
  and has the right architectural surface.
- **Non-FMA Schur kernel for bit-exact matching** (Phase 2.4.3):
  chose not to use `mul_add_f64s` / FMA instructions in the Schur
  update despite the lost IPC. 1-ULP pivot-classification flips
  at the `zero_tol` boundary cause inertia regressions that are
  not acceptable under the project's hard rule "inertia must be
  exactly correct — no tolerance on inertia counts."
- **Delayed pivoting pivot_threshold = 0.01 for sparse callers,
  0.0 for dense callers** (Phase 2.3 Step 7): the threshold is
  only meaningful when the kernel has a "landing zone" for
  rejected pivots. The dense path does not; the sparse path does
  via delayed pivoting and parent-side re-entry.
- **Phase 2.5 priority reordered** (2026-04-14): AMD is the
  small-frontal bottleneck, not column counts. Demoted the
  original plan's 2.5.1 (Liu row-subtree column counts) and
  inserted 2.5.1′ (AMD + symbolic fixes) in its place.
- **AMD stays exact minimum-degree with mark array, not real AMD**
  (Phase 2.5.1′): the mark-array fix clears the exit criterion;
  a real AMD port (approximate external degree + element
  absorption + quotient graph) is a larger surface-area change
  deferred to Phase 3+.
- **Symbolic etree built by renumbering postorder, not re-parsing**
  (Phase 2.5.1′): derived from the postorder-as-topological-
  relabeling identity. Halves the number of `EliminationTree::
  from_pattern` calls in `symbolic_factorize`.

## Abandoned Approaches

All Phase 2 abandoned approaches live in `dev/tried-and-rejected.md`.
The ones that cost the most iteration time:

- **Contribution-block deferral** (2.4.1a): null result. Did not
  improve the target partition; introduced a 0.4% dense tail
  regression. Reverted.
- **FMA-based Schur kernel** (2.4.2): correctness regression via
  1-ULP pivot classification flips. Replaced with non-FMA
  bit-exact variant in 2.4.3.
- **Single-run p90 as primary signal** (session 04): the
  etree-renumbering fix was reverted on a single noisy p90 delta
  (2.02 → 2.08), then re-applied after a 3-run median confirmed
  the real baseline was 2.12–2.14. Lesson: any decision based on
  a sparse-bucket p90 delta smaller than ~5% must be confirmed
  with at least a 3-run median.
