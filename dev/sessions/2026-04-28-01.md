# Session 2026-04-28-01

## Goal
Profile the current bench to evaluate whether feral should adopt faer for
performance, then act on findings. User reported general dissatisfaction
with feral's speed.

## Accomplished

### Profiling (faer evaluation)
- Built `src/bin/profile_hot.rs` — loops 7 representative KKT matrices
  (HS118, BATCH, AVION2, HAHN1, VESUVIO, CRESC132 + BATCH_0500) through
  the full pipeline ×200 reps with hand timers per stage.
- Stage breakdown: **sym=64%, factor=32%, solve=4%** of wall time.
- Sampled with `samply` at 4 kHz, symbolicated via `atos` against the dSYM.
  Top inclusive functions: `mc64::compute_matching` 26%, METIS ND 15%,
  `postorder` 10%, `dense::factor::factor_frontal_blocked_in_place` 11%,
  `axpy_minus_unroll4_nofma` (the actual SIMD inner kernel) **6.4% self**.
- Conclusion: faer cannot fix this. ≥51% of wall is graph algorithms
  (MC64 + METIS + postorder) that faer does not touch. Dense kernel
  upside ≤12% wall, and that hot kernel is already on faer's `pulp`
  primitive.
- Added `[profile.release] debug = true` to Cargo.toml so future
  samply runs symbolicate cleanly.

### Cross-checking against MUMPS / SSIDS / feral history
Dispatched four parallel research agents (mumps-expert, spral-expert,
two Explore agents). Findings:
- SSIDS defaults to NO scaling (`spral/src/ssids/datatypes.f90:233`).
- MUMPS SYM=2 defaults to MC64 BUT skips when
  `(missing_diag + zero_diag) < N/10`, falling through to cheap
  symmetric Ruiz equilibration (SIMSCA), at
  `mumps/src/dana_aux.F:1388-1416`. This is the canonical KKT-aware
  skip.
- feral's `Auto` scaling has Policy 4 fallback (correctness, not
  perf gating) and an arrow-detection `diag_only/n >= 0.3` route to
  MC64; it does NOT have MUMPS's inverse "skip when diagonal is
  mostly populated" rule.
- feral already has a `Solver` (src/numeric/solver.rs:85-208) that
  caches symbolic across same-pattern re-factorizations. The per-
  matrix `bench` does NOT use it — it walks 154k matrices through
  the FREE-FUNCTION API and recomputes symbolic on every call. The
  64% sym share is an artifact of the bench harness, not real
  production cost.

### New bench: `bench_solver_corpus`
Wrote `src/bin/bench_solver_corpus.rs` to measure the realistic
IPM-tail picture: walk the corpus, group `<FAM>_NNNN.mtx` by family,
per family run two scenarios (one persistent `Solver` vs free-function
loop) and report per-family + aggregate speedups. Env knobs:
`FERAL_BENCH_FAMILY_CAP` (default 64), `FERAL_BENCH_MIN_ITERS`
(default 4), `FERAL_BENCH_KKT_ROOT`, `FERAL_BENCH_FAMILY_FILTER`,
`FERAL_BENCH_MAX_FAMILIES`.

## Decisions Made

1. **`bench_solver_corpus` is the new perf-tuning ground truth.**
   The per-matrix `bench` (154k matrices via free functions) measures
   an unrealistic workload — it pays symbolic on every call, where
   real IPM tails reuse symbolic across hundreds of refactors per
   solve. Optimization decisions that target symbolic-phase cost
   must cite this bench, not the per-matrix bench. Per-matrix bench
   retained for inertia/residual correctness sweeps and oracle ratio
   comparisons on individual matrices.

2. **Do not adopt faer.** Profile shows ≥51% of wall in graph
   algorithms faer does not address; dense-kernel headroom is
   ≤12% and that path is already on `pulp` (faer's SIMD primitive).

Both decisions appended to `dev/decisions.md`.

## Next Session Should

In order of estimated wall savings against the new solver-corpus bench:

1. **Implement MUMPS's missing-diagonal MC64 skip** — when
   `(missing_diag + zero_diag) < N/10`, route to InfNorm (or a
   symmetric Ruiz equivalent) instead of MC64. Cite
   `mumps/src/dana_aux.F:1388-1416`. Validate with the existing
   cost/benefit probe pattern (`src/bin/diag_compress_costbenefit.rs`)
   so we don't repeat the rejected `MIN_N_FOR_COMPRESSION` mistake.
   Estimated 5–15% on `bench_solver_corpus`, more on the per-matrix
   bench.

2. **Strengthen `PatternFingerprint`** at `src/numeric/solver.rs:62-76`
   with a structural hash (e.g. XXH3 of `col_ptr || row_idx`).
   Eliminates the false-cache-hit footgun (two structurally different
   matrices with identical shape currently treat as same pattern).
   Should not change throughput on legitimate IPM workloads but
   removes a correctness landmine for general callers.

3. **Profile `bench_solver_corpus` with samply** to see what
   dominates *after* symbolic is amortized. The dense kernel +
   assembly should be much more visible. This re-baselining is
   needed before deciding whether further dense-kernel work is
   worth pursuing.

4. **Investigate BENNETT5 0.82× regression** — the Solver path is
   slower than free-function on this one family. Likely small-matrix
   measurement noise but worth confirming.
