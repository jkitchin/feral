# Session 2026-04-25-03

## Goal

Phase 2.12 — implement SSIDS-style column renumbering for amalgamation
to close the under-merging gap on tiny-IPM tail matrices, then run the
graduation gate (corpus bench within ±5%) and either flip the default
or land Renumber as opt-in with a decision-log entry.

## Accomplished

**Implementation (commit 8a20315):**

- `AmalgamationStrategy` enum on `SupernodeParams`. Default `Adjacency`
  (unchanged); opt-in `Renumber` runs:
  1. `predict_merges(etree, col_counts)` — symbolic merge oracle
     mirroring the SSIDS trivial-chain + size rule.
  2. `biased_postorder(etree, bias)` — emits non-merging children first,
     merging children last (`src/ordering/postorder.rs`).
  3. Reverse iteration in `find_supernodes` Step 2 — parent
     `first_col` shrinks monotonically, opening adjacency for the
     next lower-index sibling. Forward iteration only catches one
     merge per multi-child parent.

- `tests/column_renumbering.rs` — 4 structural tests, all pass: arrow
  + bushy fan collapse to 1 supernode under Renumber (was n supernodes
  under Adjacency); tridiagonal Renumber count ≤ Adjacency count;
  perm remains a bijection.

- `tests/column_renumbering_parity.rs` — 3 numeric parity tests, all
  pass: inertia triple matches across strategies on arrow SPD,
  bordered KKT, and the real `ACOPR30_0067` corpus matrix; refined
  residuals stay below 1e-6 on every fixture.

- Diagnostic binaries: `src/bin/diag_strategy_compare.rs` (5-run
  median Phase-2.10 profiler comparison) and an extension to
  `diag_amalgamation` to print both strategies side by side.

## Decisions Made

**Initial decision (commit 8a20315): keep Renumber opt-in.** Applied
the plan's ±5% corpus-median graduation gate; Renumber failed it
(sparse p50 0.30 → 0.33).

**Reversed (commit cd9af06): `AmalgamationStrategy::default()` is now
`Renumber`.** User-prompted re-evaluation: feral's spec-stated mission
is interior-point KKT solves, the IPM tail is what matters, and a
30-67% factor-time reduction on that tail is unambiguously a net win
even at the cost of +10% on the long tail of small CUTEst-Hessian
matrices. Both decision-log entries stand (append-only); the second
supersedes the first.

`Adjacency` retained as the opt-in escape hatch.

## Next Session Should

With Renumber now default, the tail-investigation lever shifts: the
new sparse Top-10 worst is KIRBY2_* (10.64x at worst) and MUONSINE_*
(7.12x), at 5-10x MUMPS. ACOPR30/CRESC100 are no longer in the
Top-10 — those targets were absorbed by Phase 2.12.

Candidate next directions, in priority order:

1. **Diagnose the new tail (KIRBY2 / MUONSINE).** Different families,
   probably a different bottleneck than the bushy-tree problem. Run
   a Phase-2.10-style profiler dive on `KIRBY2_0007` and `MUONSINE_0000`
   to pin the bottleneck (kernel? amalgamation? assembly? scaling?).
   This is the natural follow-on now that the IPM-KKT tail is the
   shorter problem.

2. **Recover the +10% median regression on small CUTEst Hessians.**
   The cost is real even if smaller-impact than the tail wins.
   Hypothesis: wider supernodes feed denser kernels that don't
   amortize on tiny-n. Either pull predict_merges back when
   col_counts are tiny (skip renumber when the prediction yields
   no merges that cross >1 column), or shape-dispatch (Phase 2.13)
   based on n / fundamental-supernode count.

3. **`AmalgamationStrategy::Auto`** (parallel to
   `OrderingPreprocess::Auto`): cheap shape predicates decide per-
   matrix between Adjacency and Renumber. Recovers (1) and (2)
   simultaneously if the predicates are right.

4. **Re-run the corpus bench end-to-end with `Renumber` as default**
   to lock in the reference numbers for downstream compares (the
   `/tmp/feral_bench_renumber.txt` I have is from the gate run and
   should be considered a reference, not a checked-in artifact).
