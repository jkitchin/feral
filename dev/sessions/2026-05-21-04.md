#47 **still reproduces** — ~2.1× wall — with the #46 cascade fix in
place. It is not the #46 cascade.

**Root cause (pinned):** explicit zeros defeat the MC64 value-bounded
scaling cache (Track B2). New probe `probe_explicit_zeros` factors the
iter-0 KKT 4× on one warm `Solver`:

```
stripped:            cold 434ms -> warm 15/14/15ms    symbolic_calls=1  mc64_cache_hits=1,2,3
explicit zeros kept: cold 468ms -> warm 359/359/360ms  symbolic_calls=1  mc64_cache_hits=0,0,0
```

- Cold factor fine either way (~450 ms) — not a cascade, not fill.
- Symbolic analysis **is** reused either way (`symbolic_calls` stays 1)
  — the pattern fingerprint / symbolic cache is not the problem.
- The MC64 cache **never hits** with explicit zeros
  (`mc64_cache_hits` stays 0) — the Hungarian match reruns every
  factor, ~345 ms of the ~360 ms warm refactor. Stripped, the cache
  hits and the warm refactor collapses to ~15 ms (24× gap).

The explicit-zero `(2,2)` diagonals make the cache's
diagonal-dominance value-bound test (`mc64_value_bound_passes`) fail,
so it conservatively recomputes scaling every iterate. That recompute
is the entire 2×. The issue's hypothesised mechanism ("the #46 fast
path keys on structural absence of the diagonal") is **wrong**.

Posted the pinned root cause as a comment on #47
(`#issuecomment-4513154794`). #47 stays open; the fix is option (a) —
feral drops explicit zeros in `from_triplets` / the symbolic phase.

## Correction (post-checkpoint)

The "Pinned root cause" section above attributes the cache miss to the
MC64 **value-bound test** failing on explicit-zero `(2,2)` diagonals.
That mechanism is **wrong** — corrected after reading the cache code
with the human. The value-bound test (`src/scaling/value_bound.rs`)
already disqualifies explicit-zero-diagonal rows (`qualifying_rows`;
test `explicit_zero_diagonal_disqualifies_row`) and is never reached
here because the cache slot is `None`.

Actual mechanism: explicit zeros tip MC64's symmetric matching into
`Mc64ScalingDegenerate` (scaling spread > `1/EPS` — the ≈ 3e82 blow-up
documented at `src/scaling/mod.rs:171`, research note
`kkt-mc64-scaling-blowup-2026-05-20.md`, issue #45) →
`ScalingInfo::Mc64FallbackToInfnorm`, **not** `Applied`. The B2 cache
populates only on `Applied` (`solver.rs:803`); on anything else it is
set to `None` (`solver.rs:812`), so it is never populated and the full
~345 ms Hungarian reruns — and is discarded — every warm factor. That
wasted Hungarian is the 2×.

Evidence: `probe_explicit_zeros` re-run with `scaling_info`
instrumentation — kept matrix reports
`scaling_info=Mc64FallbackToInfnorm{Mc64ScalingDegenerate}` and
`mc64_fallbacks` incrementing on every call, `mc64_cache_hits` stays 0;
stripped matrix reports `scaling_info=Applied`, `mc64_cache_hits=1,2,3`.
See journal §18:45. Conclusion unchanged: the fix is for feral to drop
explicit zeros — and since the stripped matrix's MC64 returns `Applied`,
that removes the *cause* of the degeneracy, not just a symptom.
