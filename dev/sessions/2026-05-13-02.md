## Decisions Made

- **Step 2 dispatch via do_*_update fast-paths, not a full driver port.** The plan called for porting `lblt_panel_frontal`'s body into `block_ldlt32` with `BLOCK_SIZE=32` const-folded. Inspection of the panel path revealed a simpler dispatch that achieves the same coverage: the existing `factor_frontal` already runs the eager BK loop with `do_1x1_update` / `do_2x2_update` per pivot, so adding fast-paths in those helpers is sufficient to fire the SIMD body. The full driver port would have been ~300 lines of duplicated BK pivot logic for no additional coverage.
- **32×32 dispatch bypasses the panel path because the panel's quad kernel is unreachable on full panels.** `apply_blocked_schur_panel` is called with `j_start = k + n_elim`. When `bs == ncol == 32`, the panel reaches `n_elim == 32` and `j_start == nrow`, so `apply_blocked_schur` early-returns. All trailing-update FLOPs on 32×32 fronts in the panel path are done by per-column `peek_ahead_replay` (single-column axpy). Routing through `factor_frontal` puts those FLOPs through the quad SIMD body.

## Abandoned Approaches

- **Full Step 2 driver port (`lblt_panel_frontal` line-for-line port into `block_ldlt32` with `BLOCK_SIZE=32` const-folded).** Not pursued. The fast-path indirection achieves the same SIMD coverage with ~30 lines of code instead of ~300. If profiling shows that the per-pivot `apply_blocked_schur_panel` dispatch overhead matters at this front size, a future session can revisit.
