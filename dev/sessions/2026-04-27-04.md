# Session 2026-04-27-04

## Goal
W-3b from `dev/plans/dense-kernel-speedup.md`: eliminate the
`ff.contrib.clone()` the multifrontal driver was performing on every
supernode. For CHAINWOO_0000's 1984-row root supernode the contribution
block is `(nrow-ncol)^2 * 8` bytes ≈ 30 MB; cloning that on every
factorization is pure malloc traffic.

Tooling side-quest: relocate `FERAL_KKT_FILTER` from a post-load to a
load-time filter so targeted single-family bench runs become
sub-second instead of waiting for the 156k corpus to load first.

under repeated IPM-style factorizations) than for single-run wall
time. W-1 and W-2 are now the unblocked levers.

## Decisions Made
- **`mem::take` over a borrowed output buffer.** The kernel already
  produces `contrib` as a freshly-allocated `Vec`; transferring
  ownership instead of cloning is a one-line change with zero kernel
  API churn. Pooling the `contrib` buffer through `FactorWorkspace`
  would be a larger refactor (the kernel would need to take an
  `&mut Vec<f64>`) — defer until measurements show it matters.
- **Load-time bench filter is a tooling-only change.** It does not
  alter solver behavior or ordering of the corpus pass; it only skips
  non-matching directory entries earlier in the walk. The post-load
  filter is preserved as a defensive net.
