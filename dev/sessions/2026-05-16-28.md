## Decisions Made
- Place stress-smoke in the existing ci.yml rather than a new
  workflow file. Reason: shares the same trigger surface
  (push + pull_request), keeps the toolchain pin in lockstep
  with the `check` job by visual proximity, and the existing
  ci.yml file is short enough that one more job does not
  hurt readability.
- Use `--max-n 1000` literally as the cap (not tighter), since
  the python-only wall-clock at that cap is 0.7s. There is no
  performance reason to go smaller; the cap exists to keep
  future SuiteSparse smalls from blowing the budget, not to
  trim today's runtime.
- Cache key strategy: per-manifest-content hash, no branch
  fallback. Reason: the matrices/ tree is large and content-
  addressable; a manifest hash change always implies new rows
  to fetch, and reusing a stale cache from another branch
  would silently miss the new row.
