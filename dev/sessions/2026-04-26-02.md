# Session 2026-04-26-02

## Goal

Item 0 from the prior session's "Next Session Should" list:
**Bench sparse-loop OOM regression.** `cargo run --bin bench --release`
SIGKILLs after the dense pass with `--- Sparse solver validation ---`
as the last printed line, blocking the standard session-end protocol.
Triggered when the bench's KKT loader expanded from one root to three
(167,614 matrices vs 154,588 in the prior baseline).

## Accomplished

**Two-layer fix landed.**

1. `KktEntry.csc` is now `Option<CscMatrix>`; the sparse loop iterates
   `kkt_entries.iter_mut()` and calls `entry.csc.take()` so the CSC
   arrays drop at end of each iteration. Same trick for the in-between
   matrix drop (`entry.matrix = None` already lived between the two
   loops). This bounds in-flight working set growth.

2. The default corpus is scoped to the historical `kkt` root via a new
   `FERAL_KKT_ROOTS` env var. `FERAL_KKT_ROOTS=all` opts into the
   expanded corpus (`kkt + kkt-expansion + kkt-mittelmann`) on beefy
   hardware; comma-separated subset also accepted. The implicit
   three-root walk that triggered the regression is gone.

The first fix alone was insufficient — under default scope (167k
matrices), the bench reached `--- Sparse solver validation ---` and
ran ~8 minutes silently before being killed at ~30 GB RSS. The CSC
working set at sparse-loop entry is dominated by cumulative
allocations independent of per-iter take/drop. macOS jetsam tolerated
30 GB for several minutes but eventually killed the process under
sustained pressure (Dropbox renderer + claude-code + rustc account
for another 10+ GB resident).

The second fix restores the 2026-04-25 baseline corpus scope and
unblocks the standard workflow. Both fixes ship together because
fix 1 is the cleanup that makes fix 2's defaults sufficient and is
the right shape for any future expanded-corpus runs.

**Tests**: 333 passed, 0 failed, 0 ignored (`cargo test --release`).
