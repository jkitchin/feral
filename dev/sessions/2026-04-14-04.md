## Accomplished

### Diagnosis (triage binary)

Built `examples/triage_discs_amd.rs` — an instrumented copy of
`amd_order` with per-phase µs counters and scalar counters for
`contains` calls, fill inserts, and per-step neighbor counts. Ran
on five representative matrices:

| matrix        |  n  |  nnz  | total µs | fill% | contains | inserts |
|---------------|----:|------:|---------:|------:|---------:|--------:|
| DISCS_0012    | 234 | 29346 |    12326 | 87.6% |   778064 |       0 |
| DISCS_0000    | 234 |  1286 |      130 | 11.5% |     7957 |     625 |
| LAKES_0000    | 324 |  1116 |      121 |  0.0% |     1165 |     341 |
| DMN15103_0000 |  99 |  9801 |     1337 | 79.7% |   156849 |       0 |
| GROUPING_0000 | 225 |  2725 |      615 | 61.3% |    71062 |    2440 |

**Bottleneck:** `adj[a].contains(&b)` inside the fill loop is an
O(deg) linear scan. On near-dense inputs the fill set is already a
clique, so every `contains` returns `true` after scanning the full
adjacency vector — DISCS_0012 pays 778k lookups for zero inserts.
DMN15103_0000 is fully dense (9801 = 99²) so every step is a no-op
but still costs 1.3 ms.

### Five fixes applied

1. **AMD mark array** (`src/ordering/amd.rs`) — scratch `Vec<bool>`
   of size n, reused across elimination steps. Mark `adj[a]` once,
   check/insert with O(1) lookups, unmark before next outer iteration.
   Drops fill phase from O(deg³) to O(deg²) per step.

2. **AMD dense-clique shortcut** (`src/ordering/amd.rs`) — when
   `neighbors.len() == n_remaining` the pivot is connected to every
   live node, so eliminating it forms a clique among all survivors.
   Push survivors in any order and return. Short-circuits DMN15103_0000
   entirely; cuts DISCS_0012 to just the first few steps.

3. **Counting-sort `permute_pattern`** (`src/ordering/amd.rs`) —
   replaced `Vec<Vec<usize>>` + sort + dedup with a two-pass
   counting-sort layout (count → prefix sum → fill), plus one
   per-column `sort_unstable` to preserve the sorted-column invariant.
   ~7× faster on DMN15103_0000 (fully symmetric, n=99); each entry
   copied once instead of twice then deduped.

4. **Dead loop in supernode detection** (`src/symbolic/supernode.rs`)
   — removed a `for child_s in 0..n_snodes` loop that called
   `find_root` on every candidate and then did nothing with the
   result (empty body). O(n²) wasted work per matrix. Snode max
   dropped 507→68 µs; share 7.3% → 1.2%. GROUPING family fell off
   the top-30 worst offenders list.

5. **Etree renumbering from postorder** (`src/symbolic/mod.rs`) —
   replaced the second `EliminationTree::from_pattern` call (on the
   final permuted pattern) with an O(n) renumbering of the
   AMD-permuted etree through the postorder:

   ```rust
   let final_parent: Vec<Option<usize>> = (0..n)
       .map(|new| {
           let old_amd = post[new];
           amd_etree.parent[old_amd].map(|old_par| post_inv[old_par])
       })
       .collect();
   ```

   Postorder is a topological relabeling of the elimination tree,
   so the structure is preserved and only node labels change. 3-run
   medians show a ~3% small-frontal p90 improvement
   (before 2.12/2.12/2.14 → after 2.03/2.06/2.08).

6. **Dead transpose computation** (`src/numeric/factorize.rs`) —
   `build_csc_transpose(&permuted)` was computed and immediately
   discarded with `let _ = ...`. A misleading comment claimed it was
   "kept for future amap-based assembly optimization" but the value
   was thrown away so no caller benefits. Removed both the call and
   the helper function (no other callers).

## Decisions Made

- **AMD stays exact minimum-degree with mark-array optimization.** Not
  upgraded to approximate external degree + element absorption (real
  AMD). The mark-array + dense-clique shortcut brings exact-degree to
  within the Phase 2.8.1 budget, and a real AMD port is a much larger
  surface-area change we don't need to take on right now. Recorded in
  `dev/decisions.md`.

- **`permute_pattern` guarantees sorted columns.** Downstream code
  doesn't strictly need it, but the previous impl produced sorted
  output and some callers may implicitly rely on the invariant. The
  counting-sort rewrite preserves it with one per-column
  `sort_unstable` pass. Recorded in `dev/decisions.md`.

- **Symbolic factorization computes the final etree by renumbering,
  not re-parsing.** From_pattern is only called once (on the
  AMD-permuted pattern); the final etree is built in O(n) from the
  postorder. Relies on the postorder-as-topological-relabeling
  property. Recorded in `dev/decisions.md`.

## Abandoned Approaches

- **Single-run p90 as primary signal during iteration.** The etree
  renumbering fix was initially reverted after a single-run reading
  showed 2.02 → 2.08 (which looked like a regression), then
  re-applied after 3-run sanity check showed the pre-fix baseline
  was actually 2.12–2.14. Lesson: any decision based on a p90 delta
  smaller than ~5% must be confirmed with a 3-run median. Logged in
  `dev/tried-and-rejected.md`.
