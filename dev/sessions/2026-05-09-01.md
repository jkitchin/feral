# Session 2026-05-09-01

## Goal

Triage and fix issue #4 (janosh): `CscMatrix::from_triplets()` documents
that entries must be lower-triangle (`row >= col`) but silently accepts
upper-triangle entries. Downstream consumers
(`symmetric_pattern`, multifrontal assembly) assume lower-triangle
storage, so the same symmetric system described with upper- vs
lower-triangle triplets produced different solve results — silent
correctness bug at the input boundary.

## Accomplished

### `from_triplets` and `validate` reject upper-triangle entries

- `CscMatrix::from_triplets` now returns `FeralError::InvalidInput` if
  any triplet has `row < col`, naming the offending triplet by index
  and `(row, col)` (`src/sparse/csc.rs:78-84`).
- `CscMatrix::validate` now also rejects `row_idx[k] < j` so any path
  that bypasses `from_triplets` (e.g. direct field assignment) is
  covered (`src/sparse/csc.rs:177-183`).
- The fix preserves the documented contract; no auto-normalization
  (transposing upper triplets to lower) — that would mask user bugs
  such as accidentally feeding a non-symmetric matrix expecting full
  storage.

### Regression tests

- `test_from_triplets_rejects_upper_triangle` — exact input from the
  issue; asserts `Err` and that the message mentions
  `"upper-triangle"`.
- `test_validate_rejects_upper_triangle_row` — direct field
  manipulation path; asserts `validate` rejects.

### Pre-existing call sites updated

Audited all `CscMatrix::from_triplets(...)` calls in `src/`, `tests/`,
`benches/`, `examples/` (regex over rs files). Only two sites passed
upper-triangle literals — both in `solver.rs` fingerprint tests that
deliberately constructed structurally-distinct sparsity patterns:

- `f2_fingerprint_distinguishes_same_n_nnz_different_pattern`
  (`src/numeric/solver.rs:560`): rows `[0, 2, 1]` → `[2, 2, 2]`.
- `f3_fingerprint_distinguishes_different_col_ptr`
  (`src/numeric/solver.rs:580`): rows `[0, 0, 1, 2]` → `[0, 1, 1, 2]`.

Both tests still exercise their original intent (distinct fingerprints
for distinct sparsity) using lower-triangle row patterns.

`io/mtx.rs` already normalizes upper-triangle entries to lower before
calling `from_triplets` (parse_mtx, line 167-173), so the MTX path was
unaffected.

## Abandoned Approaches

Considered but rejected: auto-normalizing upper-triangle triplets to
lower (option 2 in the issue text). This would silently coerce
non-symmetric inputs into a symmetric storage form, masking the
user's actual bug class (e.g. passing a full general-pattern matrix
expecting the solver to handle it). Rejecting at the boundary with a
clear message aligned better with the documented "symmetric
indefinite, lower-triangle storage" contract. (Not appended to
`tried-and-rejected.md` — this was a design choice during this fix,
not a tried-and-failed implementation.)
