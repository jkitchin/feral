## Accomplished

### 1. Calibration probe binary

`src/bin/calibrate_par_min_flops.rs` — sweeps Poisson-KKT
problem size, times best-of-N sequential vs forced-parallel
factor calls inside a single persistent `rayon::ThreadPool`,
reports per-row `(K, n_kkt, n_snodes, multichild, est_flops,
seq_ms, par_ms, par/seq, decision)`.

Forced framing: the probe calls
`factorize_multifrontal_supernodal_parallel` and
`factorize_multifrontal_supernodal_with_workspace` directly,
bypassing `should_parallelize_assembly`. This separates *driver
performance* from *gate decision*.

- **Break-even**: `est_flops ≈ 6×10⁶` (K=40).
- **Parallel wins ≥1.2×**: `est_flops ≥ 1.2×10⁷` (K=50).
- **Parallel wins ≥1.35×**: `est_flops ≥ 2.3×10⁷` (K=60).
- **Current `PAR_MIN_FLOPS = 10⁸` is ~5× conservative on this
  hardware** — misses K=50/60/80 wins of 1.2×–1.65×.

Full table and methodology in

## Decisions Made

- **Do not change the `PAR_MIN_FLOPS` const default in this
  session.** The calibration is on one workload family
  (Poisson-KKT) and one CPU (M4 Pro). Issue #19's reporter saw
  the 12× wall regression on non-M4 hardware on robot_1600 —
  a different problem family. Without their probe data we can't
  verify that any lower threshold (e.g. 3×10⁷) would be safe on
  their workload mix. The existing
  `NumericParams::min_parallel_flops` knob already lets consumers
  override the default, so this is a default-policy change, not
  a capability gap.
