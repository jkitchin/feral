### Small-n `compute_scaling` fast path — SHIPPED

Implementation:

1. **`compute_infnorm_dense(&SymmetricMatrix)`** in
   `src/scaling/infnorm.rs`. Knight-Ruiz iteration over
   column-major lower-triangle storage. Same `max_iter = 10`,
   `tol = 1e-8` as the sparse path. Bit-exact with sparse KR on
   every fast-path-gate matrix because (a) `CscMatrix::to_dense_into`
   writes only the stored entries (rest are zero) and (b) the inner
   `if v > row_max[i]` guard is a no-op for `v == 0.0`.

2. **`compute_scaling_dense_fast(matrix, sym, strategy)`** in
   `src/scaling/mod.rs`. Dispatch:
   - `Auto` | `InfNorm` → `compute_infnorm_dense(sym)`.
   - `Mc64Symmetric` | `Identity` | `External` →
     `compute_scaling(matrix, strategy)` (honored as-is).

3. **Reordered `dense_fast_factor_with_workspace`** in
   `src/numeric/factorize.rs`. Densify *before* scaling, then
   pass the dense buffer to the fast-path scaling routine, then
   apply `D · A · D` in place, then factor.

Tests added (all run as part of the lib build):

- `scaling::infnorm::tests::dense_matches_sparse_on_arrow_6x6` —
  arrow pattern (degree-1 rows + a dense last row).
- `scaling::infnorm::tests::dense_matches_sparse_on_dense_5x5` —
  fully dense 5×5 (the dense-fast-path's primary regime).
- `numeric::factorize::tests::dense_fast_factor_auto_matches_explicit_infnorm_bitwise`
  — bit-equal `D_diag` across `ScalingStrategy::Auto` vs explicit
  `ScalingStrategy::InfNorm` on an n=24 in-gate matrix. Guards
  against accidental re-introduction of Auto's MC64 branch in
  the fast path.

## Decisions Made

None new. The fast-path routing decision (Auto → dense InfNorm
when the gate fires) is consistent with the existing
`pick_scaling_strategy` behavior on dense matrices (which already
picks InfNorm at high density); the change is essentially a
faster realization of what Auto already wants for fast-path-gate
matrices, plus a deliberate short-circuit of the rare case where
a small-dense matrix has enough degree-1 columns to trip the 0.30
arrow-KKT threshold.
