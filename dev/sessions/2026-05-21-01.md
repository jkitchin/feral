### Track A — pinene_3200 iter 6-9 blowup characterized (A1)

- Established this is **issue #8**, and that the warm-state hypothesis
  was already disproved (2026-05-17): the cascade is structural to the
  iter-N matrix's *numeric content*, standalone ≈ warm.
- Ran `diag_pinene_pivot_cliff` (per-supernode 2×2 / delayed-pivot
  stats) on iterates 0008 and 0009 (n=127995, nnz=733k). **Direct
  evidence — the root front is a fully dense block, ~14 % of n:**

  | metric            |   0008 |   0009 |
  |-------------------|-------:|-------:|
  | root front nelim  |  15446 |  17538 |
  | root n_2x2        |   7064 |   8221 |
  | root n_delayed_in |  15341 |  17433 |
  | nnz_L             | 128.5M | 165.7M |
  | total_delayed     | 116541 | 133648 |

  The last supernode (node 10486) is a dense ~17.5k×17.5k root front
  (`nelim = nrow`, no trailing block); ~91 % of its columns are in
  2×2 pivots. Cost ≈ a dense 17.5k LDLᵀ ≈ 1.8e12 flops — that is the
  208 s. The three supernodes below it (10483/10484/10485, each
  ~3.8k–18k columns wide) eliminate only 4 / 11 / 493 columns — pure
  delay conduits feeding everything into the root.
- This is the **zero-(2,2)-block KKT cascade** (issue #46 family): as
  the IPM drives δ_c→0, every (2,2) row must be 2×2-paired and the
  pairing cannot be made locally, so columns delay all the way up the
  tree. The cascade worsens monotonically across IPM iters (root
  15446→17538, nnz_L +29 %). The probe's stale header ("0008 fast:
  44ms") is wrong — standalone 0008 now factors in ~3 min (≈ its warm
  135.7 s), confirming the cascade is structural to the matrix.

## Track A2 investigation (this session, after the checkpoint)

The A2 investigation continued past the checkpoint and reframed the
fix:

- **`CB=1` on pinene → 100× speedup, 493.9 s → 4.86 s** (iter 6-9
  fronts from 64-208 s to ~45 ms). Cascade-break *does* break this
  cascade — but **iter 9 returns `WrongInertia`** (got 63999/63995/1,
  want .../0): the `PerturbToEps` floor 1e-10 sits below the inertia
  zero-tolerance, so a force-accepted near-zero pivot still
  classifies as zero on the most-converged iterate. Inadmissible —
  the *same* failure issue #46 recorded for `allow_delayed_pivots
  =false`.
- This **is the issue #46 zero-(2,2)-block cascade** — and
  `probe_issue46_supernode pinene_3200_0009.mtx` localized the gap
  *beyond* #46. Co-location is good (60122/63995 = 93.9 % of MC64
  pairs adjacent; symbolic `factor_nnz_estimate` 2.4M) so #46's gaps
  A/B are clear. Yet the production path cascades to 165.7M nnz_L
  (69×, 159 s). `panel_diag` dominant fallback:
  **`fallback_2x2_need_swap_or_bound = 31542`** — the *same* fallback
  that was #46's CHO culprit. #46 widened the 2×2 *search* so the
  kernel finds the co-located partner; on pinene the partner *is*
  found but the `{k,k+1}` 2×2 candidate is **rejected at the
  numerical stability gate** → delayed → cascade. Static pivoting
  force-accepts the same columns as 1×1 (52 ms, 1.25× — but inertia
  wrong, 64001/63994).
