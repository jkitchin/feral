### Step 4b — real peek-ahead blocked panel kernel

`src/dense/factor.rs`:

- New `PanelStatus` enum (line 567) — `Full` vs `ScalarFallback`
  termination codes.
- `factor_frontal_blocked` rewritten (line 780) — outer loop:
  - `may_delay || bs < 2 || ncol <= bs` → delegate to `factor_frontal`
  - else: loop calling `lblt_panel_frontal` for full panels,
    `scalar_pivot_step` for tails and ScalarFallback recoveries.
  - After a panel returns, call `apply_blocked_schur` with
    `j_start = k + n_elim` on `Full`, `k + n_elim + 1` on
    `ScalarFallback`.
  - Final L / D / contrib extraction is the same allocation pattern as
    scalar (guaranteeing byte-identical output structure).
- `lblt_panel_frontal` (line 993) — processes up to `bs` 1×1 pivots with
  peek-ahead; returns `(n_elim, PanelStatus)`. Terminates on: `akk`
  below BK77 gamma0 threshold (ScalarFallback → scalar handles 2×2),
  rejected pivot with may_delay==false (panel treats it as a zero pivot
  and continues), or panel exhaustion (Full).
- `peek_ahead_column` (line 1098) — replays pending rank-1 updates for
  one trailing column in ascending pivot index q using
  `schur_kernel::axpy_minus_unroll4_nofma`.
- `apply_blocked_schur` (line 1133) — deferred Schur complement with
  `j_start` parameter (see bug #1 below).

The replay order matches scalar's traversal byte-for-byte: scalar
accumulates into `(i,j)` via pivot-outer / column-inner; replay
accumulates via column-outer / pivot-inner — but both traverse pivots
in ascending index order for each `(i,j)` and both use
`axpy_minus_unroll4_nofma`, so IEEE 754 rounding matches exactly.

### Bugs fixed en route

1. **1-ULP bit-parity break at the 2×2 block boundary.**
   Test `test_2x2_at_block_boundary` (n=128, zeros on diagonal at
   {63,64}) failed with `l[8129]` off by 1 ULP
   (`-3.8192811046161957e-7` vs `-3.8192811046007694e-7`).
   Root cause: when the panel returns `ScalarFallback` at column
   `c = n_elim`, the peek-ahead had already applied pivots `0..n_elim-1`
   to column `k+n_elim`; `apply_blocked_schur` was then re-applying
   the same updates to that column, producing a double rank-1 update
   on one column before the scalar step fixed it.
   Fix: added `j_start` parameter to `apply_blocked_schur`;
   `factor_frontal_blocked` passes `k+n_elim+1` on ScalarFallback,
   `k+n_elim` on Full. All 6 tests bit-exact after this.

2. **Clippy `needless_range_loop`** on the new replay loops. Replaced
   `for q in 0..c { let d_q = d_panel[q]; ... }` with
   `for (q, &d_q) in d_panel.iter().enumerate().take(c)`. Same
   semantics; clippy clean.

## Decisions Made

- **Panel terminates on 2×2 candidates** (plan §Risks #1 option b).
  The panel handles only 1×1 pivots; any 2×2 candidate returns
  `PanelStatus::ScalarFallback` and the caller runs one
  `scalar_pivot_step` before optionally re-entering. This keeps the
  replay logic simple and bit-exact, at the cost of one scalar step per
  2×2.
- **`apply_blocked_schur` takes a `j_start` parameter**, not a fixed
  `k + n_elim`. This is what prevents the double-update on the
  ScalarFallback column. Not in the original plan but required for
  bit-parity.
- **Supernodal driver NOT yet switched to the blocked path**. Step 4b
  proves the kernel; wiring it into `src/sparse/multifrontal/` is a
  follow-up because the API surface (`may_delay`, delayed-column
  handling) needs Step 5 first.

## Abandoned Approaches

None this session. Considered and kept:
- **`j_start` tagging instead of returning `n_elim` semantics**: simpler
  and keeps `PanelStatus` a 2-variant enum. Alternative was
  `PanelStatus::ScalarFallback { peek_ahead_done_through: usize }`
  which leaks internals.
