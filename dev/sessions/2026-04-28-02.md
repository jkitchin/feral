### MC64 missing-diagonal skip — investigated and rejected

Wrote `dev/plans/mc64-missing-diag-skip.md` documenting the proposed
port of `mumps/src/dana_aux.F:1388-1416` (skip MC64 when
`(missing_diag + zero_diag) < N/10`).

Built throwaway probe `src/bin/probe_missing_diag.rs` and ran it
across all 569 corpus families. Result:

- 501 of 569 families would have skipped MC64 under the rule.
- 289 of those 501 are arrow-KKT matrices (the very class the
  Lever-C empirical work in `dev/research/lever-c-adaptive-scaling.md`
  established as MC64-wins territory).

Root cause: feral's corpus matrices are pre-regularized IPM
snapshots. VESUVIO's dual diagonals are explicit `-1e-8`, not
structurally absent. HS118's slack diagonals are barrier
perturbations like `-2.07e2`. The MUMPS rule was designed for
*raw* KKTs from finite-element solvers where the dual diagonal is
truly missing; it does not translate to the regime feral runs in.

The user's challenge ("the corpus is not 'pure' KKT matrices?
they already have been regularized?") triggered this rejection
before any code change shipped.

Recorded as `dev/tried-and-rejected.md` entry "2026-04-28 — MUMPS
missing-diagonal MC64 skip (mistranslated regime)".

### δ_c sensitivity probe — built and ran

Built `src/bin/probe_deltac_sensitivity.rs` to answer the broader
methodological question ("could this be skewing feral development
in general?"). The probe detects each matrix's dual-regularization
block as the longest contiguous tail run of identical
small-magnitude diagonals (|x| ≤ 1e-2, rel-tol 1e-4, min run 4).
For each multiplier in {1e-4, 1e-2, 1, 1e2, 1e4} it perturbs the
detected entries and reruns `pick_scaling_strategy`,
`pick_ordering_preprocess`, plus a 5-run-median symbolic + numeric
+ refined-solve.

Result on 9 detectable-dual-block matrices: **0/9 routing flips**
across an 8-order δ_c span. Inertia stable. Wall ±5% (within
noise). Residuals scale with effective δ_c as expected.

Conclusion: Auto routing is δ_c-robust by structural signature
(arrow-KKT detection and low-degree compression both gate on
column-degree ratios, not magnitudes). The corpus's specific
δ_c choice does not skew the routing logic. A downstream consumer
with different mu_init / regularization will see identical routing
answers.

Recorded as `dev/decisions.md` entry "2026-04-28 — Auto routing
thresholds are δ_c-robust (probe evidence)".

Probe binary committed as `355773b` for future regression checks.

### `PatternFingerprint` strengthened with structural hash

Replaced `(n, col_ptr_len, row_idx_len)` with
`(n, nnz, structural_hash: u64)` in `src/numeric/solver.rs`. The
structural hash is `DefaultHasher` (SipHash) of `col_ptr || row_idx`.
Closes the silent-collision footgun where two structurally distinct
matrices with identical `(n, nnz)` would have shared a stale
symbolic factorization. The IPM hot path is unaffected — successive
KKT iterates have byte-identical column structure and the
`PartialEq` derive short-circuits on the prefix before hashing.

3 new tests (F1: identical → equal; F2: same `(n,nnz)` different
`row_idx` → distinct; F3: same `(n,nnz)` different `col_ptr` →
distinct) all pass. Solver unit tests 8/8, full suite stays 205/205.
Committed as `8b0c435`.

Read of the data:
1. **I/O dominates the harness** (36%). Inherent to the bench, not
   the solver. On a real IPM workload it goes to zero, lifting the
   effective share of every other category by ~1.55×.
2. **MC64 is no longer the top scaling cost** —
   `mc64::compute_matching` self is 0.7%, inclusive 7.7%. The
   session-01 finding ("26% of wall on per-matrix bench") was
   regime-specific (no Solver cache). The Solver-corpus bench
   amortizes via `Mc64Cache` reuse: 988 symbolic calls vs 19410
   free-fn calls = 19.6× fewer.
3. **Dense Schur SIMD kernel is the single biggest hot kernel
   path** — 8.5% self combined across `axpy_minus_unroll4_nofma`
   (5.2%), `axpy2_minus_unroll4_nofma` (1.9%),
   `schur_panel_minus_nofma_strided_dual` (1.4%). The W-2 2x2 +
   B-1 dual-column DSYRK work from sessions 13/14 is paying off.
4. **Allocator activity is large** (15.3% combined). Some is
   harness churn (per-iterate `from_triplets`); some is workspace
   allocation in `factor_one_supernode`. Pre-sized workspace reuse
   is a candidate lever but the workspace path was already
   refactored in session 2026-04-21.

Aggregate: 534 families, 19410 iterates, **1.67× aggregate**,
geomean 2.83×, p50 3.00×, p90 4.05×. Implied symbolic share of
free-fn wall: 40.2%, fully amortized.
