# Session 2026-04-19-01

## Goal

Execute the β refactor: move MC64/InfNorm scaling from `symbolic_factorize`
to `factorize_multifrontal` so `SymbolicFactorization` becomes purely
structural and cacheable across IPM iterations of structurally identical
KKTs. This is the precondition for per-iteration symbolic-cache reuse in
the pounce IPM driver integration (see
`dev/research/pounce-integration-interface.md`).

## Accomplished

- Removed `scaling`, `scaling_pivot_order`, `scaling_info` fields from
  `SymbolicFactorization`.
- Removed `scaling_strategy` field from `SupernodeParams` (collapsed to
  `nemin` only).
- Introduced `NumericParams { bk: BunchKaufmanParams, scaling:
  ScalingStrategy }` as the per-numeric input bundle to
  `factorize_multifrontal`, with a `NumericParams::with_bk(bk)`
  convenience constructor (default scaling).
- Added `SparseFactors.scaling` and `SparseFactors.scaling_info` so the
  numeric output owns scaling state.
- Updated ~30 callsites across `src/`, `tests/`, `examples/`, and
  `src/bin/` to the new API. Three patterns:
  - Pattern A (just BK params, default scaling): `NumericParams::with_bk(bk)`.
  - Pattern B (per-iteration scaling override): explicit
    `NumericParams { bk, scaling: strategy }`.
  - Pattern C (dense factor still wants BK): `factor(&dense, &params.bk)`.
- Added structural test
  `factorize_multifrontal_with_two_strategies_on_one_symbolic` in
  `src/numeric/factorize.rs` that factors the same `SymbolicFactorization`
  twice with `InfNorm` and `Identity` and verifies both produce inertia
  `(2, 1, 0)`. This is the cache-reuse invariant in test form.
- Removed three obsolete tests from `tests/mc64_scaling.rs` that probed
  the removed `sym.scaling*` fields; redirected via comment to the
  parity panel and the new structural test.
- Refactored `examples/profile_sparse_smallfront.rs` to drop Phase 1
  (MC64) timing since scaling no longer happens at symbolic time.
- Refactored `examples/debug_acopp30_mc64.rs` to read `factors.scaling*`
  post-numeric instead of `sym.scaling*`.

## Decisions Made

- The numeric-input bundle is `NumericParams { bk, scaling }`, not a
  free-standing `(bk, scaling)` tuple. Single-arg APIs are easier to
  extend (e.g. future iterative-refinement flags) and the
  `NumericParams::with_bk(bk)` shortcut keeps simple callsites brief.
- `SparseFactors` carries `scaling` (the vector) and `scaling_info`
  (the strategy-result enum) rather than re-deriving them. Keeps
  `solve_sparse` and `solve_sparse_refined` numerically self-contained.
