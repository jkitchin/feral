## Accomplished

### Expert consultation (MUMPS / SPRAL SSIDS / faer, in parallel)

All three reference-solver experts independently converged on the same
verdict: **none of them have a separate dense code path for KKTs**.

- **MUMPS 5.8.2**: every KKT goes through `dfac_front_LDLT_type1.F`.
  At the root MUMPS sets `AVOID_DELAYED=.TRUE.` + `STATICMODE=.TRUE.`
  with `SEUIL_LOC = max(SEUIL, epsilon)`. The Duff-Reid 2×2 test at
  `dfac_front_aux.F:1599-1606` has an explicit hardwired
  `ABSDETPIV == RZERO` rejection. On rejection, routes to the forced-
  1×1 branch which applies static `±SEUIL` regularization.
- **SPRAL SSIDS**: `ldlt_tpp.cxx`'s own header comment is
  *"Simple LDL^T with threshold partial pivoting. Intended for
  finishing off small matrices, not for performance"*. The 2×2 test
  applies three independent guards including a scale-invariant
  cancellation-aware det floor
  `|detpiv| < max(small, |detpiv0|/2, |detpiv1|/2)`
  (`ldlt_tpp.cxx:106`). With `detscale = 1/maxpiv`, this is scale-
  invariant by construction.
- **faer**: dense `bunch_kaufman::factor` has no growth bound, no det
  floor, no regularization — faer's sparse path survives KKTs purely
  because of AMD preordering, not a better kernel.

Full writeup with file:line citations at
`dev/research/task-19-dense-acopp30-expert-consultation.md`.

### Decision: deprecate dense factor() as a KKT validation oracle

Close task #19 as *"not a real bug — kernel under-constrained by
design"*. feral's dense `factor()` combines four things the references
never combine simultaneously: natural column order (no AMD/METIS),
pivot_threshold=0, no |det|==0 rejection, no SEUIL floor. The two
prior in-kernel fix attempts (Duff-Reid u backstop + reducible-column
floor; session 2026-04-13-05) each traded ACOPP30 for +6998
regressions because an absolute sqrt(eps) threshold assumes an
equilibrated matrix and the corpus is not.

### Step 1 implementation

Added `factor_single_front` wrapper in `src/dense/factor.rs:388-451`
that applies Knight-Ruiz equilibration then delegates to
`factor_frontal(may_delay=false)` with ncol=n. Repacks the returned
`FrontalFactors` 1:1 into a `Factors` struct plus `d_eq` so existing
`solve` / `solve_refined` work unchanged. Inherits `factor_frontal`'s
`try_reject_1x1_frontal` safe-rejection fallback — 2×2 rejection no
longer divides by zero on the 1×1 fallback path.

Exported via `src/lib.rs`. Rerouted bench's KKT dense validation
loop at `src/bin/bench.rs:633` to call `factor_single_front(...,
&params_kkt_sparse)` (pivot_threshold=0.01). Micro-benchmarks
(spd_N, kkt_N_M) still call plain `factor()` to time the dense
kernel itself.

Committed as `c619fda` "Task #19: reroute dense KKT bench through
factor_single_front".

## Decisions Made

- **Close task #19 as resolved by architectural move**, not by in-
  kernel patch. Recorded the three-expert consultation and the
  consensus in `dev/research/task-19-dense-acopp30-expert-consultation.md`.
- **Deprecate `factor()` as a KKT validation oracle**. Keep it for
  unit tests on well-conditioned BK examples from the Bunch-Parlett /
  Ashcraft-Grimes-Lewis literature. Synthetic timing benchmarks
  (spd_N, kkt_N_M) stay on `factor()` to measure the leaf BK kernel.
- Use `params_kkt_sparse` (pivot_threshold=0.01) for the new dense
  bench path, matching what the sparse validation loop uses.
