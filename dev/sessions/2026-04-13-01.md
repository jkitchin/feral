# Session 2026-04-13-01 — Phase 2.2.3 supernode adjacency fix

## Goal

Debug the CHWIRUT1 / CRESC100 / CRESC132 plateau (Phase 2.2.3 per
the prior session's "Next Session Should" list). Three matrices
that remained `#[ignore]`'d in `tests/mc64_regression.rs` with
residuals 8 to 16 orders above both target and canonical MUMPS
oracle, attributed in Phase 2.2.2 to "solve-side rounding" or
"iterative refinement stagnation". Strategy agreed with the user:
build a minimal repro first, then use expert agents for
independent references on the canonical supernodal solve.

## Accomplished

**Important context:** this session's bench stats are not directly
comparable to last session's. The old numbers were an artifact of
a `nemin=10000` override in `bench.rs` that accidentally masked
the bug being fixed today. See §"Benchmark Results" below.

### Phase 2.2.3 Steps 1–2: research + diagnostic (`cccd640`)

1. **Research note** `dev/research/phase-2.2.3-plateau.md` — five
   hypotheses: H1 refinement best-iterate lock-in, H2 residual at
   arithmetic noise floor, H3 scaling frame error, H4
   multi-supernode solve bug, H5 CRESC132 ±2 inertia. Covers
   `solve_sparse_refined`, Arioli ω₁ stop criterion, and the
   SSIDS/MUMPS iterative-refinement conventions.

2. **Diagnostic binary** `examples/triage_plateau.rs` — instruments
   each matrix under both `nemin=32` and `nemin=10000`, logs
   per-iteration `||r||₂`, `||dx||₂/||x||₂`, Arioli ω₁,
   `||A||₁·||x||∞/||b||∞`, refinement break reason. Immediate
   finding: **all four matrices converge under `nemin=10000` and
   diverge under `nemin=32`**, including ACOPP30 and CRESC132.

   Under `nemin=10000` (single-supernode-forcing):
   ```
   ACOPP30_0000   residual 1.02e-14 (was 1.08e-1)  INERTIA MATCH
   CHWIRUT1_0000  residual 8.69e-14 (was 8.50e+2)  INERTIA MATCH
   CRESC100_0000  residual 2.15e-16 (was 1.43e+2)  INERTIA MATCH
   CRESC132_0000  residual 3.65e-15 (was 1.37e+5)  INERTIA MATCH
   ```

   Three matrices **beat the canonical MUMPS oracle**. CRESC132's
   ±2 inertia mismatch also disappears. The plateau was a single
   root cause, not three; the "solve-side stagnation" and
   "trace-rule inertia" diagnoses from Phase 2.2.2 were wrong.

### Phase 2.2.3 Step 3: minimal repro + expert consultation

3. **Minimal repro** `examples/min_multisupernode.rs` — a 6×6
   arrow matrix (diagonal 2..7 with column 5 as the arrow) is
   the smallest case where `nemin=1` produces a wrong solve
   (residual 7.47e-1) while `nemin=10000` is exact. The dumped
   symbolic structure showed the bug signature directly:
   supernode `#0` with `first_col=1, ncol=1` and supernode `#4`
   with `first_col=0, ncol=2` — **two supernodes both claiming
   ownership of column 1**.

4. **spral-expert and mumps-expert consultation (parallel).**
   Both independently confirmed that a multifrontal forward
   sweep must additively accumulate `-L21·y_piv` updates from
   all child trailing rows into the shared global RHS (SSIDS
   `NumericSubtree.hxx:331`, MUMPS `dsol_fwd_aux.F:842-845`). A
   second SSIDS query confirmed that SPRAL handles non-adjacent
   sibling merging via a column-renumbering permutation `sperm`
   (`core_analyse.f90:644-685`) rather than a local adjacency
   check.

   Based on this context I read `src/numeric/factorize.rs` and
   `src/numeric/solve.rs` end-to-end and ruled out a
   solve-side bug: the forward scatter is symmetric with the
   gather (all `nrow` rows written back), and the L-solve loop
   correctly updates trailing-row entries. The bug had to be
   upstream of the solve, in the symbolic analysis.

### Phase 2.2.3 Step 4: the fix (`91e808b`)

5. **Adjacency check in `find_supernodes`** — step-2 amalgamation
   now requires `snode_first_col[root_s] + snode_ncols[root_s]
   == snode_first_col[root_p]` before the `trivial_chain` or
   `size_based` merge conditions can fire. Merge arithmetic
   switched from `min`-based to unconditional (adjacency
   guarantees `s_first < p_first`).

   **Why the bug went unnoticed.** Pre-fix, the `bench.rs`
   configuration `SupernodeParams { nemin: 10000, .. }` made the
   `size_based` test `child_ncol < 10000 && parent_ncol < 10000`
   true for *every* candidate merge. The loop then repeatedly
   merged every fundamental supernode into the running root. For
   any matrix with a single root in the elimination tree, this
   over-merging accumulated `snode_first_col = 0` and
   `snode_ncols = n`, producing a single "supernode" whose
   claimed range `[0, n)` happens to be contiguous by virtue of
   being the full column set. Downstream code processed it as a
   single dense frontal — effectively a plain dense LDLᵀ
   wrapped in sparse plumbing. Correct, but not multifrontal.
   The 99.8% sparse residual pass rate reported since Phase 1b
   was always measured in this degenerate configuration. See
   `dev/validation/phase-2.2.3-supernode-adjacency.md` §6 for
   the full mechanism.

## Decisions Made

- **Adjacency check is the minimal correctness fix.** SSIDS uses
  a more sophisticated approach (permissive merge + second
  permutation renumbering columns so every amalgamated supernode
  is contiguous by construction). That would produce coarser
  supernodes on arrow-like elimination trees and would likely
  close ACOPP30's regression, but it is a substantially larger
  refactor. We take the minimal fix now, land honest bench
  numbers, and log the SSIDS-style renumbering as a follow-up.
  Recorded in `dev/decisions.md`.

- **Drop the `nemin=10000` override from `bench.rs`.** The
  override was a workaround for the bug that this session fixed;
  it no longer serves any purpose and its presence would hide
  the honest rates. The historical 99.8% sparse pass rate is
  obsolete and should never be cited again. Recorded in
  `dev/decisions.md`.

- **Keep `mc64_regression.rs` tests `#[ignore]`'d.** CHWIRUT1,
  CRESC100, CRESC132 now pass their targets under `--ignored`
  but they require data matrix files not checked into git, so
  the `#[ignore]` remains the correct gating mechanism. The
  status comments are updated to reflect the PASS state.

## Abandoned Approaches

- **Hypotheses H1 (refinement best-iterate lock), H2 (noise
  floor), H3 (scaling frame), H5 (trace-rule 2×2 inertia bug)
  from the research note.** All four turned out to be wrong
  diagnoses. The diagnostic binary ruled them out in a single
  run by showing the plateau matrices all converged under
  `nemin=10000`, which isolates the bug to the multi-supernode
  code path. H4 was the correct hypothesis and was localized
  further to supernode amalgamation (not the solve itself)
  through the minimal-repro reduction. Recorded for future
  reference; not a regret, the hypothesis enumeration was
  essential to converge on the right diagnosis quickly.

- **Pivot-threshold experimentation on ACOPP30.** Ran ACOPP30
  post-fix with `pivot_threshold = 0.0` and `= 0.01` to check
  whether the column-relative rejection was the regression
  cause. Both values produce wrong inertia and residuals
  1e+5-class. The post-fix ACOPP30 regression is caused by
  fine-grained supernode structure, not pivot threshold. No
  code change to preserve.

## Next Session Should

1. **Port the SSIDS column-renumbering amalgamation.** Allow
   non-adjacent sibling merging by building a permutation
   `sperm` that renumbers columns so every merged supernode is
   contiguous by construction. Reference: SPRAL
   `src/core_analyse.f90:644-685`. Expected to close the ACOPP30
   regression this session introduced, and likely reduce the
   35,543 sparse-only failures materially by producing coarser
   supernodes with more BK pivoting headroom.

2. **Triage the SWOPF and HYDCAR20 matrix families.** These
   are the worst post-fix sparse failures (residuals 1e+10 to
   1e+12). SWOPF has a consistent inertia signature across
   many instances (83,92,0) → (63,68,44), suggesting a single
   bug; HYDCAR20 has (99,99,0) → (23,23,152) which is much
   more catastrophic. Start by running `triage_plateau` on one
   of each.

3. **Delayed pivoting (Phase 2.3).** ACOPP30's pre-fix 1.08e−1
   residual still wasn't at target; its fundamental limit is
   pivoting, not amalgamation. Once sperm renumbering lands,
   delayed pivoting is the next correctness-front for Phase 2.

4. **Retune the `nemin` default.** Historical default was 32
   (inherited from SSIDS) but SSIDS uses 8 by default and
   MUMPS uses 5. Now that multi-supernode runs for real, the
   right value should be benchmarked on the sanity-panel plus
   a random sample of the ~77% passing KKT matrices.

5. **`nemin=32` unit tests.** Add explicit unit tests that
   construct multi-child-parent elimination trees (arrow, star,
   balanced binary) and verify the supernode structure is
   correct. The pre-fix `symbolic::supernode::tests` all passed
   because they only covered chain-like / dense / diagonal
   cases — none of them exercised the multi-child-parent code
   path where the bug lived.
