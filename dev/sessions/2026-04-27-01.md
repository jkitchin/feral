# Session 2026-04-27-01

## Goal
Triage the cumulative memory growth in the sparse multifrontal loop
flagged in session 04 (RSS climbed 17 GB → 36 GB across 167k iters
with `FERAL_SPARSE_MAX=20000`). Determine whether real per-iteration
leak or allocator high-water-mark retention.

## Accomplished
- Wrote `src/bin/diag_sparse_memory` — drives the multifrontal
  factorization on a chosen sequence of size buckets (small/medium/
  big) and reports RSS in MB after each iteration.
- Established that **per-call steady-state leak is small**:
  - HS118 (n=32) × 100: 2.2 MB → 3.0 MB (~8 KB/iter, mostly
    allocator init on first ~10 calls).
  - EIGENALS (n=2550) × 50: plateaus at 5.6 MB by iter 13.
- Identified the actual cause: **HWM retention from outlier
  matrices**. CHAINWOO_0000 (n=4000) allocates a *1.1 GB factor*
  on first call, then RSS stays at 1.1 GB forever — even when the
  next 4 iterations are tiny n=32 matrices. macOS default malloc
  doesn't return freed regions to the OS.
- The 1.1 GB factor on CHAINWOO is *not* a bug in memory accounting;
  the factor is genuinely that large because feral's pivoting on
  this family produces 1000× the fill of MUMPS (same family flagged
  with 840× factor-time ratio in session 04 Phase 2.8.1).

## Benchmark Results
No new bench. Memory diag captures the operative pattern:
```
 iter    bucket      matrix           n      RSS_MB
    3     small  HS118_0000          32         2.6
    6    medium  EIGENALS_0000     2550         5.6
    9       big  CHAINWOO_0000     4000      1173.6
   12    medium  EIGENALS_0000     2550      1173.6   ← stuck
   15     small  HS118_0000          32      1173.7   ← stuck
   18       big  CHAINWOO_0000     4000      1202.9
```

## Decisions Made
- **Cumulative memory growth and Phase 2.8.1 slowness are the same
  root cause.** Both come from pathological fill on
  CHAINWOO/HYDROELL/DIXMAANH families. Fix the pivoting on these
  structures and both symptoms resolve.
- Defer the actual pivoting fix to a future session. Document the
  root cause and the 36 GB working-set budget as the operating
  constraint for the dev laptop. Bench users opt into
  `FERAL_SPARSE_MAX=N` per session 04 for tighter bounds.

## Abandoned Approaches
- Searching for a Vec accumulator in the bench loop or a hidden
  Arc-shared cache: none found. The growth is allocator HWM, not a
  leak.
- Switching the bench binary to mimalloc/jemallocator: would mask
  the symptom (allocator returning freed memory to OS) but doesn't
  address the underlying pathological fill. Held in reserve as a
  workaround if the memory ceiling becomes operationally blocking
  before the real fix lands.
