## Goal

Issue #19 follow-up: replace the implicit rayon scope context that
the parallel multifrontal driver currently builds per `factor()`
call with a `Solver`-owned `rayon::ThreadPool` that persists across
calls. Reduces the cv-wait wakeup cost that session 2026-05-15-03's
work-aware gate could only sidestep, not eliminate.

## Accomplished

### 1. `Solver::parallel_pool` lazy-init field

`src/numeric/solver.rs`:

- New field `parallel_pool: Option<Arc<rayon::ThreadPool>>`.
- New helper `ensure_parallel_pool() -> Option<Arc<ThreadPool>>`
  that lazy-builds on first call and clones the Arc on subsequent
  ones.
- Thread count matches `rayon::current_num_threads()` at build
  time (honors `RAYON_NUM_THREADS`, falls through to num_cpus as
  the global pool would).
- `Solver::factor` now reads the pool *before* taking the
  immutable `symbolic` borrow, so the mutable borrow on
  `parallel_pool` doesn't collide. The dispatcher runs inside
  `pool.install(...)` when both `use_parallel` is on and the pool
  built successfully.

Inside `pool.install(...)`, the inner driver's `rayon::scope` /
`current_thread_index` / `current_num_threads` calls bind to this
pool's workers instead of the global pool. No changes to the inner
function — the entire pool-reuse benefit comes from the single
`install` wrapping.

### 2. Tests

Two new unit tests in `numeric::solver::tests`:

- `solver_reuses_thread_pool_across_factors` — factors an
  indefinite tridiagonal twice, verifies `Arc::ptr_eq` on the
  stored pool across the two calls. Pins the reuse invariant.
- `solver_with_parallel_false_does_not_build_pool` — verifies
  that a sequentially-configured solver doesn't spawn worker
  threads it won't use.

Both pass. Lib suite: `cargo test --lib --release` → **256
passed** (was 254 + 2 new), 0 failed, 5 ignored. `cargo clippy
--all-targets --release -- -D warnings` clean. `cargo fmt --check`
clean.

### 3. End-to-end verification

Same problems / iter counts as session 2026-05-15-03 for direct
A/B:

| problem | config | wall (s) | sys (s) | Δsys vs session-03 |
|---|---|---|---|---|
| robot_1600 | default (gate→sequential) | 35.4 | 0.6 | — (pool built but unused) |
| robot_1600 | `MIN_PAR_FLOPS=0` (force par) | 25.9 | 17.9 | **-28%** (24.7 → 17.9) |
| henon120 | default (gate→parallel) | 102.7 | 19.3 | within noise |

The small-frequent-call regime (robot_1600 force-parallel: ~400
factor calls × ~63 ms / call) is exactly the profile issue #19
surfaced — and exactly where rayon spawn / cv-wait wakeup costs
add up. **28% sys-time reduction** on this regime.

Wall on M4 Pro is unchanged because cv-wait wasn't yet
wall-dominant here. On the issue reporter's hardware (where it
reportedly was), this should translate to a wall improvement too.

The large-workload regime (henon120) doesn't benefit measurably
because rayon overhead is a tiny fraction of per-call factor cost.

## Decisions Made

- **No user-facing toggle for pool reuse.** Pool reuse is strictly
  better than per-call construction (lower sys, same wall in the
  worst case). The existing `Solver::with_parallel(false)` toggle
  already disables the entire parallel path *including* pool
  construction (the new
  `solver_with_parallel_false_does_not_build_pool` test pins this).
  A "use global pool" escape hatch could be added later if a
  consumer needs to share threads with their own rayon work, but
  none of feral's current consumers do. Recorded in
  `dev/decisions.md`.
