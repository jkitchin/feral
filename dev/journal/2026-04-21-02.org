* 10:15 :step6:symbolic-wiring:
Integrated into =symbolic_factorize_with_method=. The
preprocess branch:
#+begin_src rust
OrderingPreprocess::LdltCompress => {
    let (matching, _n_matched) = crate::scaling::mc64_matching(matrix)?;
    let map = build_supermap(&matching);
    if map.ncmp() == n {
        run_external_ordering(&full_pattern, method)?
    } else {
        let cpat = compress_pattern(&full_pattern, &map);
        let super_perm = run_external_ordering(&cpat, method)?;
        expand_permutation(&super_perm, &map)
    }
}
#+end_src

The =ncmp == n= fast-path short-circuits the ~53% of the
corpus where MC64 matches on-diagonal — no graph rebuild, no
ordering retry. Otherwise order the compressed graph and
expand the super-permutation back to length n.

* 11:00 :analysis:
Interesting: =factor_nnz_estimate= goes *up* on the
high-leverage tail (+41-58% on VESUVIO/GAUSS2/HAHN1) even
though factor time goes *down* 50-60%. That's not
fill-reduction — that's cache and locality. Paired columns
adjacent in the pivot order means:
1. The 2×2 BK pivot test succeeds more often (both partners
   available without delay).
2. Schur updates hit sequential memory.
3. Fewer delayed pivots cascading to parents.

Matches Duff-Pralet 2005 §5 observations. The "fill" metric
is misleading — runtime is the real oracle.
