This is the smoking gun. MC64 equilibrates the matrix perfectly
(median D → 1.0, p95 drops from 80 to 1.55, max drops from 261 to

positions compounding through forward L-solve + D-solve + backward
L^T-solve, total amplification is in the 1e30 range — exactly the
observed 30-order regression.

* 16:45 :mc64:loose-zero-tol-still-bad:
Probed: what if I loosen zero_tol to 1e-8 on the MC64 path, so
the 3.6e-10 pivots get flushed to zero?

  MC64 + zero_tol=1e-8: rel_res = 2.27e46  (unchanged)
  inertia:              (62, 140, 7)  — 2 more zero pivots

Still 2.27e46. Why? Because the 5 smallest D entries are
3.63e-10, 6.21e-10, 6.48e-10, 1.07e-9, 1.23e-9 — all below 1e-8
but they're inside 2×2 pivot blocks whose |det| ≈ 0.084 is
*above* zero_tol_2x2 = 1e-16. So the 2×2 inversion goes through
and the (small) entries get inverted via the 2×2 formula. This
is an important wrinkle for the fix: ANY Phase 2.2.2 fix that
raises the 1×1 threshold must also tighten the 2×2 threshold
or equivalent, else the same small magnitudes leak through the
2×2 path.

* 16:50 :mc64:root-cause:
Root cause in one sentence: MC64 equilibrates A → D·A·D so pivots
cluster around 1, but in doing so it pushes the WORST pivots from
1e-8 (feral's existing Phase 1 "force-accept-below-eps" floor) to
3.6e-10 (still above EPS but catastrophically small for
inversion). Feral's zero_tol=EPSILON is way below MC64's "expected
reject threshold" (~0.01 under threshold pivoting).

This is a design mismatch between MC64 (which expects threshold
pivoting with u=0.01 to downstream-reject the small pivots it
exposes) and feral's Phase 1 ForceAccept (which only rejects
exactly-zero pivots). The MC64 wrapper is correct; the solve
wrapper is correct; the symbolic plumbing is correct. The bug is
at the meeting point.

* 16:55 :mc64:proposed-fix:
Preferred fix (Phase 2.2.2): threshold partial pivoting with
delayed pivots (textbook solution, what MUMPS/SSIDS do).

Minimal fix for Phase 2.2.2 Step 1: scaling-aware zero_tol.
When ScalingInfo != NotApplied, raise zero_tol to ~1e-12 (or
make it a column-relative threshold u·max_col). This flushes the
3.6e-10 pivots to zero, matching the Identity path's behavior on
its 1e-8 pivots. Expected result: ACOPP30 drops from 1e46 to
~1e16 (Identity baseline) while the other 6 sanity-panel matrices
retain their 2–10 order improvements.

MUST also address the 2×2 block path: fixing only 1×1 zero_tol
leaves a loophole (see 16:45). Option: tighten zero_tol_2x2 to
(zero_tol)² with the new zero_tol.

DO NOT touch MC64 itself — compute_symmetric is correct as
written.
