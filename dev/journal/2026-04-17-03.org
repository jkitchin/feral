Documented closed-vs-open neighborhood semantics: u, v are
indistinguishable iff N(u)\{v} == N(v)\{u}; case 2 (adjacent
indistinguishables) is captured by closed-neighborhood hash, case
1 (non-adjacent with N(u)==N(v)) is not. Decided to ship case-2
only for S1 — case 1 is rare in target workloads (FE meshes,
KKT) and would require either two hash passes or a per-pair fix-up.

Also nailed down hash determinism: std::collections::hash_map::
DefaultHasher is SipHash-1-3 with fixed seeds — deterministic
across runs and platforms. RandomState (HashMap default) is NOT
deterministic, so we don't use it for hashing the closed-nbhd.

While debugging (cur_cut went to -199 on a path graph), I
re-derived the FM gain-update sign rules for `gain = ed - id`:
when v moves from->to, neighbour u on `from` side gets
`gain[u] += 2w` (NOT -=). feral-metis fm_refine.rs:115 has the
opposite sign:
   if labels[u] == from {
       gain[u] -= 2 * w;   // BUG: should be +=
   }
This is internally inconsistent with `gain[v] = ed - id` and
`cur_cut -= gain[v]`. Symptom: best_prefix never advances past 0,
all moves get rolled back, FM is a no-op. Tests pass because
they only check `final_cut <= initial_cut` (trivially holds when
nothing moves) and `pass_cut == cut_size(labels)` (also holds
when initial state is preserved). Halo FM uses the corrected
signs; recording here for follow-up fix in feral-metis (tracked
in tried-and-rejected.md after this session).
