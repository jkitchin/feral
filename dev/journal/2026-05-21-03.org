* 17:40 :code-inspection:track-a3:root-cause:
iter-8 panel_diag (allow_delayed=true): scalar_1x1_delay_tiny=1 —
exactly one event, matching the one spurious inertia.zero. But the
1×1 reject path (`try_reject_1x1_frontal`, factor.rs:3513) counts
every force-accepted pivot BY SIGN and never writes `zero` (issue
#42 Option A). So the spurious zero cannot be a 1×1 — it must be a
2×2.

The two 2×2 zero-writing sites both classify via `sym2_eigenvalues`
(factor.rs:3901) and count `lam == 0.0` as zero:
- `count_2x2_inertia_val` (3875) — the `scalar_pivot_step` accept
  site (3448).
- `count_2x2_inertia` (4453) else-branch (4554).
`sym2_eigenvalues` returns `lam = 0.5*(tr ± s)` with `s` computed
cancellation-free as `sqrt((d11-d22)^2 + 4*d21^2)`. But the FINAL
step `0.5*(tr - s)` (and `0.5*(tr + s)`) is itself a subtraction:
for a genuine non-singular 2×2 whose small eigenvalue is far below
`ULP(0.5*tr)`, `0.5*(tr ∓ s)` IEEE-rounds to *exactly 0.0* even
though the eigenvalue is analytically nonzero. That exact 0.0 is
then counted as `zero`.

ROOT CAUSE: catastrophic cancellation in `0.5*(tr ∓ s)` fabricates
a spurious zero eigenvalue. iter 8's borderline saddle block (a
straddle 2×2, det<0, |tr| ≈ s) loses its small eigenvalue to this
rounding -> (1,1,0) miscounted as (0,1,1) -> inertia (63999,63995,1).

THE FIX: classify the 2×2 inertia from the cancellation-free
comparison `s` vs `|tr|`, never from a subtracted eigenvalue.
  s > |tr|  <=>  det < 0  -> straddle, (1,1,0)
  s < |tr|  <=>  det > 0  -> same sign as tr, (2,0,0) or (0,2,0)
  s == |tr| <=>  det == 0 -> one genuine zero, (1,0,1)/(0,1,1)
`s = sqrt((d11-d22)^2 + 4*d21^2)` and `|tr| = |d11+d22|` are both
reliably computed; the comparison `s <=> |tr|` is the cancellation-
free evaluation of `sign(det)`. This is mathematically exact for the
inertia of a symmetric 2×2 and produces a `zero` only on genuine
exact singularity (`s == |tr|` to the last bit). External oracle for
the tests-first unit tests: a *diagonal* 2×2 (d21=0) has inertia =
the signs of its diagonal entries by inspection — pick d11 tiny,
d22 huge so the discriminant drops d11 and `0.5(tr-s)` rounds to 0.

* 18:05 :code-inspection:track-a3:correction:
CORRECTION to the 17:40 entry. On working the diagonal-2×2 oracle
through, `s vs |tr|` does NOT fix it. Take `[[1e-30,0],[0,1e30]]`
(true eigenvalues 1e-30 and 1e30, both positive -> (2,0,0)):
  tr  = 1e-30 + 1e30  -> 1e30   (d11 lost: below ULP(1e30))
  s   = sqrt((1e-30-1e30)^2) = |1e-30-1e30| -> 1e30   (d11 lost again)
So `s == |tr|` in FP -> the classifier says det==0 -> (1,0,1) WRONG.
The 17:40 entry's *own* oracle construction ("discriminant drops
d11") is exactly the case where `s` and `tr` BOTH drop d11 — the
two cancellations are the same cancellation. `s vs |tr|` cannot
recover a diagonal entry that `d11 +/- d22` has already annihilated.

THE ACTUAL FIX: compute the determinant cancellation-free with
Kahan's fused difference-of-products and classify from `sign(det)`
+ `sign(tr)`:
  w = fl(d21*d21);  e = fma(d21,d21,-w)        # exact error of d21^2
  det = fma(d11,d22,-w) + e                    # d11*d22 - d21^2, ~2u rel err
For the diagonal oracle d21=0 so w=e=0 and det = fma(d11,d22,0) =
d11*d22 = 1e-30*1e30 = 1.0 exactly -> det>0, tr>0 -> (2,0,0) CORRECT.
The product d11*d22 never adds d11 to d22, so no diagonal entry is
lost. Jeannerod-Louvet-Muller 2013: this kernel has relative error
<= 2u for ANY inputs, so sign(det) is exact unless the block is
genuinely singular to working precision. sign(tr) is reliable
because correctly-rounded FP addition never rounds a nonzero sum
across zero. Classifier: det<0 -> (1,1,0); det>0 -> (2,0,0)/(0,2,0)
by sign(tr); det==0 exactly -> (1,0,1)/(0,1,1)/(0,0,2) by sign(tr).
External oracle stands (diagonal 2×2 inertia by inspection) — it
just disproves `s vs |tr|` and confirms the Kahan-det classifier.
