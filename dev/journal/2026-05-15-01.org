MUMPS factor failed on all 4 with INFOG(1)=-9 (memory allocation).

Result: feral inertia matches MA57 *exactly* on every dumped matrix

This *overturns* the issue's hypothesis. Cascade-break is not
producing wrong inertia. The "WrongInertia" status from pounce is
the IPM's expected count (9601) disagreeing with the matrix's actual
inertia — that's an IPM-trajectory issue, not a feral counting bug.

* 11:00 :issue-17:solve-accuracy:5oom:
On iter004 (identical input matrix in cb=off and cb=default), both
configs report Success and same inertia (14398, 9602, 0). But the
solution vectors differ:

#+BEGIN_EXAMPLE
||sol_cb_default - sol_cb_off||_inf       = 2.3e-4
||sol_cb_off||_inf                         = 16
relative diff                              = 1.4e-5
#+END_EXAMPLE

That's ~5 orders of magnitude solve-accuracy loss for cb=default vs
cb=off without any change in reported inertia. Plausible cause: the
cascade-break perturbation modifies the L factor (not just D), so
the solve sees a slightly different system than what generated the
inertia count.

* 11:30 :issue-17:f23:refinement:
Looked at where pounce calls feral's solve. ~pounce-feral/src/lib.rs:107~
calls ~Solver::solve~, not ~Solver::solve_refined~. The F2.3 iterative
refinement we shipped last week (with ~RefinementDiagnostics~) is
therefore *not absorbing this 1e-5 solve error*.

Three candidate fixes:
1. Wire ~Solver::solve_refined~ into pounce-feral. Cheapest fix; F2.3
   already has the kernel.
2. Default ~cascade_break_ratio = None~ in feral. Reverts to legacy
   delayed-pivot path; we lose the cascade-arm gate (#15) benefit
   on the matrices it actually helps.
3. Keep cascade-break but ensure the L factor isn't perturbed when
   it triggers — only D gets touched.

(2) is too blunt. (1) is recommended next step; (3) is the deeper
fix that lets cascade-break keep its inertia-correctness role while
not degrading solves.
