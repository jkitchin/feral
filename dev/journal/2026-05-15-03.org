#+TITLE: Session 2026-05-15-03 journal — issue #19 work-aware parallel gate

* 13:30 :issue-19:triage:
Issue #19 reports a 12× per-iter wall regression on robot_1600 with
~use_parallel=true~ (feral's default) and a 2.8× win on henon120.
Suspect: ~should_parallelize_assembly~ is structural only (~n_snodes
≥ N_PAR_MIN~ + ≥1 multi-child); doesn't gate on per-supernode work.
On many-tiny-supernode trees (control-NLP KKTs) the per-call rayon
spawn / cv-wait overhead dominates the assembly cost.

Reproducer measurements on Apple M4 Pro (14 cores, same machine the
user is on):

| problem | parallel | sequential | ratio (par/seq) | sys/wall(par) |
|---------+----------+------------+-----------------+---------------|
| robot_1600 | 25.3 s | 34.4 s | 0.74× (par faster) | 53% |
| henon120 | 101 s | 294 s | 0.34× (par 2.9× faster) | 21% |

Issue claims 12× regression on robot_1600; I see parallel actually
*faster* in wall (1.3×). But sys time is 27 s of 25.3 s wall — rayon
overhead is real on M4 Pro, just not yet wall-dominant. The
internally-inconsistent numbers in the issue (60/99 = 0.61 s/iter vs
60/199 = 0.30 s/iter implies 2× not 12×) suggest the issue's framing
of the magnitude is off. Direction of the fix is still correct.

* 14:00 :issue-19:plan:
Plan: gate ~should_parallelize_assembly~ on a per-supernode flop
estimate (sum of ~ncol * nrow^2~) clearing ~PAR_MIN_FLOPS = 10^8~,
in addition to the existing structural checks. Make the threshold
runtime-tunable via ~NumericParams::min_parallel_flops~. Wire env
var ~POUNCE_FERAL_MIN_PAR_FLOPS~ in pounce-feral.

Calibration: rayon spawn + cv-wait wakeup is ~100 μs on M-series
macOS; sequential Schur kernel does ~10 GFLOP/s; break-even is
~1 ms = 10^7 flops. Gate one decimal order higher (10^8) so parallel
must be a meaningful win, not a wash.

Research note at ~dev/research/issue-19-parallel-heuristic.md~.

* 14:30 :impl:flop-estimate:
Added ~PAR_MIN_FLOPS~, ~estimate_assembly_flops(&[Supernode]) -> u64~,
and ~should_parallelize_assembly_with_threshold(&sym, u64)~. The
existing ~should_parallelize_assembly~ becomes a thin wrapper passing
~PAR_MIN_FLOPS~.

Flop proxy: ~ncol * nrow^2~ per supernode, saturating multiply +
saturating sum so pathological inputs can't panic. Overestimates a
touch (true cost is ~ncol^3/3 + ncol^2*nrow_below + ncol*nrow_below^2~)
but dominated by the Schur term whenever ~nrow > 2*ncol~.

* 14:50 :impl:numericparams-override:
Added ~min_parallel_flops: Option<u64>~ to ~NumericParams~. Default
~None~ uses ~PAR_MIN_FLOPS~. Set to ~Some(0)~ to disable the flop
gate (structural-only behavior); ~Some(u64::MAX)~ to force-reject.
Wired through ~factorize_multifrontal_parallel_with_workspace~.

Cost: 8 source files needed the new field added to inline ~Numeric
Params { ... }~ initializers (bins/tests/inline test modules) — bulk
sed patch. No behavioral changes elsewhere.

Sys-time on robot_1600 drops 99% under the new default (27 → 0.3 s).
M4 Pro pays a slight wall cost (25 → 33 s) because on this CPU
parallel was actually slightly faster on robot_1600 — but the issue's
machine is somewhere else on the spectrum where parallel was a major
loss. The user-tunable override lets each consumer dial in.
