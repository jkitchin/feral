* 15:35 :code-inspection:fix1:
Inspected the two BK driver functions in src/dense/factor.rs:
- Plain driver `factor_frontal_in_place_with_scratch_impl`, loop
  1405-1425, single delay site 1423 (`Delayed => break`).
- Panel driver, loop 1719-1846, three delay sites: 1751 (scalar
  tail), 1841 (scalar fallback after panel), 1844
  (`PanelStatus::Delayed`).
Both compute `nelim = k; n_delayed = ncol - nelim` after the loop.

Key findings that de-risk swap-to-boundary:
1. `PivotOutcome::Delayed` docstring (factor.rs:1115-1118): "The
   kernel has not mutated any state for the failed pivot." So at
   delay time the front columns [k..nrow) are consistently updated
   through pivot k-1 — a clean symmetric swap of column k with
   column ncol_eff-1 is valid.
2. `swap_rows_cols(a, n, p, q, perm)` (factor.rs:3923) is the
   existing symmetric-swap helper; updates `perm`, no-ops on p==q.
3. CRITICAL: the multifrontal driver already consumes a permuted
   contribution block. factorize.rs:2267 builds the contrib block's
   global row indices as `row_indices[ff.perm[node_nelim + cj]]` —
   it applies `ff.perm` over the WHOLE contrib range, not assuming
   `perm[nelim..nrow]` is identity. So permuting `perm[nelim..ncol)`
   via swap-to-boundary is consumed correctly; delayed-column
   ordering within the block does not matter.
4. swap-to-boundary only swaps columns in [0,ncol) — `perm[ncol..nrow)`
   stays identity, contrib values + indices stay consistent.
5. Termination: each iteration either advances k (Advanced) or
   decrements ncol_eff (Delayed); `ncol_eff - k` strictly decreases
   -> O(ncol) iterations, no infinite loop.

Existing kernel delayed-pivot tests (tests/delayed_pivoting.rs)
checked: `factor_frontal_partial_elim_with_delay` and
`..._delays_first_pivot...` both use genuinely-stuck PAIRS (mutually
coupled, 2×2 fails growth bound), so swap-to-boundary still delays
both — those tests are unchanged by Fix 1. Confirmed by hand-tracing
`block_diagonal_partial_frontal`.
