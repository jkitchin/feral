#+TITLE: Session 2026-05-15-04 journal — Solver-owned rayon ThreadPool (issue #19 follow-up)

* 16:30 :issue-19:plan:
Following on session 2026-05-15-03 next-session item (2): the rayon
spawn / cv-wait overhead in ~factorize_multifrontal_supernodal_
parallel~ is independent of the work-aware gate. Even when the gate
fires correctly, every parallel ~factor()~ call rebuilds an implicit
rayon scope context. A persistent ~ThreadPool~ owned by the
~Solver~ avoids that per-call cost.

* 16:45 :impl:thread-pool-field:
Added ~parallel_pool: Option<Arc<rayon::ThreadPool>>~ to ~Solver~.
Lazy-init via ~ensure_parallel_pool()~ helper that returns an
~Arc<ThreadPool>~ (or ~None~ on builder failure). Built on first
parallel ~factor()~; pool persists for the ~Solver~'s lifetime.

Thread count matches ~rayon::current_num_threads()~ at build time —
honors ~RAYON_NUM_THREADS~ and falls through to ~num_cpus::get()~ as
the global pool would. Don't try to be clever about thread-count
tuning: the existing global-pool calibration is already there for
us.

* 16:55 :impl:wire-into-factor:
~Solver::factor~ now reads, in order:
  1. ensure pool (mutable self borrow on ~parallel_pool~)
  2. borrow symbolic (immutable on ~last_symbolic~)
  3. call ~pool.install(|| factorize_multifrontal_parallel_with_
     workspace(...))~ when both ~use_parallel~ is on and the pool
     was built successfully

Inside ~install~, all ~rayon::scope~ / ~current_thread_index~ /
~current_num_threads~ calls in the inner driver bind to this pool's
workers instead of the global pool. No further changes to the inner
function — the entire pool-reuse benefit comes from the single
~install~ wrapping.

The small-call regime (robot_1600 force-parallel) is exactly the
profile issue #19 surfaced — many ~factor()~ calls per IPM iter,
each previously paying rayon spawn + cv-wait. 28% sys-time
reduction. Wall on M4 Pro is essentially unchanged because the
cv-wait wasn't yet wall-dominant; on the issue reporter's hardware
where it was, this should translate to a wall improvement too.

The large-workload regime (henon120) doesn't benefit measurably
because rayon overhead is a tiny fraction of per-call factor cost.

* 17:45 :note:no-config-knob:
Did NOT add a user-facing toggle for pool reuse. It's strictly
better than per-call pool creation (lower sys, same wall in the
worst case), and the existing ~with_parallel(false)~ toggle already
disables the entire parallel path including pool construction.
A "use global pool" escape hatch could be added later if some
consumer needs to share threads with their own rayon work, but
none of feral's current consumers do.
