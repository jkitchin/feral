* 09:30 :fm:rebalance:
FM rebalance pre-pass. Diagnosed the A6 issue precisely:
best_cut=pass_cut and best_prefix=0 at pass start unconditionally.
When input is imbalanced, best_prefix=0 is a violating
rollback target; unless the FM trajectory reaches a balanced
state with cur_cut STRICTLY < pass_cut, rollback returns to
the imbalanced start.

Fix: best_prefix: Option<usize>. None sentinel = "no balanced
state seen yet". First balanced state is unconditionally
recorded; subsequent require cur_cut < best_cut. If pass ends
with best_prefix=None, keep full move sequence (labels at
post-last-move) rather than rolling back.

Ported identically to:
 - feral-metis::fm_refine::refine_bisection
 - feral-scotch::halo_fm::halo_fm_refine
 - feral-scotch::band_fm::band_fm_refine (inner FM)

updated:

   not meaningful across imbalanced→balanced transitions.

   start, same reasoning. Dropped `after <= before`, added I4.

* 10:00 :amd:retire:investigation:
Priority 3: retire src/ordering/amd.rs in favour of feral-amd.

Discovered the two AMDs implement DIFFERENT algorithms:
 - src/ordering/amd.rs: simplified AMD with EXACT external degree,
   no element absorption, no supervariables. Written for clarity
   and to have something working during bring-up.
 - feral-amd crate: full AMD per Amestoy/Davis/Duff 1996 + 2004
   (approximate external degree, aggressive absorption,
   supervariable detection). Matches SuiteSparse/faer on the
   oracle fixture suite.

So this is not a drop-in swap — they produce different
permutations and therefore different fill. Added AmdCrate variant
(Cargo.toml + src/symbolic/mod.rs + match arms) and wrote one-off
src/bin/amd_compare.rs to A/B them across parity+large corpora.

through feral-amd and remove the AmdCrate alias; keep the file
for now as a reference implementation but take it out of the
dispatch path.

* 10:30 :amd:retire:delayed-pivoting-fixture:
Tests after retirement: 96 lib + all crate tests green. Integration
test `delayed_pivoting.rs` FAILED — two tests assumed the tiny-pivot
column (orig col 0) would be ordered at pivot 0 (a leaf). With the
original 4×4 arrow, three leaf columns had equal degree 1. In-tree
AMD's ascending-index tie-breaker put col 0 at pivot 0. feral-amd's
different tie-breaker put col 0 at pivot 2, which then merged with
the root (pivot 3) via the SSIDS trivial-chain amalgamation rule —
resulting in the tiny pivot being INSIDE the root supernode, so no
delay propagated.

Fix: broke the degree tie by adding `(2,1) = 0.5` to the fixture.
Col 0 now has degree 1 (uniquely minimum) and any AMD puts it at
pivot 0. Cols 1, 2 have degree 2; col 3 is apex (degree 3).
Resulting structure under any min-degree AMD:
  supernode [0]: {pivot 0}        (leaf, tiny)
  supernode [1]: {pivots 1, 2, 3} (root, merged by fundamental +
                                   trivial-chain rules)
Delay from pivot 0 propagates to the root supernode. Both
delayed-pivoting integration tests now pass.

Documented the fixture swap in the function's docstring with a
pointer to this journal entry.

Two observations:
 1. feral-metis already gave (71,138,0) with MUMPS pre-retirement,
    with no test failure because the parity panel runs only Amd.
    The retirement exposes this inertia classification boundary as
    a general property of ordering choice on rank-deficient KKTs,
    not an AMD-specific regression.

    Numeric correctness is unimpaired.

This is the documented zero-vs-tiny-negative pivot classification
boundary at rank-deficient positions — the underlying matrix
is structurally singular (1 zero eigenvalue in the KKT block),
and whether the rank-loss manifests as a bit-exact zero pivot or
a |pivot| ≈ 1e-20 signed value depends on the pivot sequence,
which depends on the ordering.
