* 11:10 :bratu3d:plain-solve:numerical-bug:
The user's manual smoke run on Boeing/bcsstk38, GHS_indef/bratu3d,
GHS_indef/cont-201, Schenk_IBMNA/c-big surfaced a real bug on
**bratu3d** (GHS_indef collection):

  - matrix:           bratu3d.mtx
  - n:                27,792
  - nnz:              88,627
  - factor:           Solver::new() with defaults (ScalingStrategy::Identity,
                      default Bunch-Kaufman pivot threshold)
  - factor wall:      118.7 s on M-series
  - FactorStatus:     Success (no Singular, no WrongInertia)
  - rel residual:     ||A x - b|| / ||b|| = **4.66e6** for plain
                      =Solver::solve(&rhs)= where =rhs = A * 1=
  - rel residual_r:   2.46e-9 after =solve_refined=

Plain =solve= returns garbage with NO programmatic signal. Any
caller using =solve()= directly (notably ripopt early in its
IPM-alignment refactor) silently produces wrong answers on
bratu3d-class matrices.

Hypotheses to investigate (separate PR per user instruction —
"Do not change Solver defaults in the same PR"):

  1. Is the factor itself wrong (huge entries in L, NaN/Inf), or
     is forward-substitution losing accuracy on a well-conditioned
     factor? Print =max |L|=, =max |D|=, =min |D|= post-factor.
  2. Does =NumericParams { scaling: InfNorm, .. }= fix the plain
     solve, or only the refined solve?
  3. Does raising =bk.pivot_threshold= from 0.0 above some floor
     fix it?
  4. Is the symptom dimension-dependent (try smaller GHS_indef
     indefinite matrices), or specific to bratu3d's structure
     (3D Bratu PDE)?

Decision pending after (1)-(4):
  - (a) change defaults (scaling and/or pivot threshold), or
  - (b) Solver detects the condition post-factor and returns a
        =Quality::EscalateRefinement= signal automatically, or
  - (c) document + assert "always use solve_refined for indefinite"
        and have plain =solve= panic-or-warn on indefinite factors.

Repro:

      large_matrix_full_pipeline_smoke --nocapture
#+end_src

* 12:30 :bratu3d:fix:growth-flag:
Triaged hypotheses 1-4 with =examples/triage_bratu3d.rs= (5-config
sweep, output captured at =/tmp/feral-bratu3d-triage.log=):

  | config                              | max\vert L\vert | plain rel_res | refined rel_res |
  |-------------------------------------+-----------------+---------------+-----------------|
  | Auto / threshold=0.0 / Fail         |    8.058e16     |   4.657e6     |   2.456e-9      |
  | InfNorm / threshold=0.0 / Fail      |    8.058e16     |   4.657e6     |   2.456e-9      |
  | Mc64Symmetric / threshold=0.0       |    4.341e16     |   3.687e4     |   3.882e-11     |
  | Auto / threshold=0.01 / ForceAccept |    2.744e1      |   1.254e-14   |   1.254e-14     |
  | Identity / threshold=0.0 / Fail     |    4.391e16     |   7.012e6     |   1.794e-9      |

Conclusions:
  - **H1 confirmed**: the factor IS broken — max\vert L\vert = 8e16 with
    pivot_threshold=0.0 (BK alpha-test only). No NaN/Inf entries; pure
    growth from accepted near-zero pivots whose 2x2 inverses multiply
    column entries by ~1e10–1e16.
  - **H2 falsified**: =InfNorm= scaling alone does not fix the plain
    solve — same 4.66e6 residual. Scaling is not a substitute for
    threshold pivoting.
  - **H3 confirmed (alt)**: =Mc64Symmetric= partially helps (3.7e4
    instead of 4.7e6) but still not a usable plain solve.
  - **H3 fully confirmed**: =pivot_threshold = 0.01= (MUMPS default
    CNTL(1)) reduces max\vert L\vert from 8e16 to 27 and gives
    plain rel_res = 1.25e-14. This would be the right default to
    pursue, but per user instruction defaults are NOT changed in
    this PR.

**Fix landed (option b — detection, not default change):** added
=L_GROWTH_THRESHOLD = 1e6= and =flag_growth_for_refinement= helper in
=src/dense/factor.rs=. Called at every dense-factor exit (=factor=,
=factor_frontal_with_profile=, =factor_frontal_blocked=) so any factor
with =max\vert L_ij\vert > 1e6= sets =needs_refinement = true=. The
sparse multifrontal aggregator already ORs =FrontalFactors.needs_refinement=
into =SparseFactors.needs_refinement=, so the bratu3d-class signal
propagates to =Solver::factors().needs_refinement= without further
plumbing.

Why 1e6: well-pivoted BK with unit-diagonal L has \vert L_ij\vert
\le 1/(1-\alpha) \approx 2.78. Matrices with growth in [2.78, 1e6]
converge in 1–2 IR steps without needing to be flagged; matrices above
1e6 are catastrophic and need IR for any reasonable accuracy.

Tests:
  - =src/dense/factor.rs::growth_flag_tests= (7 unit tests on the
    helper directly: below/above threshold, catastrophic, negative
    large, idempotent on pre-set, empty L, Inf entries).
  - =tests/growth_flag.rs::well_conditioned_spd_does_not_flag_growth=
    (negative-case end-to-end through =factor()=).
  - =tests/growth_flag.rs::bratu3d_default_factor_flags_refinement=
    (=#[ignore]= because matrix is external; verifies
    =max\vert L\vert > 1e6= and =factors.needs_refinement = true=
    end-to-end).
  - Full =cargo test --release= passes — no regression.

Plain =solve()= behavior on bratu3d is unchanged: callers still get
the wrong answer if they use it directly. But now they have a
programmatic signal — =factors.needs_refinement= — to detect the
condition without having to evaluate the residual themselves.
Default-change is a separate decision deferred to a future PR.
