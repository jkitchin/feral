* 15:25 :phase-2.9:analysis:why-neutral:
Hypothesis from the research note was that ~build_row_indices~ and
per-front scratch allocation were the bottleneck on tiny fronts. The
implemented specialisation:

1. Replaces ~build_row_indices~ with symbolic-time precomputed rows —
   saves the ~build_seen~ scan and sort.
2. Skips the already-empty children loop.

But both of these costs are trivial for true leaves. The dominant
per-front costs are still present in ~factor_one_small_leaf~:

- ~frontal_buf.clear() + resize(nrow^2, 0.0)~ per member — writes zeros
  to the pooled ~ws.frontal_values~ each time. Separate memset per
  member, not amortised across a group.
- ~factor_frontal_blocked~ itself — the dense BK kernel. This is the
  actual work; calling it N times on N tiny fronts has the same
  cost whether or not they're "grouped" at this level.
- ~row_map~ write/restore — already ~O(nrow)~ per member, unchanged.

The plan's original sketch (leaf arena, members sharing a single
backing allocation) is *not* what got implemented. Implementing that
would require ~factor_frontal_blocked~ to accept a ~&mut [f64]~
backing slice instead of owning a ~SymmetricMatrix~ — a non-trivial
kernel refactor. Deferred.

* 15:35 :phase-2.9:decision:
Keep the gate. The symbolic pass and the specialised numeric path are
correct, gated Off, and carry no runtime cost when disabled. They are
a prerequisite for Phase 2.9.2/2.9.3 (the actual arena + in-place CB
reuse). The negative result on speedup is the finding; it will be
recorded in ~dev/tried-and-rejected.md~ so future work knows the naive
"just precompute row indices" specialisation is not the bottleneck.

* 15:40 :phase-2.9:next:
The per-front overhead diagnosed in 2026-04-23 is *not* in
~build_row_indices~. Candidates for the real bottleneck (to investigate
in a follow-up phase):

- The ~frontal_buf.resize(n*n, 0.0)~ memset cost on tiny fronts.
  10× 32×32 memset ~= 10× 8KB = 80KB of zeroing per group; one group-
  wide memset of 80KB is cache-friendlier.
- The ~factor_frontal_blocked~ blocked kernel itself may be
  sub-optimal at ncol ≤ 8 — the blocked path adds bookkeeping that a
  straight-line unblocked path on ≤ 8 columns can skip.
- Dispatch overhead into ~factor_frontal_blocked~ (BK pivot search
  setup, inertia accumulator init, perm vector allocation).

Phase 2.9.2 sketch: true stack-arena for the whole supernode
postorder, MUMPS-style. Members of a leaf group share one allocation;
members share ~ws.row_map~ write/restore across the group. Non-leaf
supernodes bump-allocate from the same arena.

* 16:10 :phase-2.9.1:profile:
Per-leaf phase profiler (~src/bin/diag_leaf_profile.rs~) run on
ACOPR30_0067 / CRESC100_0000 / HAIFAM_0082 / VESUVIO_0000 (1832 leaves
total, 50 repeats each). Results (aggregated %tot of per-leaf total):

| phase            | mean ns | %tot  |
|------------------+---------+-------|
| row_map_setup    |      15 |  3.8% |
| memset           |      16 |  3.8% |
| scatter          |      18 |  4.3% |
| bk_kernel        |     177 | 42.6% |
| contrib          |      23 |  5.7% |
| row_map_teardown |      15 |  3.8% |
| total            |     417 |  100% |

~Instant::now()~ measurement overhead is 35.9% of total (expected —
each phase is ~15-20 ns, same order as the timer call itself), so
absolute numbers are inflated but *ratios* are the signal.

Key finding: the outer-memset hypothesis from 2026-04-24 15:40 was
*wrong*. The ~frontal_buf.resize(n*n, 0.0)~ memset is only 3.8% of
per-leaf time. The dominant cost (42.6%) is *inside*
~factor_frontal_blocked~ itself — its internal ~vec![0.0; nrow*nrow]~
allocation-and-copy, perm/perm_inv Vec allocation, pivot-search
setup. Any Phase 2.9.2 refactor must target the dense kernel
internals, not the outer buffer management.

* 16:15 :phase-2.9.2:plan:
Next refactor target: ~factor_frontal_blocked~ accepts an external
~&mut [f64]~ for its working array ~a~ and reuses caller-provided
perm / perm_inv / d_diag / d_subdiag Vecs via ~Vec::clear()~ +
~Vec::extend_from_slice~. The leaf-batch loop would then make a
single arena-sized allocation per group. This is a substantial
kernel surgery — factor_frontal and factor_frontal_blocked both
allocate, and their callers (sparse/dense/tests) all assume owning
returns. Deferring until an isolated benchmark confirms this is where
the savings actually live (i.e. a standalone diag that calls
factor_frontal_blocked on a batch of tiny fronts vs a hypothetical
arena version).

* 16:45 :phase-2.9.2:step-a:gate-fail:
Step A sub-profile of factor_frontal (commit TBD) rejects the arena
refactor. Added ~factor_frontal_with_profile~ with optional
~FrontalProfile~ sink; production ~factor_frontal~ stays branchless
(wrapper passing None). Called through ~diag_leaf_profile~ on 1832
leaves × 50 repeats across ACOPR30_0067 / CRESC100_0000 / HAIFAM_0082
/ VESUVIO_0000. Headline sub-phase breakdown (%bk_total / %inner):

  alloc+copy   9.7% /  17.7%   <-- target of arena refactor
  setup        7.8% /  14.2%   <-- target of arena refactor
  pivot_loop  17.6% /  32.0%   <-- actual BK arithmetic
  extract     19.9% /  36.1%   <-- return-struct allocs (L, D, contrib, perm_inv)
  meas overhead ~45%            <-- Instant::now() cost

Removable by arena refactor (alloc+copy + setup) = 17.6% of bk_total,
below the 25% gate in the plan. Best-case per-leaf speedup from
eliminating all of it: 1.22× on bk_kernel × 52% bk-share-of-total =
~1.12× overall. Target was 1.5×.

The ~extract~ phase (19.9% of bk_total) is actually larger than the
refactor target. It owns five Vec allocations — ~l~, ~d_diag~,
~d_subdiag~, ~contrib~, ~perm_inv~ — that FrontalFactors requires.
Eliminating those would mean redesigning FrontalFactors to borrow
from scratch, which is a much bigger architectural change with
ripple through every call site.

Conclusion: the per-leaf dense kernel is dominated by actual BK
arithmetic + owned-return allocation, not by the internal work
array / perm / subdiag allocs. The 10× gap vs MUMPS on ACOPR30 is
not sitting inside ~factor_frontal~. It is somewhere else — maybe
scatter indirection cost, outer multifrontal driver bookkeeping,
or (more likely) a fundamentally different algorithmic setup
(nested-dissection ordering, delayed-pivot strategy, supernodal
amalgamation budget).

Action: close Phase 2.9.2, record in tried-and-rejected.md. Keep
the diagnostic hook ~factor_frontal_with_profile~ in place — it is
zero-overhead when unused and valuable for future kernel triage.

| VESUVIO_0000   | 3083 |  2215  |  3082  |  4361  |  1.97× |  1.41× |

Three findings:

1. FERAL is at-or-ahead of SSIDS on every matrix in this slice. On
   HAHN1 and BATCH we are 7-8× *faster* than SSIDS (which links
   vendor BLAS, unlike us).
2. SSIDS/MUMPS ratio scales with matrix size: 6× on BATCH (n=121),
   1.4× on VESUVIO (n=3083). That's the signature of MUMPS having
   ruthlessly optimised fixed per-invocation driver overhead that
   SSIDS (and we) have not. As matrix grows, fixed cost amortises.
3. We already beat MUMPS on a third of the corpus (AVION2 0.42×,
   BATCH 0.84×, HAIFAM_0000 1.33×).

Conclusion: the "10× vs MUMPS" framing was a distraction from
what we'd actually achieved. The near-term target should be
"match or beat SSIDS" — which we've done. The MUMPS-gap on tiny
IPM is a real deficit worth closing *eventually*, because those
are the motivating use case (interior-point solves on small KKT),
but it is a narrow problem class where even SSIDS pays 4-8× vs
MUMPS. It is engineering effort on the driver/amalgamation, not a
BLAS ceiling.

Written up at ~dev/research/reference-solver-comparison.md~. The
Phase 2.9 research note carries a retrospective pointer to the new
note.
