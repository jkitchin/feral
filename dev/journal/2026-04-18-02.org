:surprise: A6 originally asserted I4 (balance after FM) with
max_imbalance=0.5 starting from 1-7 (side_max=7 > max_side=6).
Test failed: FM returned with labels still 1-7.

Reading refine_bisection: =best_cut= initialises to the pass input
cut, =best_prefix=0=. When starting imbalanced, the first-ever
balanced state reached during the move sequence must have
=cur_cut < best_cut= to record itself. If it doesn't, rollback
returns to =best_prefix=0= = the imbalanced start.

This is NOT a bookkeeping bug — it is a FM-design choice. Classic
FM assumes balanced initialisation. The note in §5 already flagged
this: "This is one place where the current code's bookkeeping
inconsistency would show even after the sign fix — worth recording
the expected post-fix value here once the fix lands."

Conclusion: the expected post-fix value is "FM does not rebalance
an imbalanced start". Softened A6 to I1+I2 only with an explanatory
comment. Recorded as a genuine follow-up candidate (not a
regression).
