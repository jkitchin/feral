* 13:30 :bench:oom:streaming:
While running fill-parity readouts the bench itself OOM-killed
(exit 137, ~13 GB RSS) loading the kkt corpus. Root cause:
=KktEntry= held a =SymmetricMatrix= populated via =mtx.to_dense()=
at *load time* for every matrix in the manifest, even ones the
sparse path would later use exclusively. With ~154k matrices and
some n>180k, the dense block alone is hundreds of GB nominal.

Fix in =src/bin/bench.rs=: =KktEntry.matrix= becomes
=Option<SymmetricMatrix>=. A new env-controlled gate
=FERAL_DENSE_MAX= (default 1000) decides at load time whether to
build the dense copy:

let dense = if csc.n <= dense_max {
    Some(mtx.to_dense())
} else {
    None

Dense-validation loop now skips entries with =matrix.is_none()=
or =csc.n > dense_max=, counting them under
=n_dense_skipped_large=. Sparse-validation loop is unaffected —
it already only touches =entry.csc=. Tests pass; bench builds and
runs without OOM on the full corpus.

* 15:00 :diag:methodology:apples-to-apples:
User flagged: the parity numbers above use feral's *adaptive*
ordering (=symbolic_factorize= → =pick_default_method= → AMD or
MetisND). MUMPS oracle uses default =ICNTL(7)= (AMD-flavored,
because =mumps_bench= links only PORD+AMD, no METIS). SSIDS
oracle uses default =options%ordering=1= = METIS. So the honest
parity statements are:

- =feral-AMD vs MUMPS= (both AMD-flavored).
- =feral-MetisND vs SSIDS= (both METIS).
- =feral-Auto= reported alongside as a sanity check on what
  feral actually ships.

Reworked =diag_fill_parity.rs= to factor each matrix three times
(Auto, AMD, MetisND) and emit three labeled bands of parity
metrics. =OrderingMethod::Auto= is dispatched via
=symbolic_factorize_with_method=. Smoke test on ACOPP30
(n≤198, all three orderings collapse to the same ordering on
small graphs as expected): identical numbers across bands.

* 15:30 :diag:results:kkt:
Full =data/matrices/kkt= run with the three-ordering methodology,
115 s wall, 430 MB RSS, 154,588 / 169,585 factored (14,997 matrices
filtered pre-factor, 0 factor failures, 0 panics):

| ordering   | feral/MUMPS geomean | feral/MUMPS p50 | feral/SSIDS geomean | feral/SSIDS p50 |
|------------+---------------------+-----------------+---------------------+-----------------|
| Auto       |                1.25 |            1.00 |                1.80 |            1.75 |
| AMD        |                1.24 |            1.00 |                1.79 |            1.75 |
| MetisND    |                1.24 |            1.00 |                1.78 |            1.75 |

Three readings:

1. *MUMPS parity is real and median-tight*: feral / MUMPS p50 =
   1.00 across every ordering. Geomean 1.24× is driven by a
   long-tail (p99=8, max=9 at =COOLHANSLS_0000=). On the typical
   matrix feral matches MUMPS exactly.
2. *AMD ≡ MetisND on this corpus*: numbers across the three
   ordering bands are nearly indistinguishable. The kkt corpus
   is dominated by structurally regular IPM iterates where AMD
   and METIS find equivalent eliminations.
3. *The SSIDS gap is not an ordering gap*: feral / SSIDS holds
   ~1.75× p50 regardless of which ordering feral uses. The
   ~75 % extra entries come from somewhere downstream of
   ordering — most likely supernode amalgamation /
   relaxation. SSIDS is more aggressive in merging supernodes
   (zero-fill amalgamation), and =factor_nnz= = real entries
   stored, including structural zeros within supernodes. Worth
   a Phase 2.x investigation but not a correctness issue.

Worst-case outlier =COOLHANSLS_0000= sits at 9× both oracles —
it's a small, dense block that fits all three solvers about
equally poorly relative to =nnz(A)=, so the ratios bunch up.

* 16:00 :amd:bug:fix:
Re-running =diag_fill_parity= on =data/matrices/kkt-expansion= /
=kkt-mittelmann= panicked with =index out of bounds: the len is
1697 but the index is 1697= inside =factorize_multifrontal=, with
RSS ballooning to ~30 GB before the abort. Adding
=catch_unwind= in the diag let the run continue but the bug was
clearly real.

Isolated to AMD via =FERAL_DIAG_MAX_N=5000= +
=FERAL_DIAG_VERBOSE=1= — first failing matrix
=ARWHDNE_0000= (n=500), reliably reproducible. Wrote a one-shot
=src/bin/repro_arwhdne.rs= that loads the .mtx and calls
=symbolic_factorize_with_method(OrderingMethod::Amd)= +
=factorize_multifrontal=. Same panic.

Root cause in =crates/feral-amd/src/algo.rs::create_element=:
the in-place branch (=elenme==0=) walks the pivot's variable
list, marking absorbed neighbours and writing surviving ones to
=iw[pme1..]=. =pme2= is tracked as =i32= and is decremented for
every absorbed entry. When *every* neighbour is absorbed,
=pme2 = pme1 - 1= (i.e. =-1= in =i32=). The function returned
=pme2 as usize=, which wrapped =-1i32= → =usize::MAX=. Downstream
=finalize_step= uses =for pme in pme1..=pme2_incl= — turning
that wrap-around into an "infinite" loop that panics on the very
first =iw[pme]= access.

Fix: switch to exclusive-end semantics. =create_element= now
returns =pme2_excl: usize= = =pme1= when the pivot's var list is
empty, else =(pme2 + 1) as usize=. =finalize_step= renamed
parameter and converted all four loops from =pme1..=pme2_incl= to
=pme1..pme2_excl=. Updated docstrings + the in-test assertion
=pme2 - pme1 + 1 == 1= → =pme2_excl - pme1 == 1=.

Validation:

  pass, 0 fail.
- =cargo test --release --lib= → 162 pass, 0 fail.
- Repro on =ARWHDNE_0000= now: =symbolic ok, n_supernodes=469;
  numeric ok, inertia=(500,0,0)=, matching the sidecar.
