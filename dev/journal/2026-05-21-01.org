* 18:40 :perf:b2:bug:scaling:correctness:
B2 integration test `mc64_cache_hit_bit_matches_cache_off` caught a
real pre-existing bug in `ScalingStrategy::External`. On a cache hit
(`External(D₀)` injection) the solve returned exactly 10× the
cache-off oracle on `tridiag(6,10.0,1.0)`:
  cache-on : [0.8333..., 1.6667..., ...]
  cache-off: [0.0833..., 0.1667..., ...]  (correct)
10 = the matrix diagonal. Instrumented the capture/hit/result
points: `factors.scaling` is bit-identical `[0.3162...×6]` in BOTH
the External and the fresh-Mc64Symmetric path — the ONLY difference
was `scaling_info`: `NotApplied` (External) vs `Applied` (MC64).

Root cause: the solve keys `needs_scaling` off `scaling_info`
(`solve.rs:113` — `!matches!(scaling_info, NotApplied)`), and sizes
the `scaled_rhs` workspace the same way (`solve.rs:74,330`). The
contract is `NotApplied ⇔ scaling vector is all-ones no-op`. The
factor, however, applies `D·A·D` UNCONDITIONALLY for every strategy
(`factorize.rs:1298` dense, `scaling_pivot_order` supernodal). The
`External` arm of `compute_scaling_with_cache` paired a *real*
vector with `NotApplied` → factor builds `D·A·D`, solve skips the
un-scaling → returns `D⁻¹A⁻¹D⁻¹b`. With `D=0.3162·I` that is
`10·A⁻¹b`. Latent since `External` was added; never hit because no
prior code drove a non-identity `External` vector through
factor+solve (the only tests, `external_strategy_passes_through`,
checked `compute_scaling`'s return value, not a solve).

Fix: `External` arm now returns `ScalingInfo::Applied` — a
non-trivial scaling WAS applied and the solve must undo it.
`NotApplied` is now exclusively `Identity` (genuine all-ones).
Updated the `Applied`/`NotApplied` docstrings and the
`external_strategy_passes_through` test (oracle: factor scales ⇒
solve must un-scale; independent oracle = the cache-off path).
`cargo test --lib numeric::solver::tests`: 22 passed, 0 failed —
`mc64_cache_hit_bit_matches_cache_off` now bit-matches across all
3 calls. Implication: B2's `External` injection point is sound; the
plan's assumption that `External` was solve-correct was wrong, and
the integration test (cache-off as independent oracle) is what
caught it — exactly its design purpose.

Cascade-break IS the fix for speed — a 100× whole-replay speedup,
the iter 6-9 fronts collapse from 64-208 s to ~45 ms. CB genuinely
breaks this cascade (contradicts the §20:55 worry that bounded-Δ
PerturbToEps might not).

BUT: iter 9 returns WrongInertia — got 63999/63995/1, want
64000/63995/0. One spurious zero eigenvalue. iters 6-8 are
inertia-correct under CB; only iter 9 (the most IPM-converged,
smallest δ) fails. Mechanism: cascade-break force-accepts a pivot
BK wanted to delay; that pivot is genuinely ≈0 at its supernode;
PerturbToEps{abs_floor:1e-10} bumps it to ±1e-10, but if the
inertia zero-tolerance exceeds 1e-10 the perturbed pivot still
classifies as zero. The eps is an ABSOLUTE floor (factorize.rs
:2206-2209 explicitly says "callers should pre-multiply by
||A||_∞") and neither CB=1 nor the auto-CB path (solver.rs:638)
scales it — raw 1e-10. So bounded-Δ CB trades the cascade for an
inertia error on near-singular iterates.

Track A2 is therefore NOT "arm CB by default" — it is "make
bounded-Δ cascade-break inertia-exact". The perturbation must be
(a) larger than the inertia zero-tolerance so the pivot classifies
as nonzero, and (b) signed to match the true eigenvalue. This is
the issue #46 / kkt-zero-2x2-block-cascade territory — check
whether the #46 fix (session 2026-05-20-03) already produced an
inertia-exact cascade path before designing a new one.
