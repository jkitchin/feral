* 07:00 :perf:investigation:agent-team:
Dispatched four parallel research agents:
- spral-expert: SSIDS MC64 default policy
- mumps-expert: MUMPS MC64 default policy and ICNTL(8) fall-through
- Explore: feral MC64 wiring + symbolic stages + prologue map
- Explore: prior-art search across decisions/tried-and-rejected/
  research/journal for the three proposed interventions

Findings:
- SSIDS defaults to NO scaling (`spral/src/ssids/datatypes.f90:233`).
- MUMPS SYM=2 defaults to MC64 BUT has a key skip rule: when
  `(missing_diag + zero_diag) < N/10` it skips MC64 and falls
  through to cheap symmetric Ruiz equilibration (SIMSCA), at
  `mumps/src/dana_aux.F:1388-1416`. This is the canonical
  KKT-aware skip.
- feral has Policy 4 fallback (correctness, not perf gating)
  and an `Auto` route that picks MC64 when `diag_only/n >= 0.3`
  (arrow detection); it does NOT have MUMPS's inverse "skip when
  diagonal is mostly populated" test.
- feral already has a `Solver` struct in src/numeric/solver.rs
  that caches symbolic across re-factorizations of the same
  pattern — `bench_solver_reuse.rs` exercises it on 4 hardcoded
  families. The pattern fingerprint is weak (n + col_ptr.len +
  row_idx.len) but works for the IPM use case.
- The per-matrix `bench` walks 154k matrices via the FREE-FUNCTION
  API and recomputes symbolic on every call. This explains the
  64% sym share in profile_hot — it's an artifact of the bench
  harness, not real production cost.

* 09:00 :perf:probe:mc64:abandoned:
Tackled the planned MUMPS missing-diagonal MC64 skip. Built a
throwaway probe (`probe_missing_diag`) to size the threshold and
found the literal rule mistranslated to feral's regime: 501 of
569 corpus families would-skip, of which 289 are arrow-KKT that
empirically WIN on MC64 (lever-C work). Reason: feral's KKT
corpus stores explicit dual regularization (-1e-8 on VESUVIO,
slack barrier perturbations on HS118), so `missing+zero diag`
is ~0 on every matrix. MUMPS designed the rule for raw
unregularized SYM=2 inputs.

Discussed with user — they asked whether the corpus is even
representative of how feral will be used or if we are skewing
development. Pivoted to a δ_c sensitivity probe.

* 09:30 :perf:probe:deltac:robustness:
Built `src/bin/probe_deltac_sensitivity.rs`. Detects the dual
reg block as the longest contiguous tail run of identical
small-magnitude diagonals (|x| ≤ 1e-2, rel-tol 1e-4, min run 4).
For each multiplier in {1e-4, 1e-2, 1, 1e2, 1e4}, perturbs the
detected entries and reruns `pick_scaling_strategy`,
`pick_ordering_preprocess`, and a 5-run-median symbolic + numeric
+ refined-solve. 11 matrices probed, 9 with detectable dual block
(HS118 and KIRBY2 had heterogeneous slack diagonals — no clean
δ_c).

Result: 0/9 matrices flipped routing across an 8-order δ_c span.
Inertia stable. Wall time ±5% (within noise). Residuals scale
with effective δ_c as expected for refined-solve on a more
singular matrix.

Conclusion: Auto routing is δ_c-robust by structural signature
(arrow-KKT detection and low-degree compression both gate on
column-degree ratios, not magnitudes). The corpus's specific
δ_c choice does not skew the routing logic; a downstream
consumer with different mu_init / regularization should see
identical routing answers.

Recorded as a `decisions.md` entry. The MUMPS rule is recorded
as a `tried-and-rejected.md` entry citing the regime mismatch.
The throwaway `probe_missing_diag` binary was deleted; the
useful `probe_deltac_sensitivity` is committed as the basis for
the decision.

* 10:15 :refactor:solver:fingerprint:
Strengthened `PatternFingerprint` in `src/numeric/solver.rs`.
Old structure was `(n, col_ptr_len, row_idx_len)` — three integers
that collide silently on any two matrices with the same (n, nnz).
New structure is `(n, nnz, structural_hash: u64)` where
`structural_hash` is a `DefaultHasher` (SipHash) of `col_ptr ||
row_idx`. The IPM hot path is unaffected because successive
iterates have byte-identical column structure and the equality
test short-circuits on the prefix; this just closes the footgun
for general callers who might hand `Solver` two different
matrices with identical (n, nnz).

Added 3 tests: F1 byte-identical patterns equal; F2 same (n,nnz)
different `row_idx` distinct; F3 same (n,nnz) different `col_ptr`
distinct. All 8 solver unit tests pass; full suite stays 205/205.
Committed as 8b0c435.

Read of the data:
1. **I/O dominates the harness** — 36% wall on file read+parse is
   inherent to the benchmark, not the solver. On a real IPM
   workload where the matrix is already in memory, this 36% goes
   to zero, and the remaining solver wall would be dominated by
   the dense kernel and the multifrontal frontal factor.
2. **MC64 is no longer the top scaling cost** — `mc64::compute_matching`
   self-time is 0.7%, inclusive 7.7%. The hot scaling work is now
   `hungarian_match` (2.9% self) which is called *from inside* MC64
   (one of two augmenting-path passes) plus `compute_infnorm` (2.9%)
   for the InfNorm path. The session-01 finding ("MC64 is 26% of
   wall on the per-matrix bench") doesn't reproduce here precisely
   because `bench_solver_corpus` exercises the Solver-cache
   amortization path: 988 symbolic calls vs 19410 free-fn calls,
   19.6× fewer.
3. **Dense Schur kernel is the single biggest hot kernel** — 8.5%
   self combined across `axpy_minus_unroll4_nofma` (5.2%),
   `axpy2_minus_unroll4_nofma` (1.9%), and
   `schur_panel_minus_nofma_strided_dual` (1.4%). The W-2 2x2
   inline + B-1 dual-column DSYRK work that landed in the prior
   two checkpoints is paying off in `factor_frontal_blocked_in_place`
   (4.7% self / 14.7% inclusive).
4. **Allocator activity is large** (15.3% combined for malloc,
   free, memset, memmove). Some of this is benchmark harness
   churn (per-iterate `CscMatrix::from_triplets` builds new vectors);
   some is workspace allocation inside `factor_one_supernode`.
   Future lever: pre-size the workspace once per Solver lifetime
   and reuse across iterates. Not actionable today without a deeper
   look — the workspace path was already refactored in session
   2026-04-21.

Aggregate harness output (unchanged from prior bench): 534
families, 19410 iterates, **1.67x aggregate Solver vs free-fn**,
**geomean 2.83x**, p50 3.00x, p90 4.05x. Implied symbolic share
of free-fn wall: 40.2%, fully amortized by Solver.

Conclusion for the session-01 question "what dominates after
symbolic is amortized?": the dense per-supernode factor + Schur
kernel together account for ~22% of wall (combined dense self),
with the remaining numeric cost split between scaling (7.3%) and
multifrontal assembly (2.6% self / 19.1% inclusive). The biggest
remaining lever is continuing to widen the dense kernel
(Phase 2.4.x track).
