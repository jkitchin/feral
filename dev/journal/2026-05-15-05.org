* 22:00 :issue19:calibration:probe:
Wrote ~src/bin/calibrate_par_min_flops.rs~. Sweeps K, runs
symbolic once per K, then times best-of-N sequential vs
forced-parallel factor calls. The "forced" framing uses
~factorize_multifrontal_supernodal_parallel~ and
~factorize_multifrontal_supernodal_with_workspace~ directly,
bypassing ~should_parallelize_assembly~. This separates
*driver performance* (what we want to calibrate) from *gate
decision* (what the gate currently does about it).

All runs inside a single persistent ~rayon::ThreadPool~ wrapping
the parallel calls in ~pool.install(...)~, matching the
Solver-owned pool from session 04. Without this the calibration
would measure pool-construction overhead repeatedly, which is
not the production case.

* 22:30 :issue19:calibration:decision:
Decision: *don't change the const default in this session*.
Two reasons:

1. Calibration is on one workload family (Poisson-KKT) and one
   CPU (M4 Pro). Issue #19's reporter saw the 12× wall regression
   on non-M4 hardware on robot_1600, a different problem family.
   Without their probe data I can't verify 3e7 would be safe on
   their workload mix.

2. The existing ~NumericParams::min_parallel_flops~ knob already
   lets consumers override the default. Lowering the default is
   only a default-policy change, not a capability change — it
   should be made with cross-hardware data, not unilaterally.

Wrote up the methodology, data, and recommendation in
~dev/research/par-min-flops-calibration-2026-05-15.md~. Will
post to issue #19 asking the reporter to run the probe on their
hardware.
