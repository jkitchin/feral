* 17:25 :issue38:root-cause:mc64-cache-staleness:
Hypothesized stale MC64 cache. Mechanism:
- ScalingStrategy default = Auto (mod.rs:126)
- Auto on arrow-KKT routes to Mc64Symmetric (compute_scaling_auto_with_cache)
- pick_ordering_preprocess triggers LdltCompress when n>=128 and
  >=30% columns have degree<=2 (rocket KKT qualifies — 38400 slack columns
  in n=89601)
- LdltCompress at symbolic time populates SymbolicFactorization::cached_mc64
  via mc64::compute_matching(matrix) at iter 0 values
- compute_scaling_with_cache(matrix, Mc64Symmetric, Some(cache)) takes the
  Some path → scaling_from_cache(cache) → applies iter-0 cache to iter-N matrix
- MC64 matching, dual variables (u, v), and column maxes (cmax) are all
  value-dependent — but the cache reuses iter-0 values across all subsequent
  factor() calls

The bug: in IPM, every factor() call uses different values on the same
pattern. The cache is built once at iter 0 and reused, silently applying
iter-0's scaling to iter-N's matrix. As values drift, the scaling becomes
increasingly mismatched, producing wrong inertia and eventually a cascade
of delayed pivots that explode the factor cost.

* 17:35 :issue38:scope:fix-plan:
The MC64 cache must be invalidated (or recomputed) per factor() call when
the matrix values change. Options:
1. Drop the cache entirely. Lose the ~70% symbolic-overhead amortization
   that motivated the cache; but the amortization assumed reuse for the
   SAME matrix, which never happens in IPM.
2. Recompute the cache lazily on each factor() when scaling resolves to
   Mc64. Same cost as no cache for IPM, no savings on reuse.
3. Cache-with-revalidation: at factor() entry, cheap check whether values
   changed since cache was built (hash of a sentinel value? first column
   sum?), invalidate if so. Cheap check, recomputes only when needed.
4. Treat the cache as the "initial guess" for an MC64 warm-start (does
   Hungarian support warm-starting?). Most aggressive; needs research.

Option 2 is the safe immediate fix. Option 3 is the optimization. Option
4 is research scope.

Independent of which path, this needs a regression test that locks the
warm-vs-fresh inertia agreement on the rocket_12800 dumps (or a smaller
arrow-KKT reproducer that ships with the repo).
