* 21:10 :phase-2.4.1b:step-4b:design:
  Session 4a delegated `factor_frontal_blocked` to `factor_frontal`.
  Step 4b is the real peek-ahead panel. Design:

  - New enum `PanelStatus { Full, ScalarFallback }` — 2-valued to keep
    the reply surface small (plan §Risks #1 option b: panel handles
    only 1×1; any 2×2 candidate hands back to scalar).
  - `lblt_panel_frontal(a, nrow, k, ncol, bs, d_panel, params) ->
    (n_elim, PanelStatus)` — iterates c=0..bs, peek-aheads column k+c
    by replaying pivots 0..c-1 via `axpy_minus_unroll4_nofma`, then
    picks/accepts/rejects the pivot.
  - `peek_ahead_column` — replay for one trailing column.
  - `apply_blocked_schur` — deferred Schur after the panel for
    columns j = j_start..nrow (j_start set by caller, see bug below).

  Bit-parity argument: scalar accumulates each (i,j) via pivot-outer /
  column-inner; replay accumulates each (i,j) via column-outer /
  pivot-inner. Both traverse pivots in ascending q for any fixed
  (i,j), and both use `axpy_minus_unroll4_nofma`, so IEEE 754 rounding
  matches. Frozen-column invariant: pivot q never touches column q
  again after its own scaling, so no re-read-after-write problem.

* 21:45 :phase-2.4.1b:step-4b:bug:2x2-boundary:
  Diagnosed the 2×2 boundary bug. n=128, zeros on diag at {63,64}.
  Panel (bs=64) eliminates pivots 0..62, at c=63 akk=0, returns
  ScalarFallback with n_elim=63. But during c=63 the code had
  peek-ahead'd column k+63 (applied pivots 0..62). Then the caller
  invoked `apply_blocked_schur` which re-applied pivots 0..62 to all
  columns including k+63 — double rank-1 update on that one column.
  The scalar step that followed then picked up an already-corrupted
  column, producing a 1-ULP drift in the 2×2 pivot arithmetic.

  Fix: added `j_start: usize` parameter to `apply_blocked_schur`.
  Caller sets:
    - Full          -> j_start = k + n_elim   (update trailing block)
    - ScalarFallback -> j_start = k + n_elim + 1 (skip peek-ahead'd col)
