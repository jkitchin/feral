* 15:00 :policy4:bench:bug:
Implemented the 3-condition rule. cargo test --release: all
22 scaling tests pass including 4 new fixture tests. Full
release suite: 118 lib tests + integration tests + parity
tests all green.

Ran corpus bench. Sparse residual_pass: 154 225 / 154 588 --
SEVEN WORSE than the predicted 154 233 (+1) and 7 worse than
the 154 232 baseline. MSS1_0009 residual showed 2.35e-3 in
the worst-residuals table, far worse than the 9.96e-7 it had
under MC64.

Per-matrix dump showed MSS1_0007 through MSS1_0013 (7
matrices) all at residual ~2.4e-3. But the unit test
asserts that compute_scaling(&Auto) returns InfNorm scaling
for MSS1_0009, and an explicit FERAL_SCALING=infnorm bench
gives MSS1_0009 = 6.31e-12.

Same scaling vector, different residual. The difference must
be in ScalingInfo. Checked numeric/solve.rs:112: solve uses
`!matches!(factors.scaling_info, ScalingInfo::NotApplied)` to
decide whether to apply pre/post-scaling. My Policy 4
fallback returned ScalingInfo::NotApplied -- so solve was
SKIPPING the scaling step entirely.

The plan file even said "ScalingInfo::NotApplied for the
fallback path matches the InfNorm convention (InfNorm doesn't
report Applied)". This was wrong. InfNorm returns
ScalingInfo::Applied (verified in src/scaling/infnorm.rs:93).

Fix: forward the actual in_info from infnorm::compute_infnorm
(which is Applied). Re-bench: sparse residual_pass = 154 233
EXACTLY the prediction. MSS1_0007-0013 back to 6e-12.

Lesson recorded as a load-bearing invariant in
dev/decisions.md: any non-trivial scaling vector MUST return
ScalingInfo::Applied. NotApplied is reserved for paths where
the vector itself shouldn't be applied (Identity / External).
