* 15:25 :investigate:
Looked at how pulp dispatches in schur_kernel: `pulp::Arch::new().dispatch(K)`
which calls `with_simd<S: pulp::Simd>` with the platform-best S. On aarch64,
this is `pulp::aarch64::Neon` with `F64_LANES = 2`. The quad kernel
(`schur_panel_minus_*_strided_quad`) issues 8 SIMD acc regs per chunk over 4
destination columns, unrolled 2× = 16 logical regs (`Phase 2.4.3` note in
factor.rs:2398). The "wide / near-square" supernode hits this path for every
trailing-column quad ≥ 4 wide; the path is *engaged* but the question is
arithmetic intensity, not coverage.

For (1928, 1928) full-square: panel processes ncol=1928 in 30 panel passes of
bs=64 each (default block_size). For each panel, deferred Schur is a rank-64
update over a shrinking trailing block — 1864×1864 then 1800×1800 etc. The
FLOPs for this front: panel LDLᵀ ≈ n³/3 = 1928³/3 ≈ 2.39 GFLOP. If the issue
quotes 152 ms for snode 3607 → 15.7 GFLOPS sustained on real assembly. Issue
reports 1.3 GFLOPS on snode 3593 — that's the wider-trail / smaller-pivot one.
Need to measure both shapes.

Surprises:
1. The 2829×433 supernode hits 33 GFLOPS in synthetic — NOT 1.3 GFLOPS as
   the issue's back-of-envelope estimates. The issue's 196 ms for snode
   3593 in the live driver matches our 178 ms synthetic — i.e. the kernel
   itself is fine, the issue's "1.3 GFLOPS" derived from a 250 MFLOP
   estimate is off by ~25× because the real FLOP count is ~5.92 GFLOP
   (we measure 6 GFLOP, the issue text estimates 250 MFLOP via
   "(433² LDLᵀ) + (2396×433 Schur)" which is too low — the rank-433
   schur over 2396×2396 trailing rows is what drives FLOPs).
2. SPD 1928×1928 is *worse* than KKT 1928×1928 (593 vs 227 ms). Hypothesis:
   the SPD matrix's diagonal dominance keeps every pivot trivial (no rook,
   no 2x2), so every panel goes through the "full" deferred-Schur path
   which does 31 × bs=64 panel passes of growing-trail rank-64 Schur on
   ~1864² block. KKT triggers some 2x2 paths that take a shorter route?
   Need to check `panels` counter — both report panel_full=31 / partial=0,
   so the panel count is identical. The KKT speedup might actually be a
   cache-locality artifact of the indefinite path's specific access pattern.
   Worth flagging but not the main result.
3. FMA path is *slower* on aarch64 NEON for these shapes (~0.65× on shape B
   SPD, mixed on shape A). Suggests the FMA dispatch's `pulp::mul_add_f64s`
   either isn't issuing FMADD or is taking a worse code-gen path. Counter
   to issue #8 expectations.
4. SIMD coverage is >99.84% by FLOP across all configs — the scalar tail is
   negligible. The dispatch already uses NEON 2-lane vectors throughout.

* 16:00 :ablation:block-size:
Tried bs=128 → panics at `apply_blocked_schur_panel` MAX_N_ELIM=64
hardcoded buffer overflow. bs=128 is currently a latent crash if anyone
sets BunchKaufmanParams::block_size > 64 in production. Not exploited
because default is 64 and the panel clamps to `bs.min(ncol)`.

Tried bs=32:
- A.SPD: 270 ms (down from 593 ms at bs=64). 17.65 GFLOPS median.
- A.KKT: 343 ms (up from 227 ms at bs=64). 13.89 GFLOPS.
- B.SPD: 249 ms (up from 177 ms at bs=64).
- B.KKT: 247 ms (up from 177 ms at bs=64).

bs=32 cuts shape A SPD time in half but degrades the rest. Suggests
**bs is shape-dependent**: full-square fronts want smaller panels (?),
wide trailing-row fronts want bs=64. This is consistent with a
data-reuse argument — smaller panels reduce the panel-internal cache
miss rate at the cost of more outer iterations. The current default
(64) is good for the wide case and bad for the full-square root.

This already points to a cheap win: tune bs per (nrow, ncol) shape.
