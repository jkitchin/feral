* 17:10 :amf:phase-c:dtoc1nd:rootcause:
DTOC1ND n-mismatch was a *script-side* bug, not a feral bug.
run_mumps_amf.py used `out_path = workdir / f"{mtx.stem}.out.txt"`
to dispatch oracle output. The corpus has two .mtx files whose
stems differ only in case:
  - kkt-expansion/DTOC1ND/DTOC1ND_0000.mtx  (n=9990, 29960 entries)
  - kkt-mittelmann/dtoc1nd/dtoc1nd_0000.mtx (n=9685, 77854 entries)
On case-insensitive APFS both write to the same temp out.txt;
the second-running matrix clobbers the first, then both canonical
sidecars get written from the same parsed dict. All 20 DTOC1ND_*
sidecars carried dtoc1nd_* content.

Verified by running the oracle directly on DTOC1ND_0000.mtx alone:
fresh out.txt reports n=9990 nnz=29960 nnz_l_estimated=118785
(matching the .mtx header), confirming MUMPS is fine and the bug
is in the Python dispatcher.

Fix: switch to index-based out filenames (matrix_{idx:07d}.out.txt)
so collisions are impossible on any FS. Cleared 40 stale sidecars
across both DTOC1ND and dtoc1nd directories, re-ran with the fixed
script under --skip-existing: 40/40 ok. Verified post-regen:
  DTOC1ND/DTOC1ND_0000.hamf4.json  n=9990 nnz=29960  (correct)
  dtoc1nd/dtoc1nd_0000.hamf4.json  n=9685 nnz=77854  (correct)

Layout chosen: y[c*n + i] for column c row i; w[c*nrow + i] per

* 21:55 :condition-estimate:F2.0:
Authored dev/research/condition-estimate.md — F2.0 design note.
Six decisions captured before F2.1 implementation:

  D1. Algorithm: Hager 1984 + Higham 1988 refinement. Inline
      forward-form (not DLACON reverse-communication) since we
      have direct access to the factor.
  D2. Symmetric specialization: A⁻ᵀ = A⁻¹ for our LDLᵀ factor,
      so inner loop uses ONE solve per iteration not two. Total
      cost 3-5 solves vs 6-10 for the general case.
  D3. API: Solver::estimate_condition_1norm + lower-level
      Solver::estimate_inverse_norm_1, plus free-function form
      estimate_condition_1norm_free for callers holding
      SparseFactors directly.
  D4. ConditionWorkspace analogous to SolveWorkspace, allocated
      once and reused across the 3-5 internal solves.
  D5. Error contract: Err only on solve failure or dim mismatch;
      "did not converge in 5 iter" is NOT an error (LAPACK
      convention). n=0 returns Ok(0.0).
  D6. Validation: Hilbert κ within 10× of Higham 2002 §28 table;
      diagonal-spectrum within 2×; cross-validate against MUMPS
      RINFOG(11) on N≥1000 corpus matrices, geomean ratio in
      [0.5, 5.0].

Errata caught: kkt-feature-gaps.md said MUMPS exposes the
estimate via "INFOG(40)" — actually the correct field is
"RINFOG(11)" (real-valued, populated when ICNTL(11)=2). Fixed
the plan in the same commit. Field correction matters because
F2.2's MUMPS oracle extension has to write the right field.

Code-touch map identifies F2.1 work: new src/numeric/condition.rs
with matrix_norm_1, estimate_inverse_norm_1, ConditionWorkspace,
estimate_condition_1norm_free; Solver methods on solver.rs;

* 23:00 :schur:F3.0:mumps:
Authored dev/research/schur-complement.md — F3.0 design note.
Spawned mumps-expert agent for an implementation-level review of
MUMPS's Schur path (cited in the note with file:line refs into
../ripopt/ref/mumps/src/).

Six anchor MUMPS facts captured before deciding feral's API:

  - Three output modes via ICNTL(19): centralized dense full-square,
    distributed lower-tri, distributed full-square. Only mode 1
    (centralized full-square symmetric) matters for a clean-room
    shared-memory port.
  - Validation: SIZE_SCHUR=0 silently disables, <0 or >=N is
    INFO(1)=-49, missing LISTVAR_SCHUR is INFO(1)=-22. Schur is
    incompatible with parallel analysis, max-transversal, analysis
    -time scaling.
  - Ordering constraint enforced INSIDE AMD/QAMD, not as a
    post-processing step. HALO-SCHUR sets PERM(i)=N+1 for Schur
    vars and amalgamates them into a single supervariable rooted
    at LISTVAR_SCHUR(1).
  - Numeric stopping rule: per-front, NVSCHUR = #{i in front :
    PERM(i) >= N - SIZE_SCHUR + 1}. Elimination loop bounds NPIV
    <= NASS - NVSCHUR. NO "is this the root?" branch in the inner
    factor loop — purely a bound on the eliminable region.
  - Output: SIZE_SCHUR x SIZE_SCHUR column-major Fortran, full
    symmetric (both triangles) for the centralized symmetric mode.
  - Schur block is NOT pivoted, scaled, or compressed. Delayed
    pivots from descendants must be eliminated BEFORE the Schur,
    not consumed into it (PARPIVT1_SET_NVSCHUR_MAX recomputes the
    bound at the root).

Seven decisions (D1-D7) recorded in the note. Key design choice
diverging from MUMPS: D3 picks the "AMD on non-Schur subgraph,
then append Schur tail" mechanism over MUMPS's HALO-SCHUR
amalgamation. Rationale: feral does not own feral-amd's API
surface; bolting a tail-pinned variable list onto its driver
would require invasive changes. The alternative is composable
across all four ordering methods (AMD, AMF, MetisND, ScotchND)
and naturally produces the same single-root supernode after the
elimination tree + supernode-detection passes amalgamate the
Schur tail.

Other decisions:
  D1 single dense column-major full-square output (matches MUMPS
     ICNTL(19)=1 only)
  D2 schur_indices is the user's original-index list; output
     row/col order matches the input list
  D4 numeric hook computes nvschur per-front and bounds nelim;
     for the postorder layout from D3, only the root supernode
     has nvschur > 0, so non-root fronts run unchanged
  D5 full-square output buffer (n_schur^2) not packed lower
     (n_schur*(n_schur+1)/2): downstream consumers want any LAPACK
     routine to work without packed adapters
  D6 Bunch-Kaufman pivot loop at the root excludes Schur columns
     from candidate selection — matches MUMPS exactly
  D7 cross-validation oracle path: extend MUMPS oracle harness
     with a schur driver writing the dense Schur to a sidecar;
     feral compares entry-wise on N>=100 corpus matrices, max
     relative error <= 1e-10

Phase plan: F3.1 ordering hook (1-2 sessions), F3.2 numeric hook
(2-3 sessions), F3.3 cross-validation (1 session), F3.4 optional
solve_against_schur convenience. F3.4 deferred until a caller
asks (sIPOPT-style sensitivity is the most likely trigger).

Open questions resolved in the note: factor_with_schur uses the
global default ordering method (composable per D3); empty
schur_indices is treated as plain factor(); schur_indices.len()
== n returns InvalidInput; iterative refinement is the user's
problem (not feral's); inertia returned by factor_with_schur is
the inertia of A_FF only — documented prominently.

References cited: MUMPS file:line, Duff & Reid 1983 (multifrontal),
Pirnay-López 2012 (sIPOPT use case), Gould-Scott-Hu 2007 (Schur
API survey).
