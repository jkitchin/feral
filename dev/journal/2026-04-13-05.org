The new worst, MISTAKE_0101 n=22: expected =(9, 13, 0)=, got
=(8, 13, 1)= — i.e. a positive pivot became a zero pivot. Root cause:
the absolute =sqrt(eps)= floor in fix (b) assumes an equilibrated
matrix (||A||∞ ~ 1). The bench corpus is not equilibrated. For
matrices where ||A||∞ >> 1, legitimate columns with off-diagonal
couplings ~1e-10..1e-8 get flagged as reducible and their diagonals
force-zeroed.

Why: when the growth bound rejects the 2×2 at k=58, fix (a) falls

Fix (a) cannot stand alone — when the 2×2 is rejected in this
scenario there is no safe 1×1 fallback at position k. The correct
handling is either delayed pivoting (swap column k with some
downstream column that has a usable diagonal) or treating the
column as rank-deficient and moving on — but neither is the current
fallback.

* takeaways :lesson:
1. *Absolute numeric floors are dangerous without equilibration.*
   sqrt(eps) ~ 1.49e-8 is only meaningful when ||A||∞ is close to 1.
   The bench corpus spans many orders of magnitude, so any absolute
   cutoff is a regression waiting to happen.
2. *A triage example that clears on 3 matrices is necessary but not
   sufficient.* The 3 ACOPP30 cases proved the hypothesis about the
   root cause (2×2 with |det|~eps² + unequilibrated column) and
   proved the fix *reached* that code path. They did not protect
   against the same code path being activated spuriously on 7000
   other matrices.
3. *Fix (a) exposed that our 2×2 rejection fallback is broken.*
   The dense path can't safely decline a 2×2 when a[k,k]=0. This
   is a latent bug that future delayed-pivoting work will need to
   handle regardless of the ACOPP30 issue.
4. *The harness and triage infrastructure committed in c55bacf and
   555b579 are still valuable* — they remain the starting point for
   the next attempt. Commit them early so a failed factor fix doesn't
   lose them.
