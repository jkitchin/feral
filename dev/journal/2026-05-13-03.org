=== Diagnosis ===

1. The 87s is spent in ONE frontal supernode. 99.6% inclusive in
   factor_frontal_blocked_in_place_with_scratch on one rayon worker

3. **FMA does not help because axpy is memory-bandwidth bound.**
   The kernel performs ~2 flops per 16 bytes of memory traffic
   (load src, load+store dst). FMA halves the rounding per op but
   does not reduce memory traffic, so FMA on (87.500s) vs FMA off
   (87.536s) is within noise. This is the answer to why the Phase
   3 perf gate is not met — FMA was the wrong lever.

   many 2x2 pivot decisions when iterate 0008 (identical n, nnz,
   and symbolic pattern) factors in 44ms? Candidates:

   a. Near-zero 1x1 pivots that Bunch-Kaufman escalates to 2x2
      (the BK criterion: |a_ii| < alpha * max_off_col triggers
      2x2 search). Iterate 0009 may have many small diagonal
      entries due to barrier parameter scaling at this IPM step.

      saddle block (constraint Hessian) which becomes
      progressively more ill-conditioned as the barrier mu shrinks
      across IPM iterates. Iterates 0001..0008 may have been at
      mu > some threshold where the constraint block was numerically
      definite enough to pick 1x1 pivots; iterate 0009 crosses
      that threshold.

   c. We may be missing a panel restart / supernode amalgamation
      heuristic that MA57 has (which finishes in 2.6s on the full
      IPM).

5. The fix is NOT FMA. The fix has to come from one of:
   - Reduce 2x2 pivot count (better pivot threshold, scaling, or
     a panel-level static-pivoting heuristic for nearly-symmetric-
     positive blocks)
   - Reduce trailing-update cost (cache blocking, panel size
     tuning — but axpy is already as simple as a memory loop gets)
   - Use a different elimination order on this supernode (LDL^T
     within a dense block can still be left-looking to halve the
     memory traffic)

=== Delay trigger ===

The delay comes from `dense/factor.rs` panel/scalar logic. Two
sites:
  - try_reject_1x1_frontal (line 2549): |d| <= max(u*col_max, eps)
    with u = NumericParams::default().bk.pivot_threshold = 1e-8.
    Triggers when the diagonal is < 1e-8× the off-diagonal column
    max.
  - 2x2 growth_fail / det_floor_fail (line 2446) with may_delay=
    true. Triggers when the 2x2 candidate fails the Duff-Reid
    growth bound `(|d22|·rmax + |d21|·tmax)·u > |det|` (u=1e-8) or
    the SSIDS scale-invariant det floor.

On pinene IPM iterates: the KKT bottom-right block is `-δ_c · I`
where δ_c is the constraint regularisation. As μ shrinks, δ_c
shrinks too (Ipopt's default is δ_c = μ^β with β ~ 0.25). At
iterate 0009, δ_c is small enough that constraint rows fail
either the 1x1 column-relative test (|−δ_c| < 1e-8 · |J entry|)
or the 2x2 growth bound. Iterate 0008 was still safely above the
threshold for most of those rows.

=== Why MA57 doesn't hit this (best public-knowledge guess) ===

The user hinted "I guess it is factorization". Without reading
MA57 source (per session constraint), what is known publicly from
Duff (2004), Duff & Pralet (2005), and the HSL documentation:

1. **MA57 uses the same threshold partial pivoting framework.**
   `cntl[1]` defaults to 0.01 in the HSL distribution; Ipopt sets
   `ma57_pivtol = 1e-8`. Same value as FERAL's default. So the
   raw threshold isn't the difference.

2. **MA57 has a documented "static pivoting" fallback** for the
   case where delays accumulate beyond a budget. When too many
   pivots are delayed at any node, MA57 can perturb the diagonal
   (|d| := |d| + alpha) and accept the pivot in place, avoiding
   the delay. This is exposed in HSL as `cntl[4]` (static pivot
   threshold) and in Ipopt as `ma57_block_size` / `ma57_pre_alloc`.
   FERAL has NO static-pivot fallback in the panel — it always
   delays. **This is almost certainly the difference.**

3. **MA57's amalgamation may produce a different elimination tree.**
   FERAL uses METIS-ND + supernode-amalgamation; MA57 uses AMD by
   default in Ipopt. AMD often produces a fatter tree with a
   smaller root separator on saddle-point problems. With a smaller
   root, even worst-case delay accumulation doesn't balloon the
   root supernode to 14k columns.

4. **MA57 may inline scaling differently.** Ipopt's MA57 driver
   runs MC64 scaling by default unless explicitly disabled.
   FERAL's summary says `scaling=Applied` so we are scaling, but
   it's worth checking whether the scaling is producing values
   that escape the threshold on iterate 0009.

The combination of (2) + (3) is the most credible answer: MA57's
static-pivoting fallback prevents the cascade, AND its AMD-based
ordering produces a smaller root separator that bounds worst-case
work even if some pivots do delay.

   trade. The IPM-side outer regularization (delta_c

Findings:
- The user hypothesis ("try larger than 0.99") is falsified by
  this sweep: pushing higher monotonically *increases* both wall
  clock and inertia error. Closer to 1.0 means the trigger fires
  on fewer supernodes, so more delays pile up at the root and the
  Bunch-Kaufman growth bound is breached on the perturbed front
  rather than absorbed cleanly by a single mid-tree node.
- The correctness window is narrow (0.94-0.95) and not smooth.
  Whether inertia matches the oracle is fundamentally about
  *which* supernode the trigger fires on, not a continuous
  tradeoff. Between 0.93 and 0.94 a single different supernode
  flips the trigger; that supernode happens to be where one
  negative pivot was hiding in the delayed-from-below set.
- Fastest correct point on 0009 remains 0.95 at 1.28s (69x over
  default 87.8s). 0.94 is 1.07x slower (1.37s) with one
  fewer delayed_in (63656 vs 65270) and marginally better
  residual (1.0e-14 vs 1.8e-14) — not a meaningful improvement.

Implication for default: 0.95 is a defensible default *for this
matrix*, but the 0.94-0.95 window is too narrow to trust on
unseen iterates. Treat cascade_break_ratio as a knob the IPM
integrator tunes per iterate (e.g., crank up when residual is
already large, leave off in clean regimes), not a global default.
