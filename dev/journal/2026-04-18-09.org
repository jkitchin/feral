** Step 3 — lever-B prototype on 1024×1024 trailing block

New throwaway binary src/bin/blas3_prototype.rs comparing
three implementations of "apply 64 panel columns to a
1024-wide lower-triangular trailing block":
  - rank-1 cascade (current production path):     5.30 ms (12.7 GFLOPS)
  - naive deferred (triple loop):                 34.96 ms ( 1.9 GFLOPS, 6.5× SLOWER)
  - 4-way ILP-tiled deferred:                     12.49 ms ( 5.4 GFLOPS, 2.4× SLOWER)

Sanity: all three produce results identical to within 2.0e-12.
Math is the same; only the inner kernel SIMD-quality differs.

The decision rule (parent note §4: ≥3× → commit to 2.4.1b)
fires "no" by a large margin. Same lesson as the 2026-04-14
2.4.1a tried-and-rejected entry: BLAS-3 reformulation
without a vectorized inner kernel is pure overhead. The
"BLAS-3 advantage" only materializes if the rank-bs kernel
is itself register-tiled SIMD (faer's Ukr<MR, NR, T> pattern).

* 19:45 :research:lever-c:scaling:mc64:vesuvio:cresc:
Started writing the lever-C research note. While extending

characterization in dense-kernel-vesuvio-tail.md was wrong.

** The mistake

I had been describing the VESUVIO root frontal as "structurally
unavoidable due to a single dense linking column." Today's
diagnostic shows:

  shape:       n=3083 max_col_nnz=1026 diag_only=1025
  col_deg:     deg=1: 1025  deg=2-4: 2050  deg>32: 8
  Amd/infnorm: sym_max_nrow=10 actual_max_nrow=2059
               total_delays=949 root=2059x959 fac=234.7ms

Symbolic predicts max_nrow=10. Actual is 200× larger. The
"dense root" exists because 949 pivots got delayed up the
etree and accumulated at the root.

** What works

Same matrix, same ordering, same kernel — switch scaling
from InfNorm (default) to MC64Symmetric:

| matrix       | InfNorm fac | MC64 fac | speedup | InfNorm delays | MC64 delays |
|--------------|-------------|----------|---------|----------------|-------------|
| VESUVIOU_0000| 234.7 ms    | 10.4 ms  | 23x     | 949            | 0           |
| VESUVIOU_0005| 150.1 ms    |  9.4 ms  | 16x     | 580            | 0           |
| VESUVIO_0000 |  54.0 ms    |  9.6 ms  | 5.6x    | 142            | 0           |
| VESUVIO_0021 | 179.1 ms    | 10.5 ms  | 17x     | 697            | 0           |
| VESUVIA_0000 |  84.0 ms    |  8.6 ms  | 9.8x    | 264            | 0           |
| MUONSINE_0000|  20.0 ms    |  1.6 ms  | 12.5x   | 312            | 0           |
| CRESC132_0000| 5316.3 ms   | 23.2 ms  | 229x    | 4846           | 0           |

Inertia matches in every case. Delays drop to ZERO with MC64.
For VESUVIOU the new ratio against MUMPS is 4x (down from 84x).

** Why InfNorm is the current default

Per src/scaling/mod.rs:44-60, the 2026-04-13 Phase 2.2.3 follow-
up flipped default from MC64 → InfNorm because MC64 was a
silent no-op on HYDCAR20, METHANL8, SWOPF, HATFLDG. The choice
was symmetric — InfNorm helps a different class than MC64.

So the policy question is "make scaling adaptive based on a
cheap matrix feature" rather than "blanket switch to MC64".
The signature for the VESUVIO/CRESC class is exactly what
vesuvio_diag prints: diag_only ≥ 0.3·n.

** The deciding finding: POLAK6_0021

n=9 KKT. Under InfNorm: inertia (5,4,0), residual 9.21e-17. Under
both MC64 and adaptive: inertia (3,4,2) — TWO ZERO PIVOTS,
residual 1.31e13. MC64 produces a scaling that puts the matrix
into a singular-looking shape; BK completes but with wrong inertia
and the back-solve produces 13-digit-too-large residual.

CLAUDE.md hard rule "Inertia must be exactly correct" applies.
A 1+ inertia regression on a previously-correct matrix is a
default-blocker regardless of geomean or tail wins.

The adaptive policy doesn't dodge this because POLAK6_0021 must
have diag_only/n >= 0.3 (n=9, presumably 3+ diag-only rows) and
gets routed to MC64.

* 22:30 :triage:polak6:scaling:mc64:lever-c:
Triaged POLAK6_0021 per the previous entry's recommendation that
Policy 4 needs a "matched but bad" diagnostic. Result: POLAK6_0021
is the wrong test case — it is an oracle-disagreement matrix, not
a Policy-4 target.

** What was found

src/bin/polak6_diag.rs (new, throwaway) loads POLAK6_0021,
computes both InfNorm and MC64 scalings, prints scaled-diagonal
and off/diag ratio statistics, then factors and solves under each.

Raw matrix has |diag| range = 1.32e46 (1e-4 to 1e42). All four
canonical oracles disagree on inertia:
  rmumps: (5, 4, 0)   — the sidecar value bench treats as "expected"
  feral:  (5, 1, 3)   — InfNorm baseline
  MUMPS:  (5, 1, 3)   — same as feral, residual 1.90e-16
  SSIDS:  (6, 3, 0)   — different again

Verdict file says inertia_agreement: "none", verdict: "excluded".
The session-04-19 corpus-bench note's framing of POLAK6 as a
Policy-4 target was based on the wrong assumption that rmumps =
ground truth. It is not — feral InfNorm matches MUMPS exactly.

** Why the heuristics fail

Three candidate H1/H2/H3 thresholds (min |scaled diag| < 1e-12,
range > 1e8, max(|off|/|diag|) > 1e3): all THREE fire under both
InfNorm AND MC64 on POLAK6. Cannot distinguish "good" from "bad"
scaling on a matrix whose raw range is 1e46.

| heuristic                    | InfNorm | MC64    | distinguishes? |
|------------------------------|---------|---------|----------------|
| min |scaled diag|            | 5.1e-46 | 3.4e-42 | no             |
| range scaled |diag|          | 1.9e45  | 2.9e41  | no             |
| max scaled |off|/|diag|      | 1.9e45  | 2.9e41  | no             |

** What this means for Policy 4

The viable Policy-4 designs reduce to:
  A. Trial factor + retry — empirical, doubled cost on retry set.
  B. Raw |diag| range pre-filter — skip MC64 if range > 1e10.
  C. Accept the loss — Policy 3's −9 residual passes are likely
     all on excluded oracle-disagreement matrices anyway.

Cannot decide between A/B/C without first **enumerating the
−16 / −9 regression sets** from the lever-C corpus bench and
cross-referencing each entry against its .verdict.json
inertia_agreement field. That requires a bench-harness flag to
dump per-matrix residuals to a file, which doesn't exist yet.

Recommendation in the note: do NOT implement Policy 4 yet. The
prerequisite is the per-matrix residual dump + cross-reference.
If the regression set is dominated by 'excluded' matrices,
Option C (accept the loss, flip default to Auto) becomes the
defensible answer.
