* 15:00 :kahip:k2:impl:gap-bugs:
Implemented `crates/feral-kahip/src/flow.rs` (~600 LOC). First pass
compiled; ran tests with 6/29 failing:

  2. diamond_bottleneck: f=1 expected 2.
  3. grid_horizontal_cut_equals_k: f=1 expected 2.

  5. bipartite_matching_k_3_3: f=? expected 3.
  6. cut_saturation_invariant_on_random_graph: f=0 expected 42.

Bug 1 — gap detection at height 0. When the last non-source,
non-sink vertex at height 0 relabeled upward, height_count[0]
hit zero and my code triggered a "gap" — but height 0 is where
the sink lives, so "gap at 0" falsely disconnects the entire
source side. Fix: gate gap detection on `old_h > 0 && old_h < n`.
After this fix, 5/6 failures resolved; only the random-graph
cut-saturation test remained.

Bug 2 — stranded flow after gap lift. For the random 30-node
graph, excess trace at termination showed: source pushed 42,
vertices 21 and 22 ended at height 31 (= n+1, lifted by gap) with
excess 22 and 20 respectively; sink had excess 0. The algorithm
pushed flow into intermediate vertices, gap-lifted them, but then
stopped — they weren't re-inserted into the bucket system, so
their excess never got pushed back to source via reverse edges.

Fix: after gap-lift, re-add every lifted vertex `w` with
`excess[w] > 0` to `bucket[n+1]`, and bump `max_active` so the
main discharge loop picks them up. They then relabel higher and
cascade flow back to source via the reverse edges created by the
original preflow saturation.

After this fix the random graph returns f=0 with source side =
29 vertices (everyone except the disconnected sink), matching the
min-cut = 0 theorem. Added a connected hand-laid network test
(`cut_saturation_invariant_on_connected_graph`, f=10) to verify
saturated forward-cut invariant on a graph that actually has a
positive max-flow.

Design notes for future phases:
- Bucket array sized 2n+1 is sufficient. Heights can reach 2n in
  pathological cases (classical bound).
- Gap relabeling is essential for correctness here, not just
  performance: without proper handling of lifted vertices, flow
  gets stranded.
- Deterministic order is guaranteed by (a) stable adjacency-list
  iteration order (set at edge-insert time) and (b) FIFO within
  a bucket. Tested via `deterministic_under_adjacency_order`.
- The algorithm is NOT currently global-relabeling optimized.
  Cherkassky-Goldberg report 2-5x additional speedup from periodic
  BFS-from-sink resets; deferred to K6 bench.
