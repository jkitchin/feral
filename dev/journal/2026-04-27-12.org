* 12:30 :f3.4:design:
User asked to push through and close F3.4 in this session.
Spec line: "F3.4 — `solve_against_schur(s_matrix, rhs)`
convenience — optional, defer if no caller asks." The minimal,
honest reading: provide a method on `SchurBlock` that factors
the dense block via the existing dense BK LDL^T and solves
S·x=rhs. Caller controls the rest of the block-system glue
(building r_S = b_S - A_SF · A_FF^{-1} · b_F, applying
A_FF^{-1} again for x_F) — that part depends on application
ordering and isn't a clean library primitive without exposing
internal forward/backward sub.

Considered the full block-solve composition. Trace: with the
partial-elim factor returned by `factorize_multifrontal_with_schur`,
`solve_sparse(factors, [b_F | b_S])` returns
`[A_FF^{-1}·(b_F - A_FS·r_S) | r_S]` where `r_S` is the reduced
rhs. To recover the actual `x_S = S^{-1}·r_S` requires either
exposing forward/backward as separate primitives or running a
correction solve. Neither belongs in F3.4 "convenience" scope.
Declined to add; documented in commit body so a future caller
asking for it has the analysis ready.
