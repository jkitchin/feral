* 17:50 :issue47:mc64:cache:root-cause:
Localised #47. Per-factor timing instrumented in pounce-feral (temp):
stripped run has ~11 expensive (~520ms) + ~39 cheap (~19ms) factor
calls; kept run has all 44 calls at ~400ms — zero cheap refactors.

Reproduced standalone: =probe_explicit_zeros= now factors each matrix
4× on one warm Solver.

  stripped:  cold 434ms -> warm 15/14/15ms   symbolic_calls=1  mc64_cache_hits=1,2,3
  kept:      cold 468ms -> warm 359/359/360ms symbolic_calls=1  mc64_cache_hits=0,0,0

Root cause: explicit zeros defeat the MC64 value-bounded scaling cache
(Track B2). Evidence:
- Cold factor fine either way (~450ms) — not a cascade, not fill.
- Symbolic IS reused either way (symbolic_call_count stays 1). The
  pattern fingerprint (PatternFingerprint::of, solver.rs:83 — hashes
  col_ptr+row_idx only) is constant for the kept matrix since POUNCE
  fixes structure once. Symbolic cache is not the problem.
- MC64 cache never hits with explicit zeros (mc64_cache_hits stays 0),
  so the Hungarian match reruns every factor — ~345ms of the ~360ms
  warm refactor. Stripped, the cache hits -> warm refactor ~15ms (24×).

The explicit-zero (2,2) diagonals make the cache's diagonal-dominance
value-bound test (mc64_value_bound_passes) fail, so it conservatively
recomputes scaling every iterate. That recompute is the whole 2×.

Implication: the issue's hypothesised mechanism ("#46 fast path keys on
structural absence") is wrong. The fix is issue option (a) — feral
drops explicit zeros in from_triplets/symbolic so the kept matrix is
structurally identical to the stripped one and the MC64 cache engages.
#47 stays open. Posted the pinned root cause on the issue.

* 18:45 :issue47:mc64:correction:root-cause:
Corrected the #47 mechanism after reading the cache code with the
human. The §17:50 entry (and the comment first posted on #47) claimed
explicit zeros defeat the MC64 value-bounded scaling cache via its
diagonal-dominance value-bound test. WRONG — value_bound.rs already
disqualifies explicit-zero-diagonal rows (qualifying_rows:155; test
explicit_zero_diagonal_disqualifies_row) and is never consulted here
because the cache is never *populated*.

Re-ran probe_explicit_zeros with scaling_info instrumentation:
  stripped: scaling_info=Applied every call, mc64_fallbacks=0, cache_hits 1,2,3
  kept:     scaling_info=Mc64FallbackToInfnorm{Mc64ScalingDegenerate}
            every call, mc64_fallbacks 1..4, cache_hits 0,0,0.

Real mechanism: explicit zeros tip MC64's symmetric matching into
Mc64ScalingDegenerate (scaling spread > 1/EPS — the 3e82 blow-up
documented at scaling/mod.rs:171, research note
kkt-mc64-scaling-blowup-2026-05-20, issue #45). A degenerate MC64 ->
ScalingInfo::Mc64FallbackToInfnorm, not Applied. The B2 cache populates
only on Applied (solver.rs:803), else None (solver.rs:812) -> never
populated -> the full Hungarian reruns every warm factor (~345ms) and
is discarded. That wasted Hungarian is the entire 2x.

The stripped matrix's MC64 returns Applied — so explicit zeros are the
*cause* of the degeneracy, not merely a cache-key mismatch. Drop-zeros
is therefore the root-cause fix, not a workaround. Orthogonal option: a
negative cache (remember MC64-degenerate fingerprints, skip the wasted
Hungarian). Posted the corrected mechanism on #47.

* 19:10 :issue47:mc64:reference:mumps:ssids:
Asked what MUMPS 5.8.2 and SPRAL SSIDS do with explicit zeros, to
inform the #47 fix scope. Spawned mumps-expert and spral-expert.

Both converge: NEITHER strips explicit zeros from the input matrix or
the analysis pattern. Analysis (ordering, etree, symbolic) is
value-blind; the stored pattern incl. zeros is the contract. Stripping
at input would change etree/fill and break the value<->pattern index
map. This confirms the "do NOT strip the whole matrix" caution
(value-dependent fingerprint -> re-analysis churn).

The matching/scaling kernels handle zeros locally:
- SSIDS (scaling.f90:640 hungarian_wrapper, :1524 auction_match,
  match_order.f90:100): builds a private working copy and does
  `if (val == 0.0) cycle` BEFORE `log(|val|)`. A stored zero = non-edge
  for the matching. An explicit-zero (2,2) diagonal -> that column is
  unmatched -> Duff-Pralet correction (scaling.f90:767-800) sets its
  scale from matched neighbors, 1/0=1 convention -> scaling bounded.
  Equilibration is max-based, needs no strip (`where(maxentry>0)`).
- MUMPS (dana_aux.F:3854-3860): keeps zeros in the Hungarian but maps a
  zero-valued edge to a large FINITE cost FACT+RINF (not log(0)=-inf,
  not removal); RINF is a matrix-scaled infinity surrogate. Post-match
  guard: scale factor > 0.5*log(huge/2N) -> warn-and-proceed. Norm
  scalings: row/col max==0 -> scale 1.0.

Implication for #47: the fix is NOT a matrix-level strip. It is ~5
lines in feral's MC64 symmetric Hungarian kernel — drop explicit zeros
from the matching working copy before the log transform (SSIDS style),
and handle the resulting unmatched columns (Duff-Pralet correction or
identity for that column). feral currently feeds zeros into the
matching -> log(0) garbage -> 3e82 degenerate scaling ->
Mc64ScalingDegenerate -> InfNorm fallback -> cache never populates.
feral already has the MUMPS-style "detect degeneracy + fall back" half;
it lacks the SSIDS-style "don't feed log(0)" half. Smaller and
lower-risk than the from_triplets canonicalization considered earlier.
