* 14:55 :issue45:from-triplets-strict:guard:

loud-catch guard for the issue-#45 duplicate-coordinate class. Design:

- `from_triplets` behavior is UNCHANGED — it still sums duplicates.

* 18:35 :issue45:mc64:scaling-blowup:root-cause-final:
Found the precise defect. `compute_symmetric` (src/scaling/mc64.rs)
forms `s[i] = exp((u[i]+v[i]-cmax[i])/2)` from Hungarian dual
potentials, clamped only at the per-entry `exp` overflow ceiling
(LOG_HUGE=709). On the CHO KKT the potentials are large but finite
(arg ~ +-95), so the clamp never fires — and the resulting scaling
vector spans `min=2.89e-42 .. max=8.88e40`, a spread of ~3e82.
`info=Applied` (n_matched==n). Applying a scaling whose own spread
exceeds 1/EPS is numerically degenerate: D is singular to working
precision, D.A.D underflows, BK accepts exact-0.0 pivots, garbage
solve — silently `Success`. That is #45.

Corpus measurement (`probe_mc64_spread.rs`, 38 parity families):
every legitimate matrix has MC64 spread <= 3.27e15 (ssine); next is
hatfldbne 1.33e15, muonsine 1.35e12, vesuviou 5.34e11. The CHO
catastrophe at 3e82 sits in a 67-order gap above the whole corpus.
A guard at `1/EPS ~ 4.5e15` catches CHO and clears every corpus
matrix (3.27e15 < 4.5e15).

Why the existing issue-#24 guard misses it: `compute_scaling_auto_
with_cache` (mod.rs:295) has a fast-path `if raw_diag_range(matrix)
>= RAW_GUARD(1e6) { return mc64 }` (mod.rs:353) that commits to MC64
*without* checking the produced vector. The CHO KKT is ill-
conditioned (raw range >= 1e6) so it takes that fast-path and the
`mc_off` catastrophe diagnostic is never reached.

Why the real POUNCE run hits #45 but the dumped .mtx does not: the
dumped `cho_iter0_kkt.mtx` lost its explicit zero diagonals (10812
of 43332 stored) -> Auto routes to InfNorm -> correct. POUNCE's
live KKT assembly emits a complete diagonal -> Auto routes to
Mc64Symmetric -> catastrophic scaling -> #45. Diagonal completeness
only flips the `pick_scaling_strategy` router; it has zero numeric
effect (confirmed 17:55).

Fix decided (this session, correctness only):
- Add a catastrophic-spread guard to `compute_scaling_auto_with_cache`:
  after computing the MC64 vector, if `scaling_spread > 1/EPS`, fall
  back to InfNorm with a new `Mc64FallbackReason::Mc64ScalingDegenerate`.
  Placed before the `raw_diag_range` fast-path so it fires regardless.
  Closes #45 (CHO -> InfNorm -> relres 2.46e-8).
- #46 (performance: InfNorm cascade -> 28M fnnz / 11 s) is NOT fixed
  by this. The 50x speedup needs a matching-based scaling that is
  sane on saddle points; feral's MC64-symmetric is not. That is a
  separate, larger effort — documented, not attempted here.
  "Correctness before performance, always."
